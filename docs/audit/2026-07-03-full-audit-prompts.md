# Полный аудит сервиса — серия промптов (2026-07-03)

Цель аудита: подготовка сервиса к **масштабированию и продаже**. Приоритеты владельца:
устойчивость → масштабируемость → скорость/отзывчивость → качество UI → чистота кода.

## Как пользоваться

1. Каждая фаза запускается **отдельной сессией** Claude Code в корне проекта.
2. В сессию вставляется: **Блок А (общие правила)** + **промпт нужной фазы**. Целиком, без сокращений.
3. Порядок фаз — как в файле (0 → 8). Фазы 3–7 независимы, их можно гонять параллельно в разных сессиях, но фазы 0–2 — строго первыми.
4. Каждая фаза пишет отчёт в `docs/audit/2026-07-XX-phase-N-<тема>.md`. Ничего не чинит.
5. После всех фаз запускается **Фаза 9 (синтез)** — сводит отчёты в единый приоритизированный план.

```

2. К БД прода — только read-only запросы (SELECT, EXPLAIN, EXPLAIN ANALYZE на SELECT).

ВАЖНО ПРО ДРЕЙФ: часть RPC-функций и триггеров в проде РАСХОДИТСЯ с файлами миграций
в репо (правились через MCP напрямую). Источник правды по телам функций — прод:

   «Выглядит медленно» без замера в отчёт не попадает.

СТИЛЬ: описания проблем — простым языком, понятным не-программисту. Владелец продукта
не разработчик. Каждая находка должна отвечать на вопрос «что сломается и когда».

Ничего не чинить, даже однострочники. Только отчёт.

## Фаза 1 — Безопасность и изоляция клиентов (multi-tenancy) 🔴 приоритет №1 для продажи

**Зачем:** сервис продаётся как multi-tenant (воркспейсы разных клиентов в одной БД). Одна утечка данных между воркспейсами — конец продажам.

```
ФАЗА 1 АУДИТА: безопасность и изоляция данных между воркспейсами.

Прочитай дополнительно: .claude/rules/data-model.md, .claude/rules/channels.md
(раздел «Авторизация Edge Functions — матрица»), отчёт фазы 0.

Главный вопрос аудита: может ли пользователь воркспейса А любым путём увидеть или
изменить данные воркспейса Б? И может ли клиент (роль «Клиент») увидеть внутреннюю
кухню команды?

1. RLS-покрытие: через MCP получить список ВСЕХ таблиц public без включённого RLS
   и всех таблиц с RLS но без политик. Для каждой — оценить, есть ли там
   workspace-данные.
2. RLS-политики по workspace_id: выборочно (топ-20 таблиц по чувствительности:
   project_messages, project_threads, participants, documents, files, email_accounts,
   telegram_*, wazzup_*, audit_logs, message_attachments...) проверить, что политика
   реально ограничивает по членству в воркспейсе, а не только по «залогинен».
3. SECURITY DEFINER функции: полный список из прода, для каждой — установлен ли
   search_path, есть ли проверка прав внутри, кто имеет EXECUTE. Особо: гранты anon —
   каждый anon-EXECUTE обосновать (известное исключение: 4 RPC-резолвера коротких
   ссылок для proxy.ts обязаны иметь anon; get_inbox_needs_reply_threads /
   get_inbox_awaiting_reply_threads имеют anon предположительно по оверсайту —
   проверить и зафиксировать).
4. Edge Functions авторизация: пройти по всем ~93 функциям (субагентами, партиями).
   Для каждой: verify_jwt? проверка x-internal-secret? настоящий getUser + проверка
   членства в воркспейсе, или только префикс Bearer? Матрица в channels.md — эталон,
   искать отклонения. Особо искать IDOR: функции, принимающие id сущности и не
   проверяющие принадлежность вызывающему (прецедент: telegram-edit-message для
   личных диалогов, telegram-send-message для беспроектных чатов — оба чинились).
5. Storage: политики всех бакетов — может ли юзер воркспейса А скачать файл
   воркспейса Б (по прямому пути или signed URL логике).
6. Секреты: grep по src/ и supabase/functions/ на захардкоженные ключи/токены/секреты;
   проверить что SERVICE_ROLE_KEY не попадает в клиентский бандл; NEXT_PUBLIC_* —
   только публичное. Проверить git-историю миграций на закоммиченные секреты
   (известный прецедент — INTERNAL_FUNCTION_SECRET уже ротировали).
7. Роль «Клиент»: клиент видит только своё? Проверить RLS restrictive
   project_messages_visibility (team/self), скрытие внутренних тредов, boards, docs.
8. Server-side guards: src/proxy.ts + (app)/layout.tsx — все ли приватные роуты
   закрыты на сервере, а не только клиентским ProtectedRoute. Блокировка участника
   (can_login) — действует ли мгновенно.
9. Импersonация: read-only гарантии (триггер prevent_impersonation_writes) — покрывает
   ли ВСЕ таблицы, включая созданные после 2026-05-08.
10. Advisors security из фазы 0 — разобрать каждый пункт: реальная проблема или
    известный false positive.

В отчёте отдельным блоком: «Топ-5 рисков при продаже сервиса третьим лицам».
```

## Фаза 2 — БД и масштабируемость данных

**Зачем:** главный вопрос масштабирования — «что сломается при 10× воркспейсов / тредов / сообщений».

```
ФАЗА 2 АУДИТА: база данных и масштабируемость.

Прочитай дополнительно: .claude/rules/messenger-ledger.md (записи 2026-06-17 про
масштабирование инбокса — там уже сделаны материализация thread_unread_state /
thread_inbox_meta и Broadcast; НЕ предлагать это заново),
docs/feature-backlog/2026-06-17-inbox-materialization-scaling.md, отчёт фазы 0.

Модель роста для оценки: сейчас ~1 крупный воркспейс (~1100 тредов, десятки тысяч
сообщений). Цель — 20-30 пользователей на воркспейс, 1000-2000 пользователей всего,
сотни воркспейсов. Каждую находку оценивать против ЭТОЙ модели.

1. Тяжёлые RPC: топ-10 функций по вызовам/времени (pg_stat_statements если доступен,
   иначе по списку из ledger). Для каждой — EXPLAIN ANALYZE с реальными параметрами
   (read-only!), зафиксировать время и buffers. Отдельно: линейно ли растёт время
   с числом тредов/сообщений воркспейса, и есть ли зависимость от ЧИСЛА воркспейсов
   (изоляция нагрузки клиентов друг от друга).
2. Лимит 1000 строк (грабля G9 из ledger): найти ВСЕ вызовы supabase.rpc без .range(),
   возвращающие наборы, которые растут с данными (get_inbox_unread_threads,
   get_inbox_needs_reply_threads ~322 строки уже, get_inbox_awaiting_reply_threads,
   get_inbox_search_threads и любые новые). Для каждого — оценка «когда упрётся».
3. Индексы: advisors performance + pg_stat_user_indexes — недостающие индексы на
   FK/workspace_id/is_deleted; неиспользуемые индексы (только НЕ покрывающие FK и
   не дедуп-UNIQUE — см. false-positives). Таблицы с высоким bloat/dead tuples
   (особо thread_unread_state — там fillfactor 80 не случайно).
4. Триггерная нагрузка: на project_messages висит цепочка триггеров (dispatch, unread
   recompute, inbox meta, broadcast). Посчитать сколько работы делает ОДИН INSERT
   сообщения (список триггеров + что каждый делает). Оценить при 10× потоке сообщений.
5. Realtime: сколько подписок держит один онлайн-клиент, что происходит при 30 онлайн
   в воркспейсе (Broadcast уже внедрён для инбокса — проверить, не осталось ли других
   Postgres Changes подписок с RLS-проверкой на событие, кроме осознанного
   useProjectMessages на открытый тред).
6. pg_cron: все джобы — статусы за последние 7 дней, есть ли молчаливо падающие.
   Захардкоженные ключи в командах кронов — зафиксировать список (для ротации).
7. Миграции: 308 файлов — применимы ли на ЧИСТУЮ БД по порядку (мысленно/по именам:
   коллизии версий-префиксов уже были 2026-07-01). Это важно для будущих стейджингов
   и второго инстанса. Список миграций, которые упадут или применятся не в том порядке.
8. Единая точка отказа: всё на одном Supabase-инстансе (max_connections=60) и одном
   VPS. Зафиксировать текущие лимиты тарифа, connection pool, что упрётся первым.
9. Стратегия бэкапов: что реально настроено (Supabase PITR? расписание?), покрыт ли
   Storage, покрыт ли VPS (mtproto-сессии). Restore-процедура описана где-нибудь?

В отчёте отдельным блоком: «Что упрётся первым при 10× росте» — ранжированный список.
```

## Фаза 4 — Устойчивость и отказоустойчивость

```
ФАЗА 4 АУДИТА: устойчивость — что происходит, когда что-то падает.

Прочитай дополнительно: .claude/rules/channels.md (send_status, watchdog),
.claude/rules/messenger-ledger.md (раздел «Повторяющиеся грабли»), docs/bugs/.

Вопрос фазы: какие сбои сервис переживает молча и корректно, какие — теряют данные
или молча ломают функциональность, и как быстро владелец УЗНАЕТ о поломке.

1. Наблюдаемость (главный вопрос): как владелец узнаёт, что что-то сломалось?
   Есть ли алерты на: падающие pg_cron джобы, рост message_send_failures,
   5xx от Edge Functions, падение mtproto-контейнера, истечение Gmail watch,
   переполнение диска VPS. Если мониторинга нет — это находка 🔴 сама по себе.
   Предложить минимальный дешёвый вариант (uptime-пинг + алерт на cron-фейлы).
2. Пути отказа внешних сервисов: что видит пользователь и что происходит с данными,
   когда недоступен/тормозит: Telegram API, Wazzup, Gmail API, Resend, Anthropic/LLM,
   mtproto-service. Пройти по коду обработки ошибок каждого канала: ошибка доходит
   до пользователя честно или глотается? (send_status/watchdog покрывает исходящие —
   проверить входящие и вспомогательные пути: аватары, транскрипция, дайджесты.)
3. Идемпотентность и дубли: точки, где повтор операции (retry пользователя, двойной
   клик, повторный webhook) создаёт дубль или теряет данные. Дедуп входящих TG уже
   выстрадан (см. ledger) — искать аналогичные дыры в других каналах и в НЕ-мессенджер
   операциях (создание проекта, загрузка документа, формы).
4. Гонки: fire-and-forget вызовы без обработки ошибок (grep по .then( без .catch,
   supabase.functions.invoke без await/обработки); setTimeout-логика, теряющаяся при
   размонтировании.
5. Очереди и застревание: сообщения в pending навсегда (watchdog покрывает?),
   attachment_status failed без пути восстановления, черновики.
