# Realtime → Broadcast: перевод сообщений (runbook совместного сеанса)

Цель: снять `project_messages` с прямого слежения Postgres Changes (главный источник
WAL-нагрузки, ~63% времени БД), переведя живую ленту и тост нового сообщения на
Broadcast.

## Что уже сделано (в репо, НЕ в проде)

- **Шаг A (в проде):** триггер `trg_inbox_broadcast` обогащён полями `thread_id /
  message_id / op / has_attachments` (миграция `20260704120000_...`). Добавочно —
  старый потребитель «Входящих» не затронут. **Уже применено, безопасно.**
- **Шаг B/C (в репо, не задеплоено):**
  - Единственная supabase-подписка на топик `inbox:<ws>` — в `useWorkspaceMessagesRealtime`
    (в supabase-js `channel(topic)` возвращает ТОТ ЖЕ инстанс → несколько независимых
    подписок нельзя). Она переизлучает событие в локальную шину `inboxBroadcastBus`.
  - `useProjectMessages` (живая лента): INSERT/UPDATE/DELETE сообщений треда — из шины
    (было 3 postgres_changes). Реакции/вложения оставлены на postgres_changes.
  - `useNewMessageToast`: новое сообщение — из шины + дотягивание сообщения по id под
    RLS (leak-safe: нет доступа/видимости → нет тоста). Тост «Новый диалог» оставлен
    на postgres_changes (project_threads редкая).

## Что делаем в совместном сеансе (тихое окно, мало трафика)

### Деплой
1. `git push` (я выполню по твоему «да») → CI соберёт + blue/green выкатит фронт.
   Дождаться зелёного Deploy.

### Смок-тест (ты — за компом, я смотрю)
Открой чат в сервисе и проверь по каждому каналу:

| Проверка | Ожидание |
|---|---|
| Тебе пишут в **Telegram** (в открытый чат) | Сообщение появляется в ленте **само**, без F5 |
| Тебе пишут в **WhatsApp** | То же |
| Приходит **письмо** | То же |
| Ты **отправляешь** сообщение | Появляется сразу, **не двоится** (оптимистик не ломается) |
| Сообщение с **файлом** | Появляется с файлом (не «пустое», потом с файлом) |
| **Реакция** на сообщение | Прилетает в открытый чат |
| **Ответ** на твоё сообщение | Прилетает + подсветка |
| **Тост** нового сообщения (чат закрыт) | Всплывает, имя/текст/канал верные |
| **Внутреннее** сообщение «для команды» | Клиенту-участнику НЕ приходит тост (leak-check) |
| **Новый диалог** (написал клиенту первым в TG/WA) | Тост «Новый диалог» |

Если всё ок на всех каналах — переходим к финалу. Если что-то не так — **откат** (ниже).

### Финал (только после успешного смока) — Шаг D
Снять `project_messages` с publication → WAL-нагрузка падает:
```sql
ALTER PUBLICATION supabase_realtime DROP TABLE public.project_messages;
```
Проверить замером (pg_stat_statements): доля `realtime.list_changes` упала.
`message_reactions`, `message_attachments`, `project_threads` — **оставить** в publication
(их postgres_changes ещё используются, они малонагружены).

## Откат (если смок провалился)

- Фронт: `git revert <commit>` + push → CI вернёт старую версию (blue/green, без простоя).
  ИЛИ в GitHub Actions «Re-run» предыдущего успешного деплоя.
- Триггер (Шаг A) откатывать НЕ нужно — он безопасен (лишние поля игнорируются старым
  фронтом). Если всё же: восстановить прежнее тело из git-истории миграции.
- Шаг D (если уже сделали DROP TABLE) откатить: `ALTER PUBLICATION supabase_realtime ADD
  TABLE public.project_messages;` — вернёт старый режим для postgres_changes.

## Риски и как закрыты

- **Утечка контента через broadcast** → сигнал несёт только id, тост дотягивает под RLS.
- **Коллизия каналов одного топика** → одна supabase-подписка + локальная шина.
- **Оптимистик отправки** → сохранена проверка `isMutating(['sendMessage', threadId])`.
- **Вложения приходят позже сообщения** → сохранён skip по `has_attachments` + отдельный
  postgres_changes на message_attachments.
- **Мониторинга нет** → поэтому только совместный сеанс + немедленный откат под рукой.
