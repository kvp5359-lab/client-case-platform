# Аудит, Фаза 1 — Безопасность и изоляция клиентов (2026-07-03)

## Резюме

Ядро multi-tenancy построено правильно: RLS включён на **всех 166 таблицах** (нет ни одной без RLS), политики на чувствительных таблицах скоупятся по членству в воркспейсе или `can_user_access_thread`, все SECURITY DEFINER функции имеют `search_path`, server-side guards в layout'ах валидируют JWT через `getUser` + проверяют `can_login`/`is_deleted`. Но найдено **несколько реальных дыр изоляции между воркспейсами**, которые критичны именно при продаже сервиса третьим лицам: анонимный доступ к inbox-превью через 3 RPC, публичный бакет с 417 клиентскими документами, частичные IDOR в Google-функциях (утечка email участников чужого проекта, чтение чужих файлов в свой Drive) и несколько wazzup-функций, шлющих действия в чужой воркспейс без проверки членства. Ни одна дыра не даёт «слить всю базу одним запросом», но каждая — это доступ к данным/действиям другого клиента при знании UUID.

## Топ-5 рисков при продаже сервиса третьим лицам

1. 🔴 **Анонимный IDOR: чтение inbox-превью любого пользователя** — 3 RPC исполняются ролью `anon`.
2. 🟠 **Публичный бакет `docbuilder` с 417 документами клиентов** — раздаются по публичным URL, листинг разрешён.
3. 🟠 **Утечка email участников чужого проекта** через `google-sheets-create-brief`.
4. 🟠 **Инъекция сообщения/реакции в чужой воркспейс** — `wazzup-send-reaction`, `wazzup-mark-read`, инъекция в `project_messages`.
5. 🟠 **Чтение/запись чужих файлов** в свой Google Drive — `google-drive-export-documents`, `google-drive-download-file`, `export-to-drive`.

## Проверено — проблем нет

- **RLS включён на всех 166 таблицах** (0 таблиц без RLS). 6 таблиц с RLS без политик (`thread_inbox_meta`, `thread_unread_state`, `telegram_bot_sessions`, `telegram_mtproto_auth_states`, `external_outgoing_dedup`, `message_send_dispatch`) — это **намеренная блокировка** (доступ только service_role, читаются через SECURITY DEFINER RPC), не дыра.
- **Все SECURITY DEFINER функции имеют `search_path`** (0 без него) — нет инъекции через search_path.
- **Server-side guards**: `(app)/layout.tsx` и `workspaces/[id]/layout.tsx` валидируют JWT через `getUser` (не `getSession`), проверяют participant/`can_login`/`is_deleted` → блокировка мгновенна на server-render. `proxy.ts` не пускает на чужой subdomain-воркспейс.
- **Секреты**: захардкоженных ключей/токенов в `src/` и миграциях не найдено; `SERVICE_ROLE_KEY` только в `supabase-service.ts`, импортится лишь в API-route (server), не в клиент.
- **RLS ядра** (`project_messages` select/update/delete, `project_threads`, `participants`, `files`, `documents`, `message_attachments`, `email_accounts`, `audit_logs`) — корректно скоупятся по членству/`can_user_access_thread`. Роль «Клиент» отсекается restrictive-политикой `project_messages_visibility` (team/self скрыты).
- **Impersonation** — write-guard навешивается триггером на все public-таблицы динамически (миграция `20260507`), read-only соблюдается.
- **Storage воркспейс-бакеты** (`files`, `document-files`, `message-attachments`, `document-templates`) — читаются только членами воркспейса (фильтр по первому сегменту пути = workspace_id).
- **Прежние IDOR** (`telegram-edit-message`, `telegram-send-message`) — исправлены (workspace из `project_messages.workspace_id`).

## Не проверено

- Реальная эксплуатируемость листинга публичных бакетов от лица anon (нужен тест с anon-ключом — вне read-only мандата).
- Флаги деплоя `verify_jwt` конкретных функций (в config.toml нет per-function блоков — управляется флагами при деплое; надо смотреть историю деплоя или Dashboard).
- Полный обход всех 477 RLS-политик (проверены ~20 самых чувствительных таблиц; остальные — выборочно).
- git-история на закоммиченные секреты глубже последних миграций (INTERNAL_FUNCTION_SECRET уже ротировали — известно из ledger).
