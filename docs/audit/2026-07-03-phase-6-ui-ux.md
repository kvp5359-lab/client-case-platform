# Аудит, Фаза 6 — UI/UX и готовность продукта (2026-07-03)

## Резюме

Базовая дизайн-система на месте (shadcn/ui, Sonner-тосты консистентны на 100%, есть аккуратный `useConfirmDialog`, error boundary с дружелюбным русским экраном), мобильная навигация проработана на ~95% (таб-бар, drawer, `100dvh`, свайп досок). Но для продукта, который будут показывать покупателям, есть пять заметных шероховатостей первого впечатления: пустой воркспейс встречает нового пользователя тупиком без онбординга; сырые SQL/английские ошибки протекают в тосты и на error-страницы; на телефоне пропадают hover-only действия (69 мест без mobile-guard); рядом с красивыми диалогами торчат 11 нативных браузерных `confirm()`; и на ключевых экранах вместо скелетонов «моргает» текст «Загрузка…». Всё это — не баги логики, а полировка, но именно она формирует ощущение «готовый продукт vs сырой прототип». Аудит по коду; визуальную проверку в браузере отдельные пункты требуют подтвердить вживую.

## Что уже хорошо (не трогать)

- **Тосты** — 100% Sonner, один `Toaster`, консистентно.
- **Хорошие empty states с иконкой+CTA**: Проекты, Задачи, Доски/Списки, База знаний.
- **ErrorBoundary** — дружелюбный русский экран, стек только в dev.
- **Мобильная навигация** — таб-бар, drawer-сайдбар, fullscreen-панель, `100dvh` композер, свайп досок (~95%).
- **Ошибки на русском** — английские тексты только в тест-файле, не у пользователя.
- **Диалоги** консистентны (shadcn Dialog/AlertDialog + осознанный `EditorDialogContent`).

## Быстрые победы перед демо (низкий риск, высокий эффект)

1. Онбординг-карточка в пустом воркспейсе (#1).
2. Обёртка «дружелюбная ошибка» + убрать `{error.message}` с error-страниц (#2).
3. Заменить 11 нативных `confirm()` на `useConfirmDialog` (#4).
4. Массовая замена `opacity-0 group-hover` → `md:opacity-0 md:group-hover` (#3).
5. Вернуть `focus-visible:ring` вместо тотального `outline-none` в поповере (#7).

## Не проверено (требует живого прогона)

- Визуальное качество empty states, реальные layout shifts, поведение на конкретных устройствах — аудит по коду; для точной картины нужен прогон в браузере/на телефоне.
- Safari: `:has` (11) и container-queries (9) поддержаны Safari 16+; на старых Safari (<16) деградация — проверить, если целевая аудитория на старых iOS.
