# Аудит, Фаза 3 — Производительность фронтенда и отзывчивость (2026-07-03)

## Резюме

Фундамент фронта в хорошем состоянии: код-сплиттинг сделан грамотно (tiptap, react-big-calendar, tiptap-редакторы, тяжёлые табы настроек — всё вынесено в `React.lazy`, поэтому 390-килобайтный tiptap не попадает в начальный бандл), нет импортов lodash целиком, глобальные дефолты React Query разумные, realtime-хук инбокса уже оптимизирован (два темпа инвалидации + Broadcast). Реальные проблемы отзывчивости — две и обе бьют по самому частому экрану (инбокс): (1) **ручные мутации `mark-read`/реакция/отправка обесценивают собственную оптимистику**, синхронно рефетча весь инбокс (6 тяжёлых RPC) на каждый клик; (2) **инбокс и список задач не виртуализированы**, а 4 из 5 вкладок инбокса грузятся без пагинации — при сотнях тредов тяжёлый рендер. Живые per-page замеры не делались (нужен логин + риск задеть прод) — для них в проекте уже есть встроенный трейсер `cc_perf_trace`, рекомендация владельцу ниже.

## Бандл и код-сплиттинг — в порядке

- **tiptap** (~390 КБ): 30 файлов импортят его статически, но все — внутри поддерева `TaskPanel`/`MessengerTabContent`/редакторов, которое само загружается через `React.lazy` (`TaskPanelTabContents.tsx`, `TaskListView.tsx`, `QuickReplyFormDialog`, `KnowledgeBaseArticleEditorPage`). В начальный бандл tiptap не попадает.
- **react-big-calendar**: `BoardListCalendarView` подгружается `lazy` из `BoardListCard` — на доске без календарного режима не грузится.
- **pdf-lib**: изолирован в `utils/files/mergePDF.ts`.
- Нет `import lodash` целиком; чанки топ 388/384/236 КБ — приемлемо для приложения такого масштаба.
- Все табы `WorkspaceSettings` — `React.lazy`.

## React Query — что уже хорошо (не трогать)

- Realtime-хук `useWorkspaceMessagesRealtime` разведён на два темпа (лёгкие 400 мс leading+trailing / тяжёлые ≤1/1.5с), транспорт Broadcast. Претензий нет.
- Единый источник `lastReadAtByThreadId` для контура баблов и счётчиков; дублей загрузки треда нет (вычищены в июне).
- `enabled`-гейты почти везде; глобально `refetchOnWindowFocus: false`, `retry: 1`.
- item_lists виртуализированы через `TableShell` (порог 40).

## Живые замеры — не выполнены (обоснование + рекомендация)

Per-page тайминги (TTI, число запросов при открытии) не снимались: авторизованные страницы требуют логина (кред нет), а прод-данные трогать нельзя. **В проекте уже есть встроенный трейсер** `src/utils/perfTrace.ts` (тумблер `cc_perf_trace` в Настройки → Общие, или `__ccPerf.on()`). Рекомендация владельцу: включить его и открыть инбокс/доску/список задач на боевом воркспейсе (>500 тредов), собрать `console.table` — это даст точные числа для находок #1, #2, #4. Отдельная сессия при желании настроит локальный вход и снимет замеры в preview.

## Не проверено

- Реальные TTI/число запросов по страницам (нужен живой прогон, см. выше).
- Layout shifts и дёрганья скролла в цифрах (визуально — Фаза 6).
- Поведение под realtime-штормом на клиенте (нужен живой прогон с N онлайн).
