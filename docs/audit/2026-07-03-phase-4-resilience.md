# Аудит, Фаза 4 — Устойчивость и отказоустойчивость (2026-07-03)

## Резюме

Механика доставки сообщений выстрадана и надёжна: единый `send_status`, watchdog `scan_dispatch_failures` (1/мин, на момент аудита 0 открытых сбоев), дедуп входящих во всех каналах (Telegram/Wazzup/Resend), кнопка «Повторить» вместо опасного авторетрая, blue/green-деплой с healthcheck и retry-обёрткой, авто-восстановление mtproto-сессий из БД, error boundaries на фронте. Но главный риск устойчивости для продаваемого сервиса — **нулевая наблюдаемость**: внешнего мониторинга и алертов нет вообще (Sentry закомментирован, только `console.log`), поэтому владелец узнаёт о поломке (упавший крон, зависший mtproto, всплеск ошибок отправки, переполнение диска) только от жалоб пользователей. Второй риск — **миграции БД применяются вручную, вне CI**, что при живом фронт-деплое даёт окно рассинхрона «код ждёт колонку, которой ещё нет».

## Проверено — работает надёжно

- **Доставка исходящих**: `send_status` (pending/sent/failed) + watchdog `scan_dispatch_failures` (1/мин, 0 открытых сбоев сейчас); авто-resolve failures при переходе в `sent` (триггер); клиентский таймер 60с; глобальный тост `SendFailureToasts`. Авторетрая нет осознанно (риск дублей) — только кнопка «Повторить».
- **Идемпотентность входящих**: дедуп во всех каналах — Telegram (content-based UNIQUE, multi-bot), Wazzup (`wazzup_message_id` UNIQUE), Resend (inbound dedup). Повторный webhook → `outcome='duplicate'`, не дубль.
- **Деплой**: blue/green — поднять новый цвет → ждать healthcheck 200/302 → переключить nginx upstream → погасить старый. При невзлёте нового старый остаётся живым. SSH-деплой обёрнут в retry (`nick-fields/retry@v3`, 45с пауза) под транзиентные обрывы Actions→VPS. Простоя/502 при выкате нет.
- **Восстановление mtproto**: сессии переподнимаются из БД при рестарте (`sessions/manager.ts` bootstrap). `restart: unless-stopped`.
- **Кроны**: все 7 зелёные, 0 фейлов за 7 дней (Фаза 0). `cleanup-cron-job-run-details` чистит историю. `gmail-watch-refresh` продлевает 7-дневный watch (последний успех сегодня).
- **Error boundaries**: `(app)/error.tsx`, `app/error.tsx`, `components/ErrorBoundary.tsx` — краш компонента не роняет всё приложение.
- **Импersонация**: write-guard на всех таблицах (нельзя случайно испортить данные в режиме зрителя).

## Сценарии молчаливой потери данных — оценка

- **Одиночное входящее в одну секунду с идентичным текстом** (Telegram content-dedup) — теоретически второе дедуплится (ledger, на практике не встречается). 🟡 приемлемо.
- **Пропущенный webhook при недоступности нашего endpoint** — Telegram/Wazzup/Resend ретраят сами; риск потери низкий, но **без мониторинга (п.1) невидим**.
- **Крон умер молча** — `gmail-watch-refresh` не продлит watch → входящая почта перестанет приходить, узнаете не сразу. Смыкается с п.1.
- Систематической потери данных в коде не обнаружено; главный множитель всех рисков — отсутствие наблюдаемости (п.1).

## Не проверено

- Фактические настройки PITR/бэкапов (Dashboard — п.4).
- Реальное поведение при недоступности каждого внешнего API под нагрузкой (нужен хаос-тест — вне read-only мандата).
- Что происходит с несохранёнными черновиками формы при blue/green-переключении (черновики сообщений в localStorage/IndexedDB переживают — проверено ledger'ом; формы настроек — не проверено).
