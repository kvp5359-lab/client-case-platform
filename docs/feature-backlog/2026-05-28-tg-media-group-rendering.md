# Telegram media_group — рендерить альбомы как один баббл с N вложениями

**Дата создания:** 2026-05-28
**Тип:** UX / messenger rendering
**Приоритет:** средний (комфорт, не критично)
**Карантинная зона (опционально):** да, если выбираем вариант B

---

## Проблема

Когда клиент отправляет в Telegram **media_group** (альбом из 2-10 файлов), TG шлёт **N отдельных webhook-апдейтов** с разными `message_id`. В Telegram-UI клиент и собеседник видят это как ОДНО сообщение-альбом. В ClientCase каждый файл рендерится отдельным баблом → визуальный шум: 3 ваших баббла подряд + 3 клиентских.

Реальный пример (2026-05-28): клиент отправил 3 PDF альбомом → у себя видит один альбом, в ClientCase прилетели 3 отдельных карточки одного за другим. Caption «назад три» висит только на одном баббле (на котором был closer-caption в TG).

## Признаки media_group в данных

Telegram гарантирует для всех апдейтов одной media_group:
- одинаковый `media_group_id` (явное поле в `update.message`)
- одинаковый `date` (timestamp в секундах)
- один автор
- идут подряд, обычно в пределах 0.5 сек прихода

То есть **эвристика «consecutive + same sender + same date»** покрывает 99% случаев без явного поля. Можно сделать и без `media_group_id` в БД.

## Два варианта реализации

### Вариант A — эвристика на UI (рекомендую)

**Что меняем:**
- `src/components/messenger/MessageList.tsx` — при рендере группировать подряд идущие сообщения, удовлетворяющие условиям:
  - один и тот же `telegram_sender_user_id`
  - один и тот же `telegram_message_date` (или разница < 2 сек как safety)
  - хотя бы у одного есть вложение в `message_attachments` (текст-only сообщения не группируем)
- Новый компонент `<GroupedAttachmentsBubble>` (или ветка в `MessageBubble.tsx`) — рендерит:
  - заголовок (имя + аватар + время) ОДИН раз
  - список вложений из всех строк группы (как сейчас рендерятся attachments одного баббла, только из N строк)
  - caption — берём первый непустой `content` из группы (исключая fallback `📎`)
  - реакции — объединение реакций со всех строк
  - **плашка `attachment_status='failed'`** — если у ЛЮБОЙ строки группы статус failed, показать единую плашку с агрегированным списком файлов из `attachment_error.failed_files[]` всех таких строк (см. требование ниже)

**Что НЕ меняем:**
- БД, миграции, RLS — ничего.
- `_shared/syncTelegramIncomingMessage.ts` — ничего (карантин не трогаем).
- Webhook'и telegram-webhook-v2, telegram-business-webhook, mtproto-service — ничего.
- Реакции, edit, delete — продолжают работать на отдельной строке. Если клиент удалил файл из альбома — соответствующая строка пропадёт из группы (UI просто покажет на одну меньше).

**Время:** ~1 час
**Риск:** 🟢 низкий — фронт-only, без миграций, без правки карантина

### Вариант B — колонка `project_messages.media_group_id`

**Что меняем:**
- Миграция: `ALTER TABLE project_messages ADD COLUMN telegram_media_group_id text; CREATE INDEX … (telegram_chat_id, telegram_media_group_id) WHERE telegram_media_group_id IS NOT NULL;`
- `_shared/syncTelegramIncomingMessage.ts` — добавить в `TgMessageMinimal` поле `media_group_id`, передавать в insertPayload.
- UI — то же, что в A, но группирует строго по `telegram_media_group_id` вместо эвристики.

**Плюсы B:**
- Надёжность 100%, не зависит от точности `date`.
- Можно делать SQL-аналитику по альбомам (сколько файлов в среднем, сколько failed на альбом).

**Минусы B:**
- Правка в карантинной зоне `_shared/` (читают TG group, TG business, MTProto).
- Backfill старых строк через `telegram_message_date + sender_id` группировку — лишний шаг.

**Время:** ~2-3 часа
**Риск:** 🟠 средний

## 🔴 Критическое требование — плашка о потерянных файлах

В рамках бага [2026-05-27-telegram-lost-attachments](../bugs/resolved/2026-05-27-telegram-lost-attachments.md) сделана плашка «⚠️ Файл из Telegram не загружен» при `attachment_status='failed'`. На media_group она работает **на каждый отдельный баббл**: если 1 из 3 файлов в альбоме не скачался, юзер видит 3 отдельных карточки, у одной из них красная плашка с именем непрогрузившегося файла.

При группировке в один баббл плашка **обязана** агрегироваться:

- Если хотя бы у одной строки группы `attachment_status='failed'` → показать плашку в общем баббле.
- Список непрогрузившихся файлов — объединение `attachment_error.failed_files[]` всех таких строк.
- Если ВСЕ файлы группы failed — показать соответствующий текст («Файлы из Telegram не загружены»).
- Если только некоторые — «Часть файлов из альбома не загрузилась: …».

**Без этой логики фича теряет смысл** — клиент думает что прислал альбом, в ClientCase виден баббл с 2 из 3 файлов, юзер не знает что 3-й потерян.

Тесты обязательны на этот сценарий перед мержем.

## Edge cases

| Кейс | Поведение |
|------|-----------|
| Caption на 1-м файле, остальные пустые | Caption — заголовок группового баббла. |
| Caption на разных файлах (редко, но возможно) | Брать первый непустой `content`, остальные игнорировать. |
| Клиент удаляет 1 файл из альбома | Соответствующая строка пропадает (через delete-webhook или soft-delete); UI рендерит группу с N-1 файлами. |
| Клиент редактирует caption на одном из файлов | Edit летит на конкретный telegram_message_id → `applyTelegramEdit` обновляет `content` нужной строки. UI берёт обновлённое значение для caption группы. |
| Реакция от клиента на конкретный файл альбома | TG позволяет реагировать на отдельные сообщения media_group. Реакция привязана к строке, в групповом баббле показывается общий ряд реакций со всех строк. |
| Multi-bot dedup в группе с workspace+employee | Каждый файл имеет уникальный `file_unique_id` → UNIQUE-дедуп `(chat, sender, date, file_unique_id)` в `_shared/syncTelegramIncomingMessage.ts` работает корректно. На каждый файл одна строка. |
| Read/unread | По `last_read_at` участника vs `created_at` каждой строки. Если 2 файла из 3 прочитаны, а 1 нет — UI должен показать красный контур на групповом баббле (любая строка непрочитана → весь альбом непрочитан). |
| Альбом 10 файлов (максимум TG) | Без ограничений в БД, UI должен скроллить список вложений или сжимать (например, 5 на ряд × 2 ряда). |
| Один файл сильно опоздал (через секунду) | Эвристика A может его не сгруппировать (разница в `date` > 0). Тогда отдельный баббл — приемлемый деградированный UX, не критично. Вариант B решает 100%. |

---

## Что НЕ покрывает эта задача

- Wazzup / WhatsApp множественные файлы. У них нет media_group — обычно один файл = одно сообщение. Если будут жалобы — отдельная задача.
- Email вложения. Они уже один email = одно сообщение со списком вложений, проблемы нет.

## Когда начинать

В новой сессии, на свежую голову. Рекомендую **вариант A** — фронт-only, без правки карантина. Если в процессе обнаружится что эвристика по `date` ненадёжна (на проде есть случаи когда у альбома разные timestamp'ы) — мигрировать в B отдельной задачей.

**Не делать в одной сессии с другими фичами мессенджера** — изменения в `MessageList.tsx` затрагивают рендер всех сообщений, регрессия легко проскользнёт.
