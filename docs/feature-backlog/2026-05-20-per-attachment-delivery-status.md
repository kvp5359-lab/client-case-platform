# Per-attachment delivery status (частичная доставка)

**Дата заведения:** 2026-05-20
**Связано:** [2026-05-20-unify-attachments-send-path.md](./2026-05-20-unify-attachments-send-path.md)

## Проблема

Сейчас статус доставки сообщения — на уровне `project_messages`
(`telegram_message_id`, `message_send_failures`, общий флаг
`deliveryFailed` на фронте). Это значит, что бабл рисуется одним из
двух состояний: «всё ОК» или «всё провалилось».

В реальности возможна **частичная доставка** — текст ушёл, файл нет
(или наоборот). Тогда фронт показывает красную рамку и «Повторить
отправку» по всему баблу, хотя текст уже у клиента. Это сбивает
пользователя — он думает, что нужно переотправлять целиком, а на
самом деле — только файл.

Случилось на тесте отложенной отправки 2026-05-20: cron-путь отправил
текст, но файл не отправился (см. бэклог-задачу
`unify-attachments-send-path.md`). Бабл выглядел как полностью
проваленный, хотя клиент в Telegram текст уже видел.

## Желаемое поведение

Два различимых состояния:

- **Полный фейл** — ни текст, ни файлы не доставлены. Текущий красный
  бабл + кнопка «Повторить отправку» (повторить всё).
- **Частичный фейл** — текст ушёл (`telegram_message_id` есть), но
  у конкретного вложения статус `failed`. Бабл нормальный (как
  обычное отправленное), маркер «не отправлено» **на самом
  вложении** + кнопка повтора только для файла.

## Что нужно сделать

1. **БД**:
   - Добавить колонку `delivery_status text` в `message_attachments`
     (`pending` / `sent` / `failed`).
   - Опционально: `delivery_error text` для текста ошибки.
   - Дефолт `pending` для новых строк; миграция должна проставить
     `sent` существующим, чтобы не пугать ретро.
2. **Edge functions** (карантин — аккуратно):
   - `telegram-send-message` при `attachments_only=true` — при успехе
     каждого attachment ставит `sent`, при ошибке — `failed`.
   - `telegram-business-send` — то же.
   - `wazzup-send` — то же.
   - MTProto — пока «частично» по матрице, можно отложить.
   - Email (`email-internal-send`) — у Resend/Gmail attachments идут
     в одном POST; либо всё ОК, либо всё нет → можно просто копировать
     общий `delivery_status` сообщения.
3. **Фронт**:
   - В `MessageAttachments` (или соседнем компоненте, который рисует
     карточки файлов) — у каждого файла показывать иконку статуса.
     Например: 🔴 + tooltip «Не удалось отправить» + маленькая кнопка
     «↻» только на этой карточке.
   - В `MessageBubble`:
     - `deliveryFailed` теперь = «текст не дошёл» (нет
       `telegram_message_id` + есть failure по message-level), а не
       агрегат.
     - Красная рамка и общая кнопка «Повторить отправку» только при
       полном фейле.
4. **Retry-логика**:
   - Существующая `retryTelegramSendMutation` повторяет всё сообщение.
     Нужно добавить вариант «retry конкретного вложения» —
     дёргает edge function с `attachments_only=true` для одного
     attachment id (или поднимать только те, у которых
     `delivery_status='failed'`).

## Риск

Карантин (все 5 каналов мессенджера). Затрагивает существующую логику
`deliveryFailed` на фронте и `message_send_failures` в БД. Нужно
аккуратно с тем, чтобы существующие message-level ошибки продолжали
показываться красным (не сделать «всё нормально» там, где раньше был
полный фейл).

## Зависимости

- Лучше делать **после** или **вместе** с задачей унификации
  (`unify-attachments-send-path.md`) — там как раз правится поведение
  edge functions при `attachments_only`. Если делать вместе, можно
  сэкономить один обход карантина.

## Сейчас (без этой задачи)

- Частичный фейл бывает редко (cron-кейс уже починен).
- Если случится — UI красным, юзер удаляет и переотправляет руками.
  Не блокирует работу, но визуально путает.
