# План: масштабирование «Входящих» — материализованное состояние + Broadcast realtime

**Статус:** черновик плана, к реализации не приступали.
**Зона:** карантин (мессенджер) — делать строго по фазам со смок-тестом и откатом.
**Дата:** 2026-06-17.

---

## 1. Цель

Сохранить все 4 свойства «Входящих», но сделать их дёшево (масштабируемо до ~1000–2000 пользователей, онлайн в пик ~50–150):

1. **Быстро** — открытие списка ≤ ~50 мс вместо нынешних 700 мс.
2. **Динамически** — списки и бейджи обновляются в реальном времени.
3. **Все непрочитанные** — вкладка «Непрочитанные» показывает полный набор, без потолков.
4. **Онлайн-пересчёт бейджей** — счётчики непрочитанности обновляются мгновенно у всех.

Замеры, обосновавшие задачу (2026-06-17, прод, ws `client-case`, 1157 тредов):
- `get_inbox_threads_v2` — **574 мс** на вызов (EXPLAIN), 72k buffer hits.
- Нагрузочный тест: потолок ~**2.6 загрузок Входящих/сек**; на 8 параллельных список = **2.4 сек**. Открытие самого треда стабильно ~80 мс (НЕ узкое место).
- `max_connections=60` → инстанс маленький (~1–2 ядра).

## 2. Корень проблемы (по снятым телам функций)

### 2.1. Каждый вызов = полный скан воркспейса
`get_inbox_threads_v2` (SQL, STABLE SECURITY DEFINER) на **каждый** вызов выполняет ~15 CTE, каждый сканирует все `project_messages` / `message_reactions` / `audit_logs` воркспейса (DISTINCT ON, GROUP BY, коррелированные подзапросы), затем ~20 LEFT JOIN. Стоимость = O(размер воркспейса).

**Все обёртки зовут v2 целиком:**
- `get_inbox_threads_page` — `FROM get_inbox_threads_v2(...) t` + keyset поверх (считает ВЕСЬ набор, отдаёт 50).
- `get_inbox_unread_threads`, `get_inbox_needs_reply_threads`, `get_inbox_awaiting_reply_threads` — каждый `FROM get_inbox_threads_v2(...) t WHERE ...` (ещё по полному скану).
- `get_inbox_thread_aggregates` — отдельный, более лёгкий, но тоже полный скан (~118 мс).

### 2.2. Realtime умножает на поток сообщений × онлайн
`useWorkspaceMessagesRealtime` (Postgres Changes на `project_messages`/`message_reactions`/`project_threads`, throttle 400 мс) на **каждое** событие инвалидирует **7 ключей** → рефетч `threads`+`aggregates`+`unread`(+needs/awaiting если вкладка активна). То есть одно входящее = 2–3 полных скана **на каждого онлайн-пользователя**.

Плюс (находка из индустрии): сам Postgres Changes **не масштабируется** — Supabase проверяет доступ (RLS) для каждого подписчика на каждое событие. На 50–150 онлайн это второй тормоз, уже в слое realtime.

**Формула катастрофы:** `нагрузка = размер_воркспейса × поток_сообщений × число_онлайн`.

## 3. Целевая архитектура

Индустриальный стандарт (Stream/Slack/Intercom): **материализованная модель чтения** + **счётчики «фан-аут на запись»** + **Broadcast-realtime**. У нас в треде мало участников (несколько сотрудников), поэтому фан-аут без проблемы «больших каналов» — подход идеален без гибридных оговорок.

### 3.1. Таблица A — `thread_inbox_meta` (общая, 1 строка на тред)
Предпосчитанные поля треда, одинаковые для всех зрителей. Снимает ~80% стоимости v2 (весь блок «последнее сообщение / событие / реакция / вложение / counterpart / сегмент»).

Колонки (черновик):
- `thread_id` PK
- `last_message_id`, `last_message_at`, `last_message_text`, `last_sender_participant_id`, `last_sender_role`
- `last_message_attachment_name`, `last_message_attachment_mime`, `last_message_attachment_count`
- `last_event_at`, `last_event_action`, `last_event_status_color`, `last_event_actor_user_id`
- `last_reaction_emoji`, `last_reaction_at`, `last_reaction_message_preview`, `last_reactor_participant_id`/`telegram_user_id`
- `channel_type` (telegram/email/web), `counterpart_*` (имя/аватар собеседника), `email_contact`, `email_subject`
- `last_from_staff` (bool, для сегмента), `has_external` (bool, есть сообщение из внешнего source)
- `sort_at` = `COALESCE(inbox_sort_at, GREATEST(last_message_at,last_event_at), created_at)`
- (имя/иконка/акцент/тип/project_id — можно держать тут или брать из `project_threads` лёгким JOIN по PK)

Обновляется триггерами на `project_messages`, `message_reactions`, `audit_logs`.
**Имена/аватары** (sender/counterpart/email) можно резолвить на чтение лёгким JOIN по id (как сейчас), чтобы не дублировать и не протухать — обсудить на этапе дизайна.

### 3.2. Таблица B — `thread_unread_state` (персональная, 1 строка на (participant, thread))
Непрочитанность у каждого своя → фан-аут.

Колонки:
- `participant_id`, `thread_id` (PK пара)
- `unread_count`, `unread_event_count`, `unread_reaction_count` (int, инкрементальные)
- `manually_unread` (bool), `last_read_at` (timestamptz)
- (наличие строки = доступ есть; точка отсчёта непрочитанного = момент создания строки)

**Семантика непрочитанного сохраняется 1:1 с текущей формулой v2:**
- сообщение от НЕ-меня, `source != 'telegram_service'`, после `last_read_at` → `unread_count`
- событие (`audit_logs`, не моё, не silent-status-переход) после `last_read_at` → `unread_event_count`
- реакция не моя после `last_read_at` → `unread_reaction_count` / `has_unread_reaction`
- `manually_unread` — ручной флаг (как сейчас)

**Точка отсчёта = момент выдачи доступа** (как договорено 2026-06-09): строка создаётся с `last_read_at=now()`. Триггеры засева доступа уже есть (`seed_read_status_on_{project_access,thread_member,assignee}`, миграция `20260609_unread_baseline_seed_on_access.sql`) — переиспользуем/дополняем для `thread_unread_state`.

### 3.3. Чтение
- **Список Входящих** = `thread_unread_state tus JOIN thread_inbox_meta tim ... WHERE tus.participant_id = :me ORDER BY tim.sort_at DESC, thread_id DESC` + keyset. Индексировано → ~единицы-десятки мс.
- **Бейджи/счётчики/сегменты** = из этих же строк (`unread_count>0`, `last_from_staff`, `has_external`), без отдельных полных сканов.
- **Все непрочитанные** = `WHERE unread_count>0 OR unread_event_count>0 OR ... OR manually_unread` — полно и точно, без потолков (снимает и риск G9 — REST 1000-cap: либо строк мало, либо keyset-пагинация).

### 3.4. Realtime → Broadcast из БД
Перевести `useWorkspaceMessagesRealtime` c Postgres Changes на **Broadcast** (`realtime.broadcast_changes()` из триггера, приватный канал на пользователя/воркспейс). Триггер шлёт уже готовый патч inbox-строки → клиент применяет его в кэш напрямую (как уже делает на mark-read), **без** рефетча тяжёлых запросов и без RLS-проверки на подписчика. Масштабируется «до десятков тысяч» подключений.

## 4. Самый тонкий момент — права доступа

`thread_unread_state` хранит строки только для пар (participant, thread) с доступом. Поддержка при изменении доступа:
- добавили/убрали `project_participants` / `project_thread_members` / `task_assignees` → создать/удалить строки (триггеры засева уже частично есть).
- **view_all-админы / владельцы** видят ВСЕ треды проекта без явной записи участника. Фан-аут строк на все треды для них дорог и шумен.
  - **Открытый вопрос (решить на дизайне):** (а) материализовать и для админов (строки на все доступные треды), либо (б) для админов на «чужих» тредах считать непрочитанное гибридно/формулой-фолбэком (перекликается с отложенной Фазой 2 из ledger 2026-06-09 — «формула-фолбэк в get_inbox_threads_v2 для view_all-админов на тредах, где они не участники»).
- Сложный 3-way UNION доступа из v2 (проектный доступ / orphan owner-assignee-member / проектные треды по assignee-member) должен быть точно отражён в правилах засева — это главный источник потенциальных багов «вижу/не вижу тред».

## 5. Фазы внедрения (от дешёвого/безопасного к глубокому)

Каждая фаза самостоятельна, со смок-тестом и откатом. Старый `get_inbox_threads_v2` держим как фолбэк/сверку до подтверждения каждой фазы.

### Фаза 0 — realtime без over-invalidation (без схемы, низкий риск, обратимо)
- Перестать на каждое сообщение инвалидировать 7 тяжёлых ключей. Патчить кэш инкрементально из самого realtime-события (как уже делается на mark-read), либо инвалидировать только лёгкие агрегаты с увеличенным дебаунсом.
- Эффект: убирает множитель «×поток сообщений» из дефолтного пути. Самый дешёвый выигрыш.
- Смок: новое сообщение → список/бейдж обновился без всплеска запросов (проверить вкладкой Network + perf_traces).

### Фаза 1 — `thread_inbox_meta` (общие поля)
- Создать таблицу + триггеры на messages/reactions/audit. Бэкафилл из существующих данных.
- Переписать `get_inbox_threads_v2` (и/или `_page`/`_aggregates`) на чтение из `thread_inbox_meta` вместо полного скана (имена/аватары резолвить JOIN по id).
- Эффект: 574 мс → десятки мс на общей части.
- Смок: список и превью идентичны старым (сверка строк old v2 vs new), реакции/события/вложения отображаются верно.

### Фаза 2 — `thread_unread_state` (персональные счётчики, фан-аут)
- Создать таблицу + триггеры инкремента (message/reaction/audit insert → +1 нужным участникам; mark-read → reset; mark-unread → manually_unread). Бэкафилл + засев по доступу.
- Решить вопрос view_all-админов (см. §4).
- Переключить unread/бейджи/сегменты/кнопку «Прочитано» на чтение из этой таблицы.
- Эффект: финальный потолок; «все непрочитанные» и онлайн-бейджи — дёшево и точно.
- Смок: счётчики совпадают с эталоном (old формула) на выборке; mark-read/unread мгновенны; точка отсчёта = выдача доступа (новому участнику не валятся тысячи фантомных).

### Фаза 3 — Broadcast realtime
- Триггеры шлют патч inbox-строки через `realtime.broadcast_changes()` в приватный канал; клиент применяет патч.
- Эффект: снимает RLS-per-subscriber и over-invalidation; масштабирует слой реального времени.
- Смок: у двух онлайн-пользователей списки/бейджи оживают мгновенно; нагрузка на realtime не растёт с числом подписчиков.

### Сквозное (на всех фазах)
- **Сверочный/бэкафилл-джоб** (ночной pg_cron): пересчёт эталонной формулой и сравнение с материализованными значениями → метрика дрейфа + автопочинка. Источник правды — нормальные таблицы; материализация перестраиваема.
- **Наблюдаемость:** счётчик расхождений, runbook «как пересобрать».
- **MVCC-bloat:** частые +1/−1 плодят мёртвые версии строк → выставить `fillfactor` (~70–80) на `thread_unread_state`/`thread_inbox_meta`, рассчитывать на HOT-update (не индексировать часто меняющиеся счётчики), следить за autovacuum.

## 6. Риски и заметки

- **Карантин:** мессенджер — самая чувствительная зона. Любая фаза — отдельная сессия, смок-тест по матрице каналов (TG group/Business/MTProto/Wazzup/Email), откат наготове.
- **Точность непрочитанного** — главный класс багов. Держать старую формулу как эталон сверки минимум до стабилизации.
- **Drift repo↔prod:** тела inbox-функций в проде расходятся с миграциями — менять, снимая ЖИВОЕ тело из БД, применять через MCP, фиксировать миграцией.
- **Гранты/RLS:** при DROP+CREATE inbox-функций восстанавливать гранты 1:1 (default privileges возвращают `anon` — снимать). 7 функций тянут `t.*` из v2 — менять согласованно.
- **Контракт фронта:** список query-ключей и патчеров (`patchInboxThreadInCache`/`patchInboxAggregateInCache`, `invalidateMessengerCaches`) — обновить под новый realtime-патч-поток.
