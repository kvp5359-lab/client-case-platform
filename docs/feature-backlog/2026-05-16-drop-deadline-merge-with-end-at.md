# Полный переход: убрать `project_threads.deadline`, использовать `end_at`

**Создано**: 2026-05-16
**Статус**: backlog (приоритет — после обкатки календаря)

## Контекст

При внедрении календаря 2026-05-16 ввели поля `project_threads.start_at` и
`end_at`. Юзер выбрал концепцию «у задачи — только время начала и конца, без
отдельного дедлайна» (Вариант А из обсуждения).

Сейчас сделан **быстрый промежуточный шаг**: триггер
`sync_thread_deadline_end_at` (миграция `20260516_sync_thread_deadline_end_at.sql`)
синхронизирует `deadline` ↔ `end_at` автоматически — двусторонне.

Поле `deadline` остаётся в БД для обратной совместимости, но логически
дублирует `end_at`.

## Что нужно сделать в рамках полного перехода

### БД

- Все RPC, фильтры, индексы переписать с `deadline` на `end_at`. Затронуто
  86 упоминаний в `supabase/migrations/` (на 2026-05-16).
- Удалить триггер `sync_thread_deadline_end_at` и саму функцию.
- Удалить колонку `project_threads.deadline`.
- Все индексы по deadline пересоздать по end_at.

### Фронт

- 38 файлов TypeScript/TSX используют `.deadline`. Заменить на `.end_at`
  (или вычисляемое поле, если оставим nullable семантику).
- `useUpdateThreadTime`, `useUpdateThreadDeadline` объединить в один хук.
- В UI: «дедлайн» как термин остаётся, но читается из `end_at`.
- Фильтры календарь/просрочка/сегодня — везде по `end_at`.

### Edge functions

- Прогрепать `supabase/functions/` на `deadline` — пересмотреть AI-промпты,
  digest-генерацию, экспорты.

## Подводные камни

- **start_at для задач без календаря**: для задач, у которых был только
  `deadline=пятница 18:00` без start_at — что с ними? Варианты:
  - оставить `start_at IS NULL` и трактовать как «срок без слота» (как
    сейчас, но без `deadline` — `end_at` будет хранить срок без start_at);
  - принудительно проставить `start_at = end_at - 30мин` при миграции.
- **Концептуально**: если задача имеет только `end_at`, она вроде бы должна
  быть в календаре (попадает в выборку по `start_at IS NOT NULL AND end_at
  IS NOT NULL`). Если хотим «срок без слота» — выборка для календаря должна
  быть `start_at IS NOT NULL`, а просто наличие `end_at` календарь не
  показывает.
- **External API / экспорты**: если кто-то читает `deadline` через REST API
  напрямую (внешние интеграции, скрипты) — сломается.
