## Идея

Юрист в проекте/лиде опрашивает клиента. По структурированным ответам система
показывает, какие **виды на жительство (ВНЖ)** подходят: ✅ подходит / 🟡 частично /
⬜ не подходит / ❔ не отвечено. Критерии берутся из **общей экспертной базы знаний**,
той же, что питает публичные сайты экосистемы.

Два режима опроса (один UI): **широкий** (все вопросы страны → что вообще подходит) и
**узкий** (отметил интересующие ВНЖ → вопросы сужаются только под них).

│ глобально, на уровне воркспейса        │   ет →   │ внутри конкретного проекта/лида   │
│ • новая вкладка «Подбор» в Базе знаний │         │ • вкладка «Анкеты» → секция        │
│   (рядом Дерево / Таблица / Q&A)       │         │   «Подбор ВНЖ» (аккордеон)         │

- **Контур 1** = «что бывает и кому что нужно». Общий справочник, один на все проекты.
  Живёт **новой вкладкой «Подбор»** на странице `/settings/knowledge-base`.
- **Контур 2** = «этот клиент по этим ответам». Свой в каждом проекте.

⚠️ Вкладка «Подбор» читает **внешнюю базу** (mig-modules), а соседние вкладки Базы

одной странице — норм, но архитектору знать.
**Бонус-связь:** карточку ВНЖ из вкладки «Подбор» связать со статьёй-гайдом в Дереве
(`residence_type → knowledge_article`). Структура + текст в один клик. У вас уже есть
статьи по Arraigo / Cuenta Propia — к ним и привязать.

## Концептуальная модель (как сформулировано)

```
Страна
 └─ Вид ВНЖ
     └─ дерево условий (И/ИЛИ) из критериев:
         • Критерий (ОБЩИЙ, лежит в Группе)
             ├─ название («Доход»)
             ├─ тип ответа (число / выбор / да-нет / справочник)
             └─ ☑ анкетируемый? → текст вопроса
                 (если флаг стоит — критерий показывается как вопрос в подборе)
         • связка ВНЖ↔критерий: оператор + значение (порог) + важность (критично/желательно)

«Текущий статус» = ОБЫЧНЫЙ критерий (выбор из справочника ВНЖ + «нет статуса»).
   Переход «откуда куда» = условие на этом критерии: текущий статус ВХОДИТ В [...].
```

**Решения по упрощению (зафиксировано):**
- Критерий **общий** (заводится раз, в группе), порог — в связке с конкретным ВНЖ.
  Один критерий используется многими ВНЖ → в анкете спрашивается один раз.
- **Группы** критериев — обязательны (для матрицы-обзора и для «добавить в группу»).
- **И/ИЛИ-дерево** условий на ВНЖ — оставляем (как было). Одно дерево на ВНЖ,
  именованные правила НЕ используем.
- **Важность** условия: `критично` → ⬜ не подходит; `желательно` → 🟡 частично.
- **Переходы** между ВНЖ — через критерий «Текущий статус», **без отдельной матрицы**
  (заметки на пары переходов не нужны).
- **Процедуры** (способ оформления одного ВНЖ — «из-за рубежа» / «из Испании», разные
  сроки/условия) — **не используем как уровень**. Вместо них **разные пути = отдельные
  виды ВНЖ** (напр. «Cuenta propia — первичный» и «Cuenta propia — модификация»; «Цифровой
  кочевник 3 года изнутри» и «виза цифрового кочевника 1 год из-за рубежа»). Так у каждого
  вида **одно правило**, редактор без дерева-по-процедурам. Подход уже частично в данных
  (студвиза разделена на «высшее»/«языковые»).
  - **Существующие 7 многопроцедурных ВНЖ** (Cuenta propia, цифр.кочевник и др.) делит на
    виды **сам пользователь вручную, позже**. Авто-скрипт НЕ делаем. До разделения матрица
    показывает условия объединённо (значок `*` = расхождение по процедурам).

### Отвергнутые варианты (чтобы не возвращаться)

- **Доработать standalone migchoice** — отрывает подбор от клиента/кейса в client-case.
- **Своя база ВНЖ в client-case (per-workspace)** — заново вбивать десятки видов;
  экосистема одна. Оправдано только при сторонних юрфирмах со своей экспертизой.
- **Гибрид: обогатить анкету-бриф критериями** — две несовместимые задачи на одной
  форме. Отдельный модуль проще.
- **Отдельная сущность переходов / процедур / именованных правил** — лишние слои;
  всё выражается критериями + одним деревом (см. «Концептуальная модель»).

## Модель данных

### Внешняя общая база (схема `mod_choice`)

Обращение: `client.schema('mod_choice').from('...')`. Ключевые таблицы как есть:
- `countries` — страны.
- `residence_types` — виды ВНЖ (`country_id`).
- `criteria` — критерии: `field_key`, `field_type` (number/boolean/reference/text),
  `options`, `group_id`, `country_id`.
- `criteria_groups` — группы критериев. ✅ (нужны для матрицы и добавления)
- `rules` — условия: `link_id` + `rule_json` (И/ИЛИ-дерево, операторы, `severity`).
- `links`, `procedures`, `residence_statuses`, `family_statuses`, `sessions`,
  `selections` — **не используем** в client-case (см. ниже).

**Что доработать в общей схеме (аддитивно — relostart НЕ ломаем):**
- Добавить в `criteria` флаг **`is_askable`** (+ при желании `question_text`) —
  показывать ли критерий как вопрос. Новая колонка, обратная совместимость сохранена.

**Что НЕ ломаем (важно для relostart/migchoice.com):**
- `procedures` / `links`-уровень и `residence_transitions` оставляем в схеме как есть —
  просто **client-case их не показывает** (процедуру берём дефолтную/скрытую; переходы —
  через критерий «Текущий статус»). Так общая база не страдает, а UI client-case проще.
- Именованные правила: модель `rules` оставляем, но client-case ведёт **одно правило
  (дерево) на ВНЖ**.

> Это снимает прежнюю развилку «перестроить общую базу vs своя структура»: **не
> перестраиваем** — добавляем один флаг и делаем упрощённое представление поверх.

### Новое в client-case

**`case_profiles`** — один профиль на проект:
- `id`, `project_id` (FK, UNIQUE для MVP), `workspace_id`
- `country_id` — UUID из общей базы (**без FK**, разные БД)
- `selected_residence_type_ids` — `uuid[]` (пусто = все)
- `answers` — `jsonb` `{ field_key: value }` (MVP; вынести в таблицу при нужде в аналитике)
- `result_snapshot` — `jsonb`: `[{ residence_type_id, name, status, score, failed_critical[], warnings[] }]`
- `computed_at`, `created_at`, `updated_at`, `created_by`

**RLS:** доступ участникам проекта/воркспейса (как `form_kits`).

## UI

### Контур 1 — вкладка «Подбор» (матрица-обзор)

```
База знаний → [Дерево][Таблица][Q&A][ПОДБОР]

Страна: [Испания ▾]                          [+ Вид ВНЖ]  [+ Критерий]
                              │ Стартап │ Цифр.кочевник │ Оседлость │
─── 📁 Базовое ───────────────┼─────────┼───────────────┼───────────┤
  Текущий статус              │ нет/коч │ нет/стартап   │ —         │
  Гражданство                 │ не-ЕС   │ не-ЕС         │ не-ЕС     │
─── 📁 Финансы ───────────────┼─────────┼───────────────┼───────────┤
  Доход (год)                 │ ≥30 000 │ ≥25 000       │ —         │
  Капитал                     │ ИЛИ≥100k│ —             │ —         │
```
- Строки — критерии по 📁 группам; столбцы — виды ВНЖ; ячейка — оператор+значение / `—`.
- Сложное «ИЛИ» в ячейке помечаем, клик → дерево условий.
- Пустая ячейка → добавить критерий в ВНЖ. `+ Критерий` спрашивает группу.
- Для старта — только матрица; под-виды (таблица видов, карточка ВНЖ) — потом.

**Два РАЗНЫХ действия (не путать):**
- **Создать критерий** = общая сущность (без привязки к ВНЖ). Доступен всем ВНЖ.
- **Назначить на ВНЖ** = в ячейке задать порог+важность для конкретного ВНЖ (связка).

**Дефолты под-форм Контура 1 (приняты, UI докрутим):**
- **Создание/правка критерия** — боковая панель справа (матрица видна). Поля: название,
  группа (выбор/создать на лету), тип ответа (число / выбор / да-нет / справочник),
  ☑ анкетируемый → текст вопроса, варианты (для «выбора»), опц. подсказка для юриста и
  единица измерения. Тип «справочник» для старта — только «виды ВНЖ» (для «Текущего
  статуса»); гражданства/образование — обычным «выбором из списка».
- **Создание/правка ВНЖ** — модалка: название, категория, описание, ссылка на статью KB (опц).
- **Правка ячейки** — мини-попап (оператор + значение + важность) для простого условия;
  кнопка «сложное условие» → дерево И/ИЛИ (порт `RuleGroupEditor`).
- **Группы** — кнопка «Группы»: список с сортировкой и переименованием.

## Поток данных (Контур 2)

1. Юрист → проект → «Анкеты» → разворачивает «Подбор ВНЖ».
2. Выбирает страну → фронт тянет из общей базы `residence_types` + `criteria` + `rules`.
3. *(опц.)* отмечает ВНЖ → вопросы сужаются (`выбранные ВНЖ → rules → field_keys → criteria`,
   только `is_askable`).
4. Заполняет → `case_profiles.answers`.
5. Фронт live гоняет `rule-evaluator` → превью.
6. «Рассчитать» → **Edge `residence-match`** → официальный результат → `result_snapshot`.
7. *(будущее)* выбрал ВНЖ → создать кейс из `project_template`.

---

## Контракт Edge `residence-match`

- **Вызов:** фронт, JWT юриста. (Заложить вызов по `profile_id` без юзер-контекста — на будущую публичность.)
- **Вход:** `{ profile_id }`.
- **Действия:** читает профиль+answers (client-case) → читает `rules`/`residence_types`
  (общая база, по `country_id` и `selected_residence_type_ids`) → `rule-evaluator` →
  пишет `result_snapshot`+`computed_at` → возвращает результат.
- **Доступ к общей базе:** env `MODULE_SUPABASE_URL` + ключ (anon на чтение; запись
  критериев — отдельный защищённый ключ, только в Edge).

## Риски / зависимости

- **Внешняя БД на free-tier — АВТОПАУЗА.** Проект засыпает после ~7 дней без запросов →
  DNS пропадает (NXDOMAIN) → подбор падает с `Failed to fetch`. Урок: NXDOMAIN ≠ удалён,
  это и пауза тоже. Снять риск — Pro-план или миграция в client-case (отложено).
- **Env для прода client-case** — `NEXT_PUBLIC_MODULE_SUPABASE_URL` / `..._ANON_KEY` сейчас
  только в локальном `.env.local`. Перед деплоем добавить в окружение прода client-case
  (docker-compose/GitHub secrets), иначе на проде подбор не подключится.
- **Внешняя БД** — client-case зависит от доступности внешнего проекта и env.
- **Нет FK между базами** — ID видов хранятся строкой; «висячий» ID при удалении вида в
  общей базе. Снимок — копия на момент расчёта, ок; «живые» данные — проверять наличие.
- **Изменение общей схемы** (`is_askable`) — аддитивно, но требует согласования, т.к.
  база общая с relostart/migchoice.com.

### ⏸ Осознанно отложено (с причинами)
- **Правка ячеек (назначение порога ВНЖ↔критерий)** — это манипуляция деревом `rule_json`
  на боевой базе. Делать вслепую без прогона рискованно → следующая итерация с автором.
- **Главный feature-flag `visa_selection`** — выключенный по умолчанию флаг скрыл бы
  вкладку. Ставить вместе с Контуром 2 (общий рубильник модуля).
- **Флаг `is_askable` у критерия** — нужна новая колонка во внешней базе (DDL через
  REST нельзя; добавить в Supabase-дашборде/через MCP внешнего проекта). Пока все
  критерии анкетируемые (как в migchoice).
- **Резолв имён в reference-ячейках** (Текущий статус показывает «…») — нужен справочник
  значений; MVP показывает заглушку.
