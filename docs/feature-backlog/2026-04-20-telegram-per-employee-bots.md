# Персональные Telegram-боты для сотрудников

**Статус:** на обдумывании
**Добавлено:** 2026-04-20

## Идея

Сейчас клиент в Telegram-группе видит все исходящие сообщения от одного рабочего бота с общей аватаркой — идентификация сотрудника только по имени в тексте сообщения.

Задумка: создать отдельного Telegram-бота на каждого сотрудника с его аватаркой и именем. При отправке из сервиса клиент видит сообщение от «настоящего» сотрудника — правильная аватарка, правильное имя, без приставки «bot».

## Как это работает технически

### Модель ботов в группе

- **1 бот-workspace** (главный, как сейчас) — privacy OFF, слушает всю группу, синхронизирует входящие сообщения клиента в сервис.
- **N ботов-сотрудников** — privacy ON, только отправляют сообщения от имени сотрудника. На входящие сообщения не реагируют.
- **(Опционально) 1 бот-технический** — команды самообслуживания для клиента: `/status`, `/docs`, `/pay`.

В группу добавляются: `workspace_bot` + 1-2 `employee_bot` релевантных проекту + (опционально) `technical_bot`.

### Имена ботов в Telegram

У Telegram-бота два имени:
- **@username** (handle) — обязан оканчиваться на `bot`. В ленте сообщений **не показывается**, виден только в профиле.
- **Display name** — любое, задаётся в BotFather через `/setname`. Именно оно отображается как «отправитель» в чате.

Значит клиент видит «Денис Крылов» без «bot» — как обычного участника.

### Код

Весь код — общий. Нужно:

1. Таблица `telegram_bots` в БД:
   ```
   id uuid
   workspace_id uuid
   owner_user_id uuid null   -- null для workspace/technical ботов
   bot_type text            -- 'workspace' | 'employee' | 'technical'
   token text               -- секрет
   telegram_bot_username text
   display_name text
   created_at timestamptz
   ```

2. Webhook-эндпоинт принимает апдейты от всех ботов на один URL-путь вида
   `/telegram-webhook/{bot_id}` (так надёжнее, чем определять по токену в теле).
   Edge Function по `bot_id` находит запись в `telegram_bots` и ветвит логику:
   ```
   if (bot.bot_type === 'workspace')  handleWorkspaceBot(update, bot)
   if (bot.bot_type === 'employee')   handleEmployeeBot(update, bot)
   if (bot.bot_type === 'technical')  handleTechnicalBot(update, bot)
   ```
3. При отправке сообщения из сервиса — сервис по `user_id` отправителя находит
   его `employee_bot`, берёт токен, шлёт сообщение через Bot API от этого токена.
   Если у сотрудника нет своего бота — fallback на workspace-бота с именем в
   тексте, как сейчас.

## Что нужно реализовать

- [ ] Миграция: таблица `telegram_bots` + RLS.
- [ ] Ручной onboarding: инструкция для админа — как создать бота в BotFather
  (create, `/setname`, аватарка, `/setprivacy`, `/setuserpic`) и добавить его
  в сервис (форма в настройках воркспейса: имя сотрудника + токен).
- [ ] Edge Function `telegram-webhook` — доработать routing по `bot_id` и
  ветвление по `bot_type`.
- [ ] Sender-selection в сервисе: при отправке сообщения выбираем токен бота
  текущего пользователя или fallback на workspace-бота.
- [ ] Процедура добавления ботов-сотрудников в клиентскую группу (админ делает
  вручную или бот-admin добавляет бота-сотрудника, если у него есть права).
- [ ] UI: раздел «Telegram-боты сотрудников» в настройках воркспейса — список,
  добавление, привязка к участнику.

## Операционная нагрузка

- **Создание бота в BotFather:** 5-10 минут на сотрудника, одноразово.
- **Лимит BotFather:** ~20 ботов на Telegram-аккаунт. Если упрётся — создавать
  ботов с другого рабочего Telegram-аккаунта.
- **При смене сотрудника:** либо переиспользовать бота (сменить имя и
  аватарку), либо заблокировать старого и создать нового.
- **В клиентские группы** нужно вручную добавлять релевантных ботов-сотрудников.

## Риски и ограничения

- **Telegram может прислать одно входящее сообщение нескольким ботам** (если
  бот упомянут или если privacy OFF). Для избежания дублей: у ботов-сотрудников
  privacy строго ON, слушает только workspace-бот.
- **Ответ клиента на сообщение сотрудника** — reply доставляется в webhook
  бота-отправителя. Сервис должен склеить это с общим тредом по `chat_id`.
- **Хранение токенов** — секрет, хранить в БД в зашифрованном виде или в
  Supabase Secrets, а в таблице — только ссылку.
- **Потеря бота** (удалили в BotFather, сменился токен) — обрабатывать
  ошибку 401 от Bot API и помечать бота как неактивный, fallback на
  workspace-бота.

## Альтернатива — MTProto (user-сессии сотрудников)

Сотрудник привязывает **личный Telegram-аккаунт** через Telethon/gramjs, и
сервис шлёт сообщения от его имени напрямую. Плюсы: настоящая аватарка и
имя, без приставок, без участников-ботов в группе. Минусы: сотрудник должен
привязать личный Telegram, риск бана аккаунта за «массовую автоматизацию»,
сложнее хранение и обновление session-токенов. Для небольшой команды
мульти-бот подход проще и безопаснее.
