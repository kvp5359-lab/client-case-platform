# Унификация пути отправки сообщений с вложениями

**Дата заведения:** 2026-05-20
**Контекст:** появилось при внедрении отложенной отправки. Текущая
асимметрия не блокирует фичу, но создаёт техдолг.

## Проблема

«Типовая процедура отправки» сейчас на самом деле двухступенчатая:

1. **Текст без файлов** — фронт делает INSERT в `project_messages`,
   триггер `notify_telegram_on_new_message` видит строку и шлёт через
   `dispatch_message_to_channels` → edge function канала.
2. **Текст с файлами** — фронт делает INSERT, триггер видит
   `NEW.has_attachments = true` и **сознательно ничего не делает**
   (early return). Фронт **отдельным вызовом**
   `supabase.functions.invoke('<channel>-send', { ..., attachments_only: true })`
   инициирует отправку файла.

Это создаёт две неприятные вещи:

- **Асимметрия каналов отправки**: один путь работает через триггер БД,
  второй — через прямой вызов с фронта. Любой новый сценарий (cron,
  retry, импersonированные мутации) должен помнить про обе ветки.
- **Отложенная отправка сломалась на этом**: cron-воркер
  `dispatch_scheduled_messages()` вызывает тот же диспетчер, что и
  триггер. Для запланированных с файлами фронта в момент отправки нет,
  поэтому файл не уходил. Пришлось добавить параметр
  `p_force_attachments boolean` в `dispatch_message_to_channels` —
  фактически тот же ручной костыль, который раньше делал фронт.
  См. миграцию `20260520_scheduled_messages.sql`.

## Желаемое поведение

Единый путь: фронт делает только INSERT и забывает. Триггер отправляет
**и текст, и вложения** через одну вызов edge function. Cron работает
ровно так же, потому что использует тот же диспетчер.

## Что нужно сделать

1. **В `dispatch_message_to_channels`**:
   - Убрать early return на `has_attachments = true`.
   - Для каналов, которые принимают `attachments_only` (TG group,
     Business, Wazzup) — всегда добавлять флаг в payload, когда у
     сообщения есть файлы.
   - Параметр `p_force_attachments` становится не нужен — удалить.
2. **На фронте** убрать прямые вызовы вторым шагом:
   - `src/hooks/messenger/useSendMessage.ts` — `attachments_only: true`-ветка.
   - `src/services/api/messenger/messengerDraftService.ts` —
     `publishDraftMessage` шлёт TG напрямую, тоже убрать.
   - `src/services/api/messenger/messengerService.ts` (или где это для
     Wazzup) — аналогичный блок с `wazzup-send`.
   - Также проверить `useDelayedSend.ts` — там тоже своя локальная
     отправка после публикации черновика.
3. **Смок-тесты** перед merge — обязательно прогнать **все 4 канала**:
   - TG group: текст / текст+файл / реплай / реплай+файл.
   - TG Business: текст / текст+файл.
   - TG MTProto: текст (вложения там и сейчас «частично», см. матрицу
     в `infrastructure.md`).
   - Wazzup: текст / текст+файл / quotedMessageId.
   - Email: текст / текст+файл.
4. **Идемпотентность**: убедиться, что edge-функции при вызове с
   `attachments_only=true` корректно отрабатывают, даже если уже была
   попытка отправки. Сейчас фронт всегда шлёт строго один раз, после
   изменения путь будет проходить через триггер — нужны защиты от
   двойной отправки.

## Риск

Карантинная зона мессенджера — баги в этой области исторически чинятся
дольше всех (см. `infrastructure.md`, раздел «RLS на `project_threads`»
и «Дедуп между несколькими ботами»). Любая правка в
`notify_telegram_on_new_message` / `dispatch_message_to_channels` —
точка регресса для всех 5 каналов сразу. Поэтому не вкатывать вместе с
другой работой, делать отдельным небольшим PR со смок-тестом.

## Сейчас работает корректно (без этой задачи)

- Обычная отправка с файлом — работает как и работала.
- Отложенная отправка с файлом — работает за счёт ветки
  `p_force_attachments=true` в SQL.
- Поведение для конечного пользователя идентично, просто симметрии в
  коде нет.
