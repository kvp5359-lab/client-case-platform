# Кастомные поля для проектов — ТЗ

**Дата:** 2026-05-01
**Статус:** ТЗ согласовано, ждёт реализации.
**Связь:** пререквизит к [CRM-фрейму](./2026-04-30-crm-frame.md), но **самостоятельная фича** — нужна для всех проектов, не только лидов.

## 1. Цель

Дать возможность прицеплять к проектам произвольные поля (источник, бюджет, тип ВНЖ, дата подачи и т.д.) — индивидуально под каждый шаблон проекта.

Сейчас механизм кастомных полей в сервисе **уже есть для форм, документов и справочников** (таблица `field_definitions`, страница `/templates/field-templates`, движок типов и валидаций). К проектам он не подключён — это и нужно сделать.

## 2. Принципы

- **Не изобретать движок полей с нуля** — переиспользуем существующий справочник `field_definitions`.
- **Поля общие на воркспейс**, а не отдельный набор под каждый шаблон проекта. Шаблон только **выбирает**, какие поля из общего справочника показывать в проектах этого типа. Иначе — дубли и рассинхрон при правках.
- **Полностью аддитивная фича.** Старые проекты без кастомных полей продолжают работать как раньше — секция полей просто пустая.

## 3. Шаги реализации

### Шаг 0. Баг-фикс: привязать справочник полей к юрфирме

**Проблема.** Сейчас страница «Шаблоны полей» лежит в настройках конкретной юрфирмы (`/workspaces/[id]/settings/templates/field-templates`), но в коде запрос грузит **все поля без фильтра по юрфирме**. Соответственно как только в сервисе появится вторая юрфирма — она увидит чужие поля. Это уже баг безопасности, не зависящий от CRM.

**Что сделать.**
- Привязать определения полей к юрфирме (новая колонка `workspace_id` в таблице `field_definitions`).
- Существующие поля при миграции привязать к текущему воркспейсу.
- Запрос на странице фильтровать по юрфирме.
- RLS на таблице — каждая юрфирма видит только свои поля.

**Это первый шаг, до всего остального**, потому что строить на сломанном фундаменте смысла нет.

### Шаг 1. Добавить тип поля «Справочник»

**Зачем.** Сейчас среди типов полей есть «Список значений» — но это **захардкоженный набор**, привязанный к конкретному полю. Если «Источник лида» используется в нескольких шаблонах — придётся в каждом поле заполнять список заново. И при добавлении нового источника — править в трёх местах.

**В системе уже есть пользовательские справочники** (`/directories/custom`) с произвольными значениями. Логично сделать тип поля, который ссылается на справочник. Один справочник — один источник правды.

**Что сделать.**
- Новый тип поля «Справочник» в существующих типах.
- При создании поля этого типа — выбор: на какой справочник ссылается.
- В UI рендера поля — выпадашка с актуальными значениями из справочника.

**Задел на будущее.** Аналогичная механика пригодится для:
- Поля «Контакт» (ссылка на справочник контактов из CRM-фрейма).
- Поля «Пользователь» (ссылка на участников воркспейса).

Можно сделать как один тип «Связь» с выбором источника, или несколько разных типов — это UX-вопрос, не архитектурный. На MVP достаточно типа «Справочник», остальное — наращивать по мере появления новых сущностей.

### Шаг 2. Прицепить поля к шаблонам проектов

**Где настраивать.** В редакторе шаблона проекта (`/templates/project-templates/[id]`) — новый раздел «Поля» (по аналогии с уже работающим разделом «Статусы»).

**Что в нём:**
- Кнопка «Добавить поле» → выпадашка с полями из общего справочника воркспейса + кнопка «Создать новое».
- Список выбранных полей с возможностью переупорядочить.
- На каждом поле флаги: обязательное / необязательное, опционально — значение по умолчанию.
- Опционально (на потом): группировка полей в секции внутри карточки.

**Под капотом** — связь «шаблон проекта ↔ поле» (отдельная таблица), без дублирования определений.

### Шаг 3. Показать поля в карточке проекта

**Где.** На вкладке **«Настройки»** проекта, отдельным блоком — рядом с уже существующими блоками «Основное», «Интеграции», «Участники». Эта вкладка фактически уже работает как «карточка проекта» (показывает статус, дедлайн, шаблон, описание, интеграции, участников), кастомные поля логично встают в ту же логику свойств проекта.

Название блока — «Поля» / «Дополнительно» / по названию шаблона (например, «Параметры лида»). Решается на этапе UI.

Переименование самой вкладки «Настройки» в более точное («Карточка», «О проекте», «Свойства») — опционально, не обязательно. Если решим переименовать — отдельным шагом.

**Логика.**
- Берёт шаблон проекта → грузит его список полей → рендерит каждое через существующий механизм рендеринга (тот же, что у форм).
- Заполненные значения сохраняются в новой таблице (значения полей конкретного проекта).
- При смене шаблона у проекта — старые значения остаются в БД (на случай возврата), но не показываются. Видны только поля текущего шаблона.

**Создание значений** — лениво (при первом редактировании поля), не предзаполняем пустыми записями всю таблицу.

## 4. Что НЕ делаем сейчас (на потом)

- **Композитные поля** (составные, как у форм) для проектов — на старте необязательно. Если нужны — добавим позже.
- **Поиск и фильтрация по кастомным полям** в досках/таблицах. Архитектурно нужно держать в уме (значения хранить так, чтобы фильтрация была возможна), но в MVP не делаем.
- **Группировка полей в секции** внутри карточки проекта — UX-улучшение, на потом.
- **Расчётные поля** (формулы) — отдельная история.

## 5. Риск ломки и модульность

**Полностью аддитивная фича.** Новые таблицы, новые компоненты, новый раздел в карточке проекта. Существующее не трогаем (кроме баг-фикса в шаге 0, который улучшает поведение).

**Не выделяем в отдельный платный модуль.** Кастомные поля — слишком фундаментальная фича, должны быть в базовом тарифе.

## 6. Порядок работы

1. **Шаг 0** — починка привязки полей к юрфирме. Маленький, но обязательный.
2. **Шаг 1** — тип поля «Справочник». Маленький.
3. **Шаг 2** — раздел «Поля» в редакторе шаблона проекта. Средний.
4. **Шаг 3** — рендер и сохранение полей в карточке проекта. Средний.

Общая оценка по объёму: **одна-две недели работы** при последовательном выполнении. Каждый шаг — самостоятельный релиз, можно катить по очереди.

## 7. Открытые вопросы

- **Тип «Справочник» как один тип или несколько** (отдельные «Контакт», «Пользователь») — решим при реализации шага 1.
- **Группировка полей в секции внутри карточки** — нужна ли в MVP или отложить.
- **Где именно в карточке проекта показывать поля** (шапка / боковая панель / отдельная вкладка) — UX-вопрос, проработать на этапе шага 3.
