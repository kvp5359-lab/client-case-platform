# Модуль «План» — наглядный план проекта для клиента

**Дата создания:** 2026-05-30
**Тип:** новый модуль проекта (фича)
**Приоритет:** средний
**Статус:** Фазы 0–4 реализованы (2026-05-30). Отложено на доработку:
slot-блоки в шаблоне плана (две системы шаблонов слотов — нужен аккуратный
резолв) и вкладка «только План» без модуля «Задачи» (правка таб-системы
ProjectPage). Миграция применена на проде, типы перегенерированы.

---

## Зачем

Сейчас по проекту есть только плоский список задач. Клиенту и команде этого мало — нужен
наглядный **план-витрина** с пояснительным текстом, шагами и чек-листом документов
(пример — текстовые планы, которые сейчас вручную отправляем клиенту в мессенджер).

«План» = документ-конструктор из трёх типов блоков:

- **текст** — разделы, пояснения, стратегия;
- **задача** — ссылка на существующую задачу проекта (живой статус, срок);
- **документ (слот)** — ссылка на существующий слот документа (загрузка файла, статус, срок).

Задачи и слоты **не копируются**, а подтягиваются по ссылке — одна точка правды,
статусы меняются разом везде.

## Ключевые решения (итог обсуждения)

1. **Где живёт.** Не отдельная вкладка. На вкладке «Задачи», **под списком задач** —
   сворачиваемый блок «План». Логика задач не трогается.
2. **Независимость от задач.** Фильтры/сортировка/режим список-календарь действуют только
   на список задач сверху. План снизу независим, со своим фиксированным авторским порядком.
   Это снимает конфликт «текст между задачами ломается при фильтре/сортировке».
3. **План — полноценный модуль** наравне с `tasks`. Вкладка видна, если включён **хотя бы
   один** из модулей (`tasks` или `plan`). Содержимое собирается из доступных кусков:

   | Включено для роли | Что видит |
   |---|---|
   | tasks + plan | список задач + блок плана |
   | только tasks | список задач |
   | только plan | **только план** (рабочий список скрыт) |
   | ничего | вкладки нет |

4. **Два слоя видимости.** Модуль `tasks` управляет рабочим списком задач. Галочка
   «виден клиенту» на блоке управляет тем, что из плана видит клиент. **Задача-кубик
   внутри плана** показывается как часть плана (управляется своей галочкой), независимо
   от модуля `tasks` — иначе план развалится.
5. **Клиенту — урезанно.** Задача-кубик клиенту: только название + готово/не готово.
   Клиентский экран плана — чистый, без рабочего списка задач.
6. **Шаблонизация.** «Рыба» плана собирается в шаблоне проекта и разворачивается
   автоматически при создании проекта, привязываясь к реальным задачам/слотам клиента.
7. **Срок у слота.** Поле `deadline` у слота **уже есть в БД** (`folder_slots.deadline`,
   также `assignee_id`, `status`) и уже загружается (`useFolderSlots` тянет `*`). Просто
   не выводится в UI. В плане показываем/редактируем — без миграции под хранение.
8. **Отправка текстом.** Сериализация плана в текст для мессенджера — снимок на момент,
   живой прогресс только в кабинете.

## Архитектурные принципы

**Модульность.** План — самодостаточная единица: свои таблицы, свой хук, свои компоненты,
свой модуль в реестре. Можно включать/выключать на роль и на шаблон, как любой модуль.

**Переиспользование.** Ничего не пишем с нуля там, где есть готовое:

| Берём готовое | Откуда | Как |
|---|---|---|
| Текстовый редактор/рендер | Tiptap 3.x (уже в стеке) | блок `text` |
| Перетаскивание блоков | `@dnd-kit` (уже в стеке) | редактор плана |
| Данные + загрузка слота | `useFolderSlots.ts`, RPC `fill_slot_atomic` | блок `slot` (read + action, без правки) |
| Данные задачи по id | `useProjectThreads` | блок `task` |
| Разворачивание из шаблона | seed-паттерн в [`TaskPanelTabbedShellRenderer.tsx:166`](../../src/components/tasks/TaskPanelTabbedShellRenderer.tsx:166) | резолв `source_template_id` (задачи) / `folder_template_slot_id` (слоты) |
| Видимость/права модуля | реестр [`moduleRegistry.ts:63`](../../src/page-components/ProjectPage/moduleRegistry.ts:63) + `useProjectModules` | автоматически по `enabled_modules` + `module_access` |

**Минимальное касание существующего (zero-touch).** Полностью без правок существующих файлов
новый модуль не появится — нужны точки монтажа. Но все правки **аддитивные** (добавление,
не изменение поведения):

1. `moduleRegistry.ts` — +1 запись модуля `plan`.
2. `src/types/permissions.ts` — +поле `plan` в `ProjectModuleAccess` (тип `ProjectModule`
   расширяется автоматически).
3. Миграция seed-ролей — +дефолт `module_access.plan` (не меняет существующие значения).
4. Редактор шаблона проекта — +чекбокс модуля `plan` (в `enabled_modules`).
5. Контент вкладки «Задачи» в ProjectPage — +монтаж `<PlanSection/>` под списком задач,
   условно по доступу модуля. Одна вставка, логику задач не трогает.

**Что НЕ трогаем вообще:** список задач, доски/календарь, вкладку «Документы» и её `SlotItem`
(срок слота в плане рисуем своим компонентом; вывод срока в «Документах» — отдельная
опциональная задача, вне этого плана), карантинные зоны (мессенджер/email/mtproto).

## Модель данных

Две новые таблицы. Существующие не меняем.

### `project_plan_blocks` (живой план в проекте)

```
id              uuid pk
project_id      uuid not null  -> projects
workspace_id    uuid not null  -> workspaces
type            text not null  check (type in ('text','task','slot'))
sort_order      int  not null  -- авторский порядок блоков
visible_to_client boolean not null default false
content         jsonb null     -- Tiptap-документ, только для type='text'
thread_id       uuid null      -> project_threads  (только type='task')
folder_slot_id  uuid null      -> folder_slots     (только type='slot')
created_at, updated_at
```

### `project_template_plan_blocks` (шаблон плана)

```
id                  uuid pk
project_template_id uuid not null -> project_templates
type                text not null check (...)
sort_order          int  not null
visible_to_client   boolean not null default false
content             jsonb null
thread_template_id  uuid null  -> thread_templates  (только type='task')
slot_template_id    uuid null  -> document_kit_template_folder_slots (только type='slot')
created_at, updated_at
```

**RLS:**
- `project_plan_blocks` SELECT — у кого есть доступ к проекту (клиент дополнительно
  ограничен `visible_to_client`, либо фильтруем на клиентском экране + RLS по членству).
  INSERT/UPDATE/DELETE — сотрудник с правом редактирования проекта.
- `project_template_plan_blocks` — по правам на шаблон (как остальные `*_template_*`).

**Индексы:** `(project_id, sort_order)`, `(project_template_id, sort_order)`,
FK-индексы на `thread_id` / `folder_slot_id`.

### Разворачивание шаблона → план

Повторяем seed-паттерн вкладок панели (ленивый сев при первом открытии — потому что задачи
и слоты создаются асинхронно при создании проекта, и резолвить надо после их появления):

- блок `task`: `thread_template_id` → реальная задача по `project_threads.source_template_id`
  (уже проставляется в [`CreateProjectDialog.tsx:250`](../../src/components/projects/CreateProjectDialog.tsx:250));
- блок `slot`: `slot_template_id` → реальный слот по `folder_slots.folder_template_slot_id`;
- блок `text`: копируем `content` как есть.

Реализация — RPC `seed_project_plan_from_template(project_id)` (идемпотентная: не сеет, если
блоки уже есть) **или** клиентская сеялка по образцу `TaskPanelTabbedShellRenderer`. Если
часть задач/слотов ещё не материализовалась — ждём следующего рендера, не сеем частично.

## Фазы реализации

Каждая фаза самодостаточна и отгружаема отдельно.

### Фаза 0 — Фундамент (невидимо для пользователя) · ~S

- Миграция: 2 таблицы + RLS + индексы.
- Регистрация модуля `plan`: запись в `PROJECT_MODULES`, поле в `ProjectModuleAccess`,
  дефолты в seed-ролях, чекбокс в редакторе шаблона проекта.
- `supabase gen types typescript ... > src/types/database.ts`.
- `planKeys` в `src/hooks/queryKeys/`.

**Итог:** модуль существует, вкладка показывает пустой блок-заглушку. Ничего не сломано.

### Фаза 1 — Просмотр плана (read-only) · ~M

- Хук `useProjectPlan(projectId)` — читает блоки + подмешивает живые данные задач/слотов.
- `PlanSection` — сворачиваемый блок под списком задач (точка монтажа в контенте вкладки).
- Рендер трёх типов блоков на чтение:
  - `text` — Tiptap read-only;
  - `task` — компактная строка (название, статус, срок) из данных задачи;
  - `slot` — переиспуем существующий слот-компонент/данные (загрузка, статус, срок).
- Показ блока в режиме «календарь» тоже (свёрнут по умолчанию).

**Итог:** план виден и работает на чтение (если блоки засеяны вручную/из шаблона).

### Фаза 2 — Редактор плана для сотрудника · ~L

- Добавление/удаление/перетаскивание блоков (`@dnd-kit`).
- Вставка `text` (Tiptap edit), выбор существующей задачи, выбор существующего слота.
- Переключатель «виден клиенту» на каждом блоке.
- Показ/редактирование срока у слота **внутри плана** (свой рендер, «Документы» не трогаем).

**Итог:** план можно собрать руками в конкретном проекте.

### Фаза 3 — Шаблон плана + авторазворачивание · ~M

- Редактор шаблона плана в редакторе шаблона проекта (`project_template_plan_blocks`),
  по образцу `PanelTabsSection` / `SlotsEditor`.
- Сеялка при создании/первом открытии проекта (резолв через `source_template_id` /
  `folder_template_slot_id`).

**Итог:** «рыба» разворачивается автоматически у новых проектов шаблона.

### Фаза 4 — Видимость по модулям + клиентский экран · ~M

- Сборка видимости вкладки: показывать по (`tasks` OR `plan`); содержимое — по доступу
  каждого модуля (см. таблицу в «Ключевых решениях»).
- Два слоя видимости: задача-кубик в плане живёт по своей галочке, не по модулю `tasks`.
- Чистый клиентский вид плана: только блоки `visible_to_client`, задачи урезанно.
- Если у роли включён только `plan` — вкладку подписать «План».

**Итог:** клиент видит план онлайн в нужном объёме.

### Фаза 5 — Экспорт текстом · ~S

- Сериализатор блоков → текст/markdown для отправки в мессенджер (снимок на момент).

**Итог:** план можно выгрузить и отправить клиенту текстом.

## Открытые вопросы / решения

- **Задача-кубик vs модуль `tasks`** — *решено:* кубик управляется своей галочкой,
  показывается независимо от модуля задач.
- **Название вкладки при `plan`-only** — «План» (Фаза 4).
- **Срок слота в «Документах»** — вне этого плана; при желании отдельной мелкой задачей.
- **Клиентский экран** — пока тот же таб с фильтрацией контента по модулям. Отдельная
  страница/публичная ссылка для клиента — возможное развитие (план как самостоятельная
  сущность это позволяет).

## Риски

- **Гонка при разворачивании** (задачи/слоты создаются асинхронно) — снимается ленивой
  идемпотентной сеялкой, как у вкладок панели. Не сеем частично.
- **Производительность** — план тянет задачи/слоты через существующие хуки, без новых
  тяжёлых запросов.
- **Соблазн расширить срок слота в «Документах»** — держим вне scope, чтобы не задеть
  работающий `SlotItem`.

## Грубая оценка

S ≈ 0.5–1 день, M ≈ 1–2 дня, L ≈ 2–4 дня. Итого порядка **8–12 рабочих дней** на все фазы.
Минимально полезный результат (Фазы 0–2: план собирается и работает в проекте руками) —
~4–6 дней. Шаблоны и клиентский экран (Фазы 3–4) — следом.
