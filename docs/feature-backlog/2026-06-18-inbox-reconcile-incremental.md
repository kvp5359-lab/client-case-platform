# Бэклог: инкрементальная сверка материализованных таблиц Входящих

**Статус:** не начато. Не срочно (актуально при росте до ~сотен тысяч пар).
**Дата:** 2026-06-18.

## Проблема

Ночная авто-сверка (`reconcile_thread_inbox_meta` + `reconcile_thread_unread`,
cron `inbox-reconcile` в 04:00) и ручная кнопка (`reconcile_inbox_report`)
делают **полный глобальный пересчёт всех пар (участник × тред) по всем
воркспейсам**, построчным циклом (`PERFORM recompute_*` на каждую пару).

Замер: ~11 сек на 1 воркспейс (1157 тредов, 2258 пар, ~3 мс/единица).

Экстраполяция на масштаб (≈3 мс/пара):

| Сценарий (≈50 воркспейсов × 40 юзеров) | Пар всего | Время полной сверки |
|---|---|---|
| ~1000 тредов/ws | ~300 тыс. | ~10–12 мин |
| ~3000 тредов/ws | ~1.2 млн | ~40–50 мин |

То есть полный глобальный пересчёт к 2000 юзеров — **десятки минут**, ежечасно
нельзя, даже ночью тяжело. (Real-time через триггеры масштабируется нормально —
проблема только у полной сверки-страховки.)

## Решение (любой из вариантов, по нарастанию)

1. **Инкрементальная сверка** (рекомендуется): пересчитывать только треды с
   активностью за последние ~25 часов (по `project_messages.created_at` /
   `audit_logs.created_at` / `message_read_status.updated_at`). Триггеры держат
   остальное; сверка-страховка достаточно проверять «свежее». Бессрочно дёшево
   (секунды), можно гонять хоть раз в час.
2. **Set-based пересчёт** вместо построчного цикла — даже полный пересчёт
   ускорится в разы (одним `INSERT … SELECT` с агрегатами вместо N вызовов функции).
3. **Дешёвый детектор дрейфа** — лёгкий запрос контрольных сумм по сэмплу/счётчикам,
   полный пересчёт только при найденном расхождении.

## Где трогать

- `reconcile_thread_inbox_meta()`, `reconcile_thread_unread()` — переписать на
  инкрементальный/set-based.
- `reconcile_inbox_report()` — то же (использует те же reconcile-функции).
- cron `inbox-reconcile` — можно участить, когда сверка станет дешёвой.

## Связанное

- Главный множитель пар — **view_all-админы** (каждый = пара на каждый тред
  воркспейса). При множестве админов пары растут быстрее тредов. См. открытый
  вопрос view_all-админов в `messenger-ledger.md` (Фаза 2).
- Перф самих чтений уже решён (cutover Фаза 2.6, ×9 под нагрузкой) — этот бэклог
  только про стоимость сверки-страховки на масштабе.
