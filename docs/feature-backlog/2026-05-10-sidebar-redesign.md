# Гибкий сайдбар воркспейса

**Дата:** 2026-05-10
**Статус:** Backlog. Делается **после** реализации `item_lists` (списков тредов и проектов), вынесено отдельной задачей.

## Зачем

Текущая модель `workspace_sidebar_settings.slots` плоская — массив элементов двух типов (`nav:<key>`, `board:<uuid>`) с размещением в одну из двух зон (`topbar`, `list`). При росте числа досок/списков навигация развалится. Хотим:

1. **Папки** — группировать слоты в один уровень папок (без глубокого дерева).
2. **Системные разделы** — закреплять любые внутренние страницы, в т.ч. вкладки настроек (Корзина, Шаблоны проектов, Справочники, Сайдбар-настройки и т.д.) и конкретные страницы (например, конкретный шаблон проекта).
3. **Списки тредов и проектов** — закреплять `item_lists` так же, как сейчас закрепляются доски.

## Решения по требованиям

| Параметр | Решение |
|----------|---------|
| Глубина вложенности | **1 уровень** (папки, без поддерева) |
| Что можно класть в папку | Смешанно: `nav` + `board` + `list` + ссылки на страницы |
| Папки общие или личные | На старте — **общие** (как сейчас). В будущем разделение по ролям/юзерам (см. ниже). |
| Поведение папок | **Pop-up при клике** — папка не разворачивается inline, при клике открывается поповер со списком вложенных элементов. Состояние «развёрнуто» не нужно. |
| Прямые ссылки на страницы | Да, новый тип слота `link:<path>` (например, `/workspaces/[id]/settings/trash`) |

## Будущая разбивка по ролям/юзерам

Сейчас `workspace_sidebar_settings` — одна строка на воркспейс. В будущем хочется:
- **per-role**: у менеджера один сайдбар, у консультанта другой
- **per-user**: каждый юзер кастомизирует под себя

План миграции на `per-role` (когда дойдёт): добавить поле `role_id` в таблицу, дефолтная строка — `role_id IS NULL` (общая для всех). Резолв: сначала ищем по роли юзера, при отсутствии — берём общую.

## Новая модель данных

### Тип слота

```ts
type SidebarSlot = {
  id: string         // 'nav:<key>' | 'board:<uuid>' | 'list:<uuid>' | 'link:<path>' | 'folder:<uuid>'
  type: 'nav' | 'board' | 'list' | 'link' | 'folder'
  placement: 'topbar' | 'list'
  parent_id?: string  // id папки, если слот вложен (только 1 уровень: папка не может быть в папке)
  order: number
  badge_mode: 'disabled' | 'my_active_tasks' | 'all_my_tasks' | 'overdue_tasks' | 'unread_messages' | 'unread_threads'

  // Для type='link'
  href?: string
  label?: string
  icon?: string

  // Для type='folder'
  name?: string
  folder_icon?: string
  folder_color?: string
}
```

### Таблица

`workspace_sidebar_settings.slots` остаётся JSONB, схема меняется (добавляются `parent_id`, новые типы). При миграции существующие записи получают `parent_id = null` (плоская структура сохраняется).

### Папки

- `id = 'folder:<uuid>'`, `type='folder'`, `placement='topbar' | 'list'`, `parent_id = null` (папка не может быть в папке)
- Хранится name, опц. иконка/цвет
- При клике в UI — popover со списком вложенных слотов (отрендеренных как обычные пункты)

### Ограничения и валидация

- `parent_id` указывает на слот с `type='folder'` (FK-типа проверка на стороне фронта — это JSONB)
- Слоты внутри папки имеют `placement` равный placement самой папки (либо игнорируется, потому что показываются в поповере)
- Удаление папки → её содержимое возвращается в «Доступные» (или переносится в корень — обсудить)

## UI настройки сайдбара (изменения)

Сейчас три зоны: «Верхняя строка», «Список», «Доступные». Добавляется:

- **Управление папками**: создать/переименовать/удалить папку. Drag слота в папку или из папки. Папка показывается как «контейнер» с раскрывающейся секцией внутри списка слотов.
- **Тип «Системный раздел»**: выпадашка со списком доступных системных страниц (хардкод-реестр в коде — `SYSTEM_PAGES = [{key, label, href, defaultIcon}]`), юзер выбирает → создаётся слот `link:<path>`.
- **Тип «Список»**: добавляется секция в «Доступные» по аналогии с досками.

## Модель доступа

- `manage_workspace_settings` (как сейчас) — управление сайдбаром.
- При переходе на per-role / per-user — потребуется новый permission `manage_my_sidebar` (для personal override).

## Чек-лист реализации

1. **БД**: миграция, расширяющая схему `slots` (добавление `parent_id`, типов `link`, `folder`); writer-функция для миграции существующих записей.
2. **Типы**: расширить `SidebarSlot` в `src/lib/sidebarSettings.ts`.
3. **Реестр системных страниц**: `SYSTEM_PAGES` со списком доступных для закрепа маршрутов и их иконок.
4. **UI настроек** (`SidebarSettingsTab.tsx`): управление папками, drag-n-drop в/из папок, добавление системных страниц, добавление списков.
5. **Рендер сайдбара** (`WorkspaceSidebar`): отрисовка папок (popover при клике), новые типы (link, list).
6. **Бейджи**: те же режимы, что сейчас, плюс возможно специфичные для list-слотов (количество элементов в списке).
7. **Миграционный скрипт**: существующие плоские `slots` получают `parent_id = null` — без потери данных.
