# Мониторинг подключения каналов (алерт при отключении номера)

**Дата:** 2026-07-18
**Приоритет:** высокий (владелец должен узнавать об отключении номера сразу)
**Статус:** backlog

## Проблема

Сессии WhatsApp отваливаются в реальности: телефон надолго офлайн, пользователь разлогинил бота в «Связанных устройствах», WhatsApp разорвал связь, бан. Пока номер отключён — входящие/исходящие по нему молча не работают, а владелец узнаёт об этом постфактум от клиента. Нужен **автоматический алерт**, если какой-то из подключённых номеров (WAHA или Wazzup) перестал быть на связи.

Касается всех «телефонных» каналов, но в первую очередь **WAHA** (self-hosted, полностью на нас) и **Wazzup**.

## Что уже есть (переиспользовать, не изобретать)

- **`platform_alert_config` + `run_platform_alerts()` + pg_cron `platform-alerts`** (каждые 10 мин) — уже шлёт владельцу в Telegram при провалах отправки и падениях крона. Сейчас «спит» (нужно заполнить `enabled/bot_token/chat_id`). Правильное место, чтобы добавить проверку каналов. См. `infrastructure.md` → «Алерты о сбоях в Telegram».
- **`scripts/channel-health.mjs`** (read-only, в CI Ops Checks) — уже проверяет застрявшие исходящие, незакрытые `message_send_failures`, протухший Gmail watch, число MTProto-сессий. Логичное место для read-only проверки статусов сессий.
- **`waha_sessions.status`** — уже синкается: WAHA шлёт `session.status` webhook (`waha-webhook` → `handleSessionStatus`) + `op=status` в `waha-sessions`. Значения WAHA: `STARTING`/`SCAN_QR_CODE`/`WORKING`/`FAILED`/`STOPPED`. То есть «отключился» = `status != 'WORKING'` (был WORKING).
- **Wazzup**: статус канала приходит в webhook (`channelsUpdates`), хранится в `wazzup_channels`. Аналогично — отслеживать переход в не-рабочее состояние.

## Предлагаемое решение (два уровня)

### 1. Мгновенный алерт по событию (основной, самый отзывчивый)
В `waha-webhook` `handleSessionStatus`: при **переходе** `WORKING → (FAILED/STOPPED/SCAN_QR_CODE)` — сразу слать владельцу воркспейса уведомление (Telegram через ту же связку, что `run_platform_alerts`, или отдельный лёгкий вызов). Дебаунс: слать только на смену состояния (не на каждый одинаковый webhook), хранить «последний оповещённый статус» в `waha_sessions` (доп. колонка `alerted_status` или сравнение со старым `status` перед update).
- Плюс: узнаём в секунды.
- Для Wazzup — аналогично в `wazzup-webhook` на `channelsUpdates`.

### 2. Периодическая страховка (на случай пропущенного вебхука)
Расширить `run_platform_alerts()` (pg_cron): раз в N минут проверять `waha_sessions` и `wazzup_channels`, где номер был рабочим, а сейчас не-`WORKING` дольше X минут → алерт владельцу (с дебаунсом, чтобы не спамить каждые 10 мин — например, повтор раз в час, пока не восстановится). Плюс дополнить `channel-health.mjs` read-only проверкой (для CI-дашборда).
- Ловит случай, когда WAHA-контейнер лежал и `session.status` не пришёл.

## Детали/нюансы

- **Дебаунс обязателен** — иначе при частых WhatsApp-переподключениях владельца завалит уведомлениями. Слать на **смену** состояния + периодический повтор «всё ещё офлайн» не чаще раза в час.
- **Кому слать:** ответственный за номер (`waha_sessions.owner_user_id` / канал Wazzup) + владелец воркспейса. Минимум — владелец (уже есть chat_id в `platform_alert_config`).
- **Текст:** «⚠️ Номер <phone/имя> (WhatsApp) отключился — переподключите: Настройки → Интеграции → WhatsApp». Для WAHA приложить, что нужен новый QR, если `SCAN_QR_CODE`.
- **Не будить на штатное** `STARTING`/переподключение <1-2 мин — только на устойчивое не-рабочее состояние.
- **Восстановление:** желательно слать «✅ номер снова на связи», когда вернулся в `WORKING` (по желанию).

## Оценка
- Уровень 1 (событийный, WAHA): ~0.5 дня (правка `handleSessionStatus` + отправка + дебаунс-колонка).
- Уровень 2 (cron-страховка + Wazzup + `channel-health.mjs`): ~0.5–1 день.
- Предпосылка: заполнить `platform_alert_config` (bot_token/chat_id) — сейчас спит.
