# Финансовый модуль — ТЗ (MVP)

**Дата:** 2026-05-09
**Статус:** ТЗ согласовано, готово к реализации.
**Источник:** упрощённая версия старого ТЗ из `Проекты/ClientCase/specs/FINANCE_MODULE_SPEC.md`. Из 1045-строчного документа выкинуты: контрагенты как отдельная сущность (заменены на `participants`), исполнители у услуг, взаиморасчёты между сотрудниками, мультивалютность, экспорт, аудит.

## 1. Цель

Дать в каждом проекте простой учёт денег:

- сколько проект стоит (услуги клиенту);
- что пришло (доходы);
- что ушло (расходы);
- сколько на нём заработали (прибыль и % оплаты).

Без бухгалтерии, без расчётов между участниками, без интеграций с банком. Чисто «руками вношу — вижу картину».

## 2. Принципы

- **Один справочник людей — `participants`.** Контрагент в финансах = тот же `participant`, что и контакты проекта, и участники с доступом. Отдельной таблицы контрагентов не делаем.
- **Один справочник услуг на воркспейс** (`finance_services`) — используется и для услуг проекта, и как «статья» для доходов/расходов.
- **Доходы и расходы — одна таблица** с полем `type` (`income` / `expense`). Структура у них одинаковая.
- **Модуль подключается через `enabled_modules`** в шаблоне проекта, как остальные.
- **Валюта одна — EUR.** В будущем добавим колонку `currency`, сейчас зашита.
- **Мягкое удаление через `is_deleted`** (как везде в проекте).

## 3. Структура данных

### 3.1. `finance_services` — справочник услуг (воркспейс)

| Поле | Тип | Описание |
|------|-----|----------|
| `id` | uuid PK | |
| `workspace_id` | uuid FK → workspaces | |
| `name` | text NOT NULL | Название |
| `base_price` | numeric(12, 2) NOT NULL DEFAULT 0 | Базовая цена в EUR (0 — индивидуальная) |
| `is_deleted` | bool NOT NULL DEFAULT false | |
| `deleted_at`, `deleted_by` | | как в `projects` |
| `created_at`, `updated_at` | | |

RLS: видят все участники воркспейса, редактируют — с правом `finance:manage`.

### 3.2. `project_services` — услуги в проекте

| Поле | Тип | Описание |
|------|-----|----------|
| `id` | uuid PK | |
| `project_id` | uuid FK → projects ON DELETE CASCADE | |
| `service_id` | uuid FK → finance_services | Источник из справочника |
| `name` | text NOT NULL | Snapshot имени на момент добавления (можно править под проект) |
| `quantity` | numeric(10, 2) NOT NULL DEFAULT 1 CHECK (> 0) | Кол-во |
| `price` | numeric(12, 2) NOT NULL CHECK (>= 0) | Цена за единицу |
| `total` | numeric(12, 2) GENERATED ALWAYS AS (quantity * price) STORED | Сумма |
| `sort_order` | int NOT NULL DEFAULT 0 | DnD-сортировка |
| `is_deleted`, `deleted_at`, `deleted_by` | | |
| `created_at`, `updated_at` | | |

Индексы: `(project_id, is_deleted)`, `(service_id)`.

### 3.3. `project_transactions` — доходы и расходы

| Поле | Тип | Описание |
|------|-----|----------|
| `id` | uuid PK | |
| `project_id` | uuid FK → projects ON DELETE CASCADE | |
| `type` | text NOT NULL CHECK (`income`/`expense`) | |
| `date` | date NOT NULL DEFAULT CURRENT_DATE | Дата операции |
| `participant_id` | uuid FK → participants NULL | Контрагент (от кого / кому). Может быть NULL |
| `service_id` | uuid FK → finance_services NULL | Статья (за что). Может быть NULL |
| `amount` | numeric(12, 2) NOT NULL CHECK (> 0) | Сумма EUR |
| `comment` | text NULL | |
| `is_deleted`, `deleted_at`, `deleted_by` | | |
| `created_at`, `updated_at` | | |

Индексы: `(project_id, type, is_deleted)`, `(date DESC)`, `(participant_id)`, `(service_id)`.

Поля `participant_id` и `service_id` **необязательные** — в кассу можно вписать «налик 100 EUR» без указания от кого и за что.

### 3.4. Связи между доходами и услугами проекта

**Не делаем.** Доход указывает услугу из справочника, не из `project_services`. Это сознательное упрощение: «клиент заплатил за консультацию» — достаточно. Если позже понадобится «закрыть конкретную позицию из счёта проекта» — добавим nullable FK на `project_services`.

## 4. UI

### 4.1. Вкладка «Финансы» в проекте

Новый модуль `finance` в `PROJECT_MODULES`. Иконка — Wallet/Euro/Banknote из lucide. Доступ — по флагу `finance` в `module_access` шаблона проекта.

**Структура страницы (сверху вниз):**

1. **Карточки** (5 штук в ряд):
   - Стоимость проекта = `SUM(project_services.total)`.
   - Доходы = `SUM(project_transactions where type='income')`.
   - Расходы = `SUM(project_transactions where type='expense')`.
   - Прибыль = Доходы − Расходы.
   - % оплаты = Доходы / Стоимость проекта × 100 (если стоимость 0 — прочерк).
2. **Таблица «Услуги проекта»** — `project_services`.
3. **Таблица «Доходы»** — `project_transactions where type='income'`, по `date DESC`.
4. **Таблица «Расходы»** — `project_transactions where type='expense'`, по `date DESC`.

### 4.2. Таблица «Услуги проекта»

Колонки: Название · Кол-во · Цена · Сумма · действия (редактировать, удалить).

Кнопка «+ Добавить услугу» открывает диалог:
- **Услуга** — комбобокс по `finance_services` (поиск + кнопка «Создать новую услугу» прямо в диалоге, чтобы не уходить в настройки).
- **Кол-во** — по умолчанию 1.
- **Цена** — подтягивается из `base_price` выбранной услуги, можно править.
- **Сумма** — рассчитывается, read-only.

DnD-сортировка по `sort_order`.

### 4.3. Таблицы «Доходы» и «Расходы»

Колонки одинаковые: Дата · Контрагент · Статья (услуга) · Сумма · Комментарий · действия.

Кнопка «+ Добавить» открывает диалог:
- **Дата** — date picker, по умолчанию сегодня.
- **Контрагент** — комбобокс по `participants` воркспейса (поиск, опционально).
- **Статья** — комбобокс по `finance_services` (опционально).
- **Сумма** — number, > 0.
- **Комментарий** — textarea, опционально.

### 4.4. Справочник услуг

Страница `/directories/finance-services` (или вкладка в `/directories/custom`, если решим унифицировать — обсудить на этапе реализации).

Колонки: Название · Базовая цена · действия. Стандартный CRUD.

При изменении названия услуги в справочнике **проектные `project_services.name` НЕ обновляются** — они хранят snapshot. Это сознательно: история не переписывается задним числом. Цена в проекте тоже фиксируется на момент добавления.

## 5. Права

Новый ключ модуля `finance` в `PROJECT_MODULES` + флаг в `module_access` проектной роли.

**На уровне воркспейса** (через `workspace_roles`):
- `finance:manage` — управление справочником услуг.

**На уровне проекта** (через `module_access.finance`):
- Только просмотр / просмотр + редактирование — стандартная двухуровневая модель, как у других модулей.

**Клиент** проекта по умолчанию **финансы не видит**.

## 6. Этапы реализации

### Этап 1. БД и типы
- Миграция: 3 таблицы + RLS + индексы.
- Регенерация `src/types/database.ts`.
- TS-типы домена в `src/types/finance.ts`.

### Этап 2. Справочник услуг
- Страница `/directories/finance-services`.
- Хуки: `useFinanceServices`, `useCreateFinanceService`, `useUpdateFinanceService`, `useDeleteFinanceService`.
- Право `finance:manage` в воркспейс-ролях.

### Этап 3. Регистрация модуля
- Добавить `finance` в `PROJECT_MODULES`.
- Иконка, название, описание модуля.
- Подключение через `enabled_modules` в шаблоне проекта.
- Вкладка «Финансы» в проекте (пустая заглушка).

### Этап 4. Услуги проекта
- Таблица + диалог добавления/редактирования.
- DnD-сортировка.
- Хуки: `useProjectServices`, мутации.

### Этап 5. Транзакции
- Таблицы «Доходы» и «Расходы» (можно общим компонентом с пропсом `type`).
- Диалог добавления/редактирования.
- Хуки: `useProjectTransactions(projectId, type)`, мутации.

### Этап 6. Сводка
- Карточки сверху.
- RPC `get_project_finance_summary(project_id)` или агрегация на фронте через `select`.

### Этап 7. Права в UI
- Проверка `module_access.finance` в `ProjectTabsContent`.
- Скрытие вкладки и кнопок без права редактирования.

### Этап 8 (отложено). Сводка по воркспейсу
- Страница `/workspaces/[id]/finance` — общая прибыль / доходы / расходы по всем проектам.
- Делаем после того, как проектный функционал отлежится.

## 7. Что НЕ делаем в MVP

- Контрагенты как отдельная сущность (используем `participants`).
- Исполнители у услуг (балансы между сотрудниками).
- Связь дохода с конкретной позицией в `project_services`.
- Мультивалютность.
- Экспорт в Excel/PDF.
- Аудит изменений финансов.
- Уведомления.
- Интеграции с банком/платёжками.

Каждый из этих пунктов — потенциальное расширение, добавляется отдельным ТЗ.
