# План работ по интеграциям

**Дата:** 2026-05-02
**Связь:** часть [CRM-фрейма](./2026-04-30-crm-frame.md), раздел 4 «Интеграции».
**Статус:** этап 0 сделан, этап 1 в работе.

## 1. Принципы

- **Всё на уровне юрфирмы.** В env-vars сервиса не остаётся ничего, что относится к конкретной юрфирме. Появление второго клиента сервиса не требует пересборки/редеплоя.
- **Один бот = одна задача.** Не пытаемся слить функции «слушать группу», «отправлять от лица сотрудника», «отвечать в личке клиенту» в одного бота. Каждая задача — своя сущность с привязкой к воркспейсу (и иногда к сотруднику).
- **Сначала аккуратно отделить существующее, потом добавлять новое.** Текущий рабочий бот переезжает первым, без ломки.

## 2. Текущее состояние

### Что работает
- **Telegram-бот** обслуживает 62 групповых чата. Токен живёт в env (`TELEGRAM_BOT_TOKEN` / `TELEGRAM_BOT_TOKEN_V2`). Уже есть архитектурный задел: helper `_shared/telegramBotToken.ts` выбирает токен по `project_telegram_chats.bot_version`.
- **Gmail** — модель уже per-workspace (`email_accounts.workspace_id` NOT NULL).
- **Все служебные таблицы Telegram** (`project_telegram_chats`, `telegram_link_tokens`) — уже имеют `workspace_id`.

### Что добавлено в Этапе 0 ✅
- Таблица `workspace_integrations(workspace_id, type, config jsonb, secrets jsonb, is_active, ...)` с RLS.
- Вкладка «Интеграции» в настройках воркспейса (read-only витрина).
- Существующая логика Telegram/Gmail не тронута.

## 3. Четыре роли Telegram-ботов в финальной модели

Чтобы было понятно, к чему мы идём — кратко описываю целевую картину.

| Роль | Назначение | Привязка |
|---|---|---|
| **Бот-секретарь** | Слушает групповые чаты с клиентами, переносит входящие в сервис. Обрабатывает команды клиента в группах (`/status`, `/docs`). | Один на юрфирму |
| **Личный бот сотрудника** | Только **отправляет** в группы от лица сотрудника (его имя, его аватарка). На входящие не реагирует. | Один на сотрудника |
| **DM-бот** | Принимает сообщения от клиентов **в личку** (не в группе). Каждый диалог становится тредом в сервисе. | Один на юрфирму |
| **Telegram Business** | Подключает личный Telegram-аккаунт сотрудника (Premium). Переносит **личные** переписки клиент↔сотрудник в сервис. | Один на сотрудника |

**Telegram Business работает только в личных чатах**, в группах не действует. Если клиент пишет в группу — это отрабатывает бот-секретарь, даже если сотрудник в группе сидит со своим Premium-аккаунтом.

В каждой группе с клиентом сотрудник присутствует одним способом: либо личным ботом, либо личным аккаунтом, либо вообще не присутствует. Параллельно «и бот, и человек» — путаница для клиента, не делаем.

## 4. Этапы работ

### Этап 1. Бот-секретарь → к юрфирме (≈ полдня)

**Цель.** Перевести существующего бота с уровня сервиса на уровень юрфирмы. После этого env-vars перестают быть точкой настройки бота — токен живёт в БД.

**Что меняется в БД:**
- В `workspace_integrations` создаётся запись для текущего воркспейса:
  ```
  type:    'telegram_workspace_bot'
  config:  { bot_version: 'v1' }
  secrets: { token: '<копия из TELEGRAM_BOT_TOKEN>' }
  is_active: true
  ```
- В `project_telegram_chats` добавляется колонка `integration_id uuid` (nullable, FK).
- Все 62 существующие группы получают `integration_id` = id новой записи.

**Что меняется в коде:**
- `_shared/telegramBotToken.ts` (`resolveBotToken`): если у группы есть `integration_id` → берёт токен из `workspace_integrations.secrets.token`. Иначе → env-фоллбэк (старая логика по `bot_version`).
- 8 edge-функций (`telegram-webhook`, `telegram-send-message`, `-edit-`, `-delete-`, `-set-reaction`, `-setup-webhook`, `-webhook-v2`) уже используют helper — менять их не нужно.

**Что НЕ меняется:**
- Webhook URL и `setWebhook` в Telegram остаются как есть.
- Идентификация воркспейса при входящем апдейте — через `chat.id` → `project_telegram_chats.workspace_id`. Никакого `secret_token` пока не нужно (бот один).
- Поведение бота для конечных пользователей — нулевые отличия.

**Стратегия теста:** на выходных. Пользователь шлёт тестовое сообщение в реальную группу → бот отвечает → проверяем приём и отправку. Если что-то идёт не так — env-фоллбэк работает как safety net.

### Этап 2. Личные боты сотрудников (≈ 2–3 дня)

**Цель.** Каждый сотрудник имеет своего бота с его именем и аватаркой. При отправке сообщений в группу клиент видит «Денис Крылов» с аватаркой — а не «бот юрфирмы (Денис)».

Технические детали уже расписаны в [2026-04-20-telegram-per-employee-bots.md](./2026-04-20-telegram-per-employee-bots.md). Здесь — только финальная модель и UI.

**Модель данных:**
- В `workspace_integrations` записи типа `telegram_employee_bot`, с `config: { owner_user_id }` и `secrets: { token }`.
- Лимит — один личный бот на сотрудника на юрфирму.

**UI на вкладке «Интеграции»:**
- Секция «Личные боты сотрудников».
- Список участников юрфирмы. У каждого: статус «Не подключён» / «Бот: @username (имя)».
- Кнопка «Подключить» открывает диалог с пошаговой инструкцией:
  1. Открой `@BotFather` → команда `/newbot`.
  2. Имя бота — твоё ФИО.
  3. Username бота — любой свободный, оканчивается на `_bot`.
  4. `/setuserpic` — загрузить аватарку.
  5. `/setprivacy` → `Disable` (для employee_bot — наоборот, `Enable`, чтобы не дублировать с секретарём).
  6. Скопировать токен в поле формы → сохранить.

**Логика отправки:**
- При отправке сообщения в группу из сервиса:
  - Если у текущего пользователя есть личный бот → отправка через его токен.
  - Иначе → через бота-секретаря с приставкой «(Имя) текст», как сейчас.
- При reply-attribution: входящие от Telegram продолжают идти через секретаря. Если клиент ответил на сообщение бота-сотрудника — это reply, секретарь видит, склеиваем по `chat_id` + `message_id`.

**Привязка Telegram-аккаунта сотрудника** (для случая когда сам сотрудник пишет в группу руками — нужно понимать, что это он): через одноразовый токен `telegram_link_tokens` (механизм уже есть).

**Операционка:**
- Боты сотрудников нужно **вручную добавлять в клиентские группы** (либо админ юрфирмы, либо бот-секретарь с правами админа).
- Лимит ~20 ботов на один Telegram-аккаунт. При упоре в лимит — создавать с другого Telegram.

### Этап 3. Gmail UI (≈ полдня)

**Цель.** Привести Gmail к управляемой карточке на вкладке «Интеграции».

**Что меняется:**
- На вкладке «Интеграции» карточка Gmail становится управляемой:
  - Кнопка «Подключить ящик» → существующий OAuth (`gmail-auth` → `gmail-callback`).
  - Список подключённых ящиков с кнопкой «Отключить» (уже есть `gmail-disconnect`).

**Что не меняется:** модель `email_accounts`, edge-функции `gmail-*`, `pg_cron gmail-watch-refresh`. Только UI.

### Этап 4. DM-бот (≈ 1–2 дня, после CRM-контактов)

**Цель.** Клиент пишет напрямую боту в личку → в сервисе создаётся тред. Альтернатива групповому чату для тех, кому проще общаться 1-на-1.

**Зависимость:** сущность «Контакт» из CRM-фрейма. Без неё непонятно, к какому клиенту привязывать диалог.

**Что добавляется:**
- В `workspace_integrations` тип `telegram_dm_bot`.
- В `telegram-webhook` — ветка для входящих в личке: ищем контакт по `from.id`, если нет — создаём новый, открываем тред.
- В UI «Интеграции» — карточка «DM-бот» с кнопкой подключения.

### Этап 5. Telegram Business (≈ 2 дня, последним)

**Цель.** Подключение Premium-аккаунтов сотрудников. Описание — в [CRM-фрейме п. 4.3](./2026-04-30-crm-frame.md).

**Зависимость:** после DM-бота, чтобы переиспользовать инфраструктуру личных диалогов клиент↔сотрудник.

**Что добавляется:**
- В `workspace_integrations` тип `telegram_business`, с `config: { owner_user_id, business_connection_id }`.
- В `telegram-webhook` — обработка апдейтов `business_connection`, `business_message`, `edited_business_message`, `deleted_business_messages`.
- В `telegram-send-message` — поддержка параметра `business_connection_id`.
- В UI «Интеграции» — карточка «Telegram Business» с инструкцией для сотрудников: подключить бота-секретаря в Telegram → Business → Chatbots.

## 6. Риски и митигация

| Этап | Риск | Митигация |
|------|------|-----------|
| 1 | Сломать приём/отправку для 62 живых групп | env-fallback в `resolveBotToken`, тест на выходных, откат через `integration_id = NULL` |
| 2 | Неправильный токен от сотрудника отрежет его бота от Telegram | Валидация токена при сохранении (вызов `getMe`); ошибка показывается сразу |
| 2 | Лимит 20 ботов на Telegram-аккаунт админа | Документируем, при упоре — второй админский Telegram |
| 4 | Спам в DM-бот от незнакомцев | Контакт создаётся «черновым», нужно вручную привязать к проекту/лиду |
| 5 | Сложная настройка Premium у сотрудника | Чёткая инструкция; диагностика «когда последний раз приходил business-апдейт» |

## 7. Метрики готовности

После этапов 1–3:
- На вкладке «Интеграции» владелец юрфирмы видит и управляет всеми каналами.
- Подключение нового канала = пара кликов, без env-vars и редеплоя.
- При появлении второго клиента сервиса все каналы у него работают независимо.

После этапов 4–5:
- Полное покрытие Telegram-сценариев: группы (секретарь + личные боты), DM (DM-бот + Telegram Business), Gmail.
- Поведение бота-секретаря и Gmail для конечных пользователей не изменилось ни на одном этапе.
