# План: видимость сообщения (4 режима) + подписка на тред + @теги

**Статус:** план, к реализации не приступали.
**Зона:** мессенджер (карантин — только Фаза 2, skip внешней доставки в триггере маршрутизации).
**Дата:** 2026-06-18.

## 1. Проблема

Сейчас «доступ к треду» = «кому прилетает непрочитанное»: красное считается всем, у кого есть доступ. Отсюда:
- владелец/админ (view_all) видит фантомное непрочитанное в чужих командных чатах (костыли `suppressUnread`, отложенная «Фаза 2» от 2026-06-09);
- отправитель не понимает, кто реально прочитает и уйдёт ли сообщение клиенту;
- нет способа написать внутреннюю реплику или личную пометку прямо в клиентском треде.

## 2. Финальная модель — три независимых слоя

Главный принцип: каждый слой отвечает строго за своё и **не протекает** в другой.

| Слой | Кто управляет | Что решает |
|---|---|---|
| **Доступ** | роли/участие/assignee/member/view_all (как сейчас) | кто **может открыть и прочитать** тред |
| **Видимость сообщения** (4 режима) | отправитель, на каждое сообщение | кто это сообщение **видит** + уходит ли клиенту |
| **Подписка на тред** | каждый сам за себя | прилетают ли **тебе** уведомления/непрочитанное по треду |
| (+) **@теги** | отправитель, в тексте | точечно **дёрнуть** конкретного коллегу вне зависимости от подписки |

«Видит» ⊆ «имеет доступ». «Уведомляется» ⊆ «видит» ∩ «подписан (или @тегнут)».

## 3. Переключатель видимости — один, 4 режима

Один сегментный переключатель над композером. Это **шкала охвата от широкого к узкому** (не две оси для пользователя, а одна линейка «как далеко уходит»):

| Режим | Кто видит | Уведомление | Уходит клиенту | Цвет бабла |
|---|---|---|---|---|
| **Клиенту** | клиент + команда | подписчики команды | ✅ да | цветной акцент чата |
| **Команде** | только команда | подписчики команды | ❌ | нейтральный чёрно-серый |
| **Заметка** (команде · тихо) | только команда | никого (только @теги) | ❌ | нейтральный чёрно-серый |
| **Только я** | только автор | никого | ❌ | жёлтый «стикер» |

- **4 режима = ровно 4 осмысленные комбинации** (видимость × уведомление). Пропущенные («клиенту без пинга команды», «только себе, но кого-то уведомить») бессмысленны/не нужны — ничего не теряется.
- Под капотом это две оси (видимость + уведомлять-ли), но для пользователя — одна понятная линейка.
- **«Команде» и «Заметка» дают одинаковый цвет бабла** (оба нейтральные) — это намеренно: разница «громко/тихо» — событие момента отправки, а не свойство истории. В **переключателе** их развести явно («Команде» / «Команде · тихо»), в **истории** красить одинаково.
- **@теги:** активны в «Клиенту / Команде / Заметка»; **выключены в «Только я»** (нельзя тегнуть человека в то, что видишь только ты).

## 4. Цветовая модель баблов (сквозная)

Переиспользуем уже прижившийся в задачах язык: **чёрные баблы (моё) / серые (входящее)** = внутреннее. Расширяем его на командные сообщения внутри клиентских чатов.

- **Палитра = аудитория.** Цветной акцент = касается клиента; нейтральный чёрно-серый = только команда; жёлтый = только я.
- **Оттенок внутри палитры = своё/чужое.** Тёмный = моё, светлый = входящее (работает и для акцента, и для нейтрали).
- Нейтральный должен быть **без цветового подтона**, чтобы серый «внутренний входящий» не путался с самым светлым «входящим от клиента».
- Так в одном клиентском чате уживаются оба стиля; сканируешь мгновенно: цветное — клиенту, нейтральное — внутри, жёлтое — лично.
- **Бонус:** активный режим переключателя красит поле/кнопку отправки в свой цвет → видишь будущий цвет бабла **до** нажатия = защита от случайной отправки клиенту.
- Реализация ложится в существующий механизм условного стиля [`messageStyles.ts`](../../src/components/messenger/utils/messageStyles.ts) (чёрно-серый акцент там уже есть) + [`MessageBubble.tsx`](../../src/components/messenger/MessageBubble.tsx).

## 5. Подписка на тред (личное право)

Это закрывает дыру «меня когда-то отметили, а отписаться нельзя» и фантомы view_all.

- **Подписан** → получаешь уведомления/непрочитанное по всем сообщениям треда, которые тебе видны. **Отписан** → доступ остаётся (читать можешь), но не цепляет.
- Управляет **только сам пользователь** (чужая правка видимости сообщения подписку не трогает).
- **Дефолт подписки** при создании/входе в тред: участники (`project_thread_members`) и исполнители (assignees) — да; **пассивный view_all-админ — нет** (он имеет доступ, но не подписан → фантом исчезает).
- **Авто-подписка по действию:** тебя @тегнули или ты сам написал в тред → автоматически подписан (как GitHub). Дальше можешь отписаться вручную.
- **@тег пробивает отсутствие подписки** (мягкий вариант, как GitHub): не подписан, но явно дёрнули — увидишь.

## 6. Модель данных

- **`project_messages.visibility`** enum `'client' | 'team' | 'self'` (DEFAULT `'client'`). Определяет аудиторию + (через неё) внешнюю доставку.
- **`project_messages.notify_subscribers`** boolean (DEFAULT true). Имеет смысл только при `visibility='team'`: false = «Заметка» (тихо). При `'client'` команда уведомляется всегда; при `'self'` — никого.
  - (Альтернатива: один enum из 4 значений вместо visibility+notify. Решить при реализации; пара полей гибче для формулы непрочитанного.)
- **`message_mentions (message_id, participant_id)`** PK(message_id, participant_id), индекс по participant_id — @теги, точечные адресаты уведомления.
- **Подписка:** `project_thread_subscriptions (thread_id, user_id, subscribed boolean, updated_at)` ЛИБО колонки на `project_thread_members`. Per-user (auth), т.к. непрочитанное материализуется пер-участник в `thread_unread_state`.

## 7. Непрочитанное = «видно мне И (я подписан ИЛИ @тегнут)»

Меняем условие в `recompute_thread_unread_for` (материализация `thread_unread_state` уже построена — Фаза 2 от 2026-06-17). Сообщение `pm` непрочитано для пользователя U, если:

```
pm.sender <> U  AND  pm.created_at > last_read_at(U)
AND видно_U(pm)            -- visibility='client' → команда+клиент; 'team' → команда; 'self' → только автор
AND ( подписан(U, thread)  -- подписка
      OR EXISTS(message_mentions mm WHERE mm.message_id=pm.id AND mm.participant_id=U) )  -- явный @тег
AND ( pm.visibility <> 'team' OR pm.notify_subscribers )  -- «Заметка» не копит непрочитанное у подписчиков, только @теги
```

Фан-аут на запись становится чище: триггер инкрементит непрочитанное только подписчикам/@тегнутым, а не всем с доступом. **view_all-фантомы исчезают в корне** (не подписан → не считается).

> ⚠️ Семантика непрочитанного **меняется намеренно** — НЕ сверять с v2/старой формулой. Эталон — НОВОЕ ожидание на тест-кейсах. Кнопка «Сверка Входящих» на этих тредах покажет «расхождения» by design — обновить эталон сверки.

## 8. Дефолты режима (sticky)

- Режим следующего сообщения **наследуется от предыдущего сообщения треда** (Planfix-механика «sticky last»). Клиентский диалог сам встаёт на «Клиенту», внутренняя задача — на «Команде». Открыл → написал → отправил, переключатель трогаешь только когда реально нужно сузить.
- **Первое сообщение треда** (наследовать не от чего): клиентский тип треда (есть TG/email/контакт) → «Клиенту»; внутренний → «Команде».
- **Пришло входящее от клиента** → следующий ответ по умолчанию снова «Клиенту» (иначе ответишь «в стол»).
- Переключатель — **сегментный** (все 4 на виду), не выпадашка: всегда видно, куда уйдёт.

## 10. Фазы внедрения

Порядок: сначала самое дешёвое и бьющее в главную боль, карантин и большой билд — позже.

- **Фаза 1 — подписка + новая формула непрочитанного (низкий риск, НЕ карантин).** `project_thread_subscriptions` + дефолты подписки (участники/assignee да, пассивный view_all нет) + авто-подписка по действию + правка `recompute_thread_unread_for`. Чип «Подписка» в шапку. **Уже убирает фантомы view_all** — без введения типов сообщений. Переиспользует материализацию.
- **Фаза 2 — видимость сообщения + цвет + skip внешней доставки (карантин).** `visibility`/`notify_subscribers` + сегментный переключатель + цветовая модель баблов + skip-кондишн в триггере маршрутизации для не-`client`. **Смок-тест всех каналов обязателен** (клиентские по-прежнему доходят; team/self наружу не утекают). «Заметка» и «Только я» здесь же (visibility-only, доставки не касаются).
- **Фаза 3 — @теги (высокий объём, НЕ карантин, билд с нуля).** Tiptap Mention + popover участников + `message_mentions` + точечное уведомление + пробитие подписки. До этого «втянуть» человека = добавить участником/подписчиком вручную.
- **Фаза 4 — полировка.** Цвет-превью композера по активному режиму, индикатор «кто видит» в шапке (через `inbox_accessible_participant_ids`), запоминание дефолтов, опц. метка «тихо» в истории.

## 11. Чем это лучше Planfix-наброска (почему отказались от per-message списка получателей)

Старый набросок предлагал над каждым сообщением **список конкретных получателей** (галочки сотрудников + клиент), как в Planfix. В ходе проработки нашли дыру и более чистую модель:

- **Дыра per-message списка:** получателей выбирает отправитель. Если коллега уберёт тебя из *своего* сообщения, а sticky-наследование протащит это дальше — входящее от клиента тебя уже не заденет, хотя ты сам не отписывался. «Кого касается тред» нельзя наследовать от чужого сообщения.
- **Решение:** разнести «кого касается тред» (подписка — **постоянная, управляешь только ты**) и «кто видит конкретное сообщение» (видимость — выбор отправителя на сообщение). Тогда чужая правка живёт строго в пределах своего сообщения.
- **Меньше ручной возни:** отправитель не ведёт список коллег вручную — выбирает один из 4 режимов; кого из команды касается, решает каждый своей подпиской. @тег — для точечного исключения/добавления.
- Это, по сути, проверенная боем модель helpdesk-систем (Zendesk/Front/HubSpot: ответ / внутренняя заметка / @mention / подписчики) + наш сквозной цвет и уровень «Только я».

## 12. Открытые вопросы / решения

- **visibility+notify двумя полями или один enum из 4** — решить при реализации (пара полей гибче для формулы непрочитанного).
- **@тег в «Только я»** — запретить (рекомендация) либо повышать видимость до «Команде». Запрет проще/предсказуемее.
- **«Команде» vs «Заметка» в переключателе** — назвать так, чтобы «тихо» было очевидно («Команде» / «Команде · тихо»). В истории — одинаковый цвет.
- **Подписка: per-user или per-participant** — материализация пер-участник, привязать соответственно.
- **Реакции/события** — оставить как есть (адресность только у сообщений) либо тоже учесть подписку — решить отдельно.
- **Старые сообщения** без `visibility` → DEFAULT `'client'` (текущая семантика истории сохраняется), без тяжёлого бэкафилла.
