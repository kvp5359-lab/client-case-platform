# CRM-фрейм — ТЗ

**Дата:** 2026-04-30
**Статус:** ТЗ согласовано на уровне модели и подходов. Реализация поэтапная (см. порядок ниже).

## 1. Цель

Превратить существующую систему ведения проектов в полноценную CRM. Унифицировать работу с лидами и клиентами через **существующую сущность проекта** (с разными шаблонами), не плодить параллельные сущности под лиды.

Подключить полноценные каналы коммуникации с клиентами: Telegram (групповые чаты, личные DM, личные аккаунты менеджеров через Telegram Business), Gmail. Все каналы — настраиваются на уровне воркспейса.

## 2. Принципы

- **Не изобретать сущность «Лид»** — это проект с шаблоном «Лид».
- **Не дублировать UI и логику** — переиспользовать треды, статусы, шаблоны, права, дневник.
- **Контакт — константа** в воркспейсе. Связывает все проекты одного клиента (лид + рабочие проекты).
- **Каналы — на уровне воркспейса**, не глобально.
- **Telegram через MTProto/номер телефона — отбрасываем.** Используем Telegram Business как штатный путь для личных аккаунтов менеджеров.

## 3. Модель данных

### 3.1. Контакт (`workspace_contacts`)

Уникален в рамках воркспейса. Хранит ФИО, заметки, прочие общие поля.

```
workspace_contacts
  id, workspace_id, name, notes, created_at, created_by, ...
```

### 3.2. Каналы контакта (`contact_channels`)

Связь контакта с идентификаторами в каналах. Один контакт может иметь несколько каналов одного типа (два email, рабочий + личный Telegram).

```
contact_channels
  id, contact_id, channel_type ('telegram' | 'email' | ...), external_id, label
  UNIQUE (workspace_id, channel_type, external_id)  -- индекс для быстрого поиска контакта по входящему
```

При входящем сообщении система ищет контакт по `(channel_type, external_id)`. Если не находит — создаёт новый контакт + канал.

**Мердж дублей.** Если выяснилось, что два контакта — это один человек, нужна операция «объединить контакты»: переносит каналы, проекты, переписку в один. UI — кнопка в карточке контакта. Реализуется на этапе после MVP.

### 3.3. Привязка проекта к контакту

```
projects.contact_id  → workspace_contacts.id  (NULL допустим — для проектов без клиента)
```

Один проект — один контакт. Один контакт — много проектов.

**Single, не multi.** Решено для MVP: одно поле «Контакт» = один контакт на проект. Если упрёмся в кейсы (семейная пара, юрлицо с несколькими представителями) — переход single → multi делается миграцией без потери данных.

### 3.3.1. Нейминг полей в UI

Чтобы не путать новое поле и существующее «Клиенты», нейминг разводим по CRM-стандарту:

- **Новое поле «Контакт»** (single, `projects.contact_id`) — бизнес-связка с записью человека из справочника. Без прав, без доступа. Именно через это поле работает маршрутизация входящих, привязка тредов из Gmail/Telegram, связь лид↔рабочие проекты.
- **Существующее поле «Клиенты»** (мульти, `project_participants` с ролью `CLIENT`) — про **доступ к ЛК**: кому из зарегистрированных пользователей открыт этот проект изнутри. Логику и таблицу не трогаем.

**Стандарт CRM:** в Pipedrive/HubSpot/Salesforce/AmoCRM/Bitrix24 поле «человек на сделке/проекте» называется именно **«Контакт» / Contact**. Слово «Клиент» в CRM используется как **статус/тип**, а не как поле. Поэтому новое поле = «Контакт». Существующее поле «Клиенты» — нестандартно по природе (это про ЛК-доступ, а не про список клиентов сделки), но переименование откладываем — это смущало бы существующих менеджеров.

**Связь между полями:** если у контакта заполнен `linked_user_id` (контакт получил ЛК) — этого пользователя опционально добавляют в «Клиенты» того же проекта, чтобы он видел проект в ЛК. Без дублирования данных в схеме.

### 3.4. Лид и рабочие проекты — это просто разные шаблоны

- Шаблон «Лид»: статусы воронки продаж (Новый → Квалификация → Переговоры → Договор → Выиграно / Проиграно), поля продаж (источник, бюджет, вероятность).
- Шаблоны рабочих проектов («ВНЖ», «Открытие ИП», «Бухсопровождение» и т.д.): статусы рабочего процесса, поля под услугу.

**Один лид может породить N рабочих проектов** (клиент пришёл с пакетом услуг). Конверсия — явная операция, см. п. 6.

### 3.5. Кастомные поля проекта (пререквизит)

Сейчас кастомных полей у проекта нет. Это нужно сделать в любом случае (не только под CRM).

```
project_custom_fields  -- определения полей (на воркспейс или на шаблон)
project_custom_field_values  -- значения для конкретного проекта
```

Поля могут быть привязаны к шаблону проекта (поля «Лида» отличаются от полей «ВНЖ»). Типы полей — текст, число, дата, выбор из списка, связь со справочником, связь с пользователем. Реализуется как отдельная фича до начала работ по CRM.

### 3.6. Финальные статусы — подтипы

У статусов уже есть `is_final`. Добавить `final_kind` (`won` | `lost` | `abandoned`) для статусов лидов — нужно для аналитики воронки.

## 4. Интеграции — раздел в настройках воркспейса

### 4.1. Общая таблица

Все интеграции воркспейса — в одной таблице. Не плодим отдельную таблицу под каждый канал.

```
workspace_integrations
  id, workspace_id, type ('telegram_bot' | 'telegram_business' | 'gmail' | ...),
  config jsonb, secrets jsonb (зашифровано), is_active, created_at, updated_at
  UNIQUE (workspace_id, type, ...) -- по необходимости
```

Раздел в UI: `/workspaces/[id]/settings/integrations` — список подключений с возможностью добавить, отключить, переконфигурировать.

### 4.2. Telegram-бот воркспейса (per-workspace бот)

Заменяет глобального бота. Каждый воркспейс настраивает своего: имя, аватар, токен.

**Один общий webhook-эндпоинт `telegram-webhook` обслуживает N ботов** через **секретный токен Telegram'а** (`secret_token` параметр в `setWebhook`):
- При подключении бота воркспейс генерирует уникальный секрет, прописывает его в `setWebhook(secret_token=...)`.
- Telegram присылает каждый апдейт с заголовком `X-Telegram-Bot-Api-Secret-Token`.
- `telegram-webhook` находит воркспейс по секрету (поиск в `workspace_integrations`).

Покрывает оба сценария:
- **Бот в группах** (текущий механизм — оставляем).
- **Бот для личных DM с клиентом** (новый сценарий — клиент пишет напрямую боту воркспейса).

### 4.3. Telegram Business (личные аккаунты менеджеров)

Менеджер с Telegram Premium подключает бота воркспейса в настройках Telegram → Business → Chatbots. После этого в `telegram-webhook` приходят апдейты типа `business_message` от его личных чатов.

- В апдейте есть `business_connection_id` — связывает сообщение с конкретным менеджером воркспейса.
- Отправка ответов — `sendMessage` с `business_connection_id`.
- **Никаких MTProto, GramJS, session-string.** Обычный bot API.
- **Ограничения по времени нет.** 24-часовое окно — это WhatsApp Business, не Telegram. В Telegram Business `can_reply` — это статичное permission-флаг без таймера. Бот может отвечать в чатах, к которым менеджер дал доступ, без ограничения по времени, даже через год после последнего сообщения. Проверено по официальной документации Telegram и framework grammY.
- Единственное ограничение: бот не может **первым** писать пользователю, который никогда не контактировал с менеджером. Это анти-спам мера. Для CRM-сценария это естественное поведение (менеджер сам инициирует первое касание руками).
- **Premium нужен менеджерам** (тем, кто общается с клиентами от своего личного Telegram), не клиентам. Стоимость для команды ~$5/мес на менеджера.

Хранение в `workspace_integrations` (тип `telegram_business`): для каждого подключённого `business_connection_id` — связь с пользователем (менеджером) воркспейса.

### 4.4. Gmail per-workspace

**Текущая модель:** `email_accounts` привязаны к юзеру.
**Целевая модель:** ящик принадлежит воркспейсу, видят все менеджеры (с учётом прав).

- OAuth-флоу для Gmail у тебя уже есть (`gmail-auth`, `gmail-callback`, `gmail-webhook`, `gmail-watch-refresh`).
- Нужна миграция модели `email_accounts` под привязку к воркспейсу + возможно справочник «кто из менеджеров может писать с этого ящика».
- Существующий `pg_cron` `gmail-watch-refresh` сохраняется без изменений.

### 4.5. Что НЕ делаем сейчас

- **MTProto / Telegram через номер телефона** — отбрасываем как избыточно сложное при наличии Telegram Business.
- **IMAP/SMTP** — отложить. Для большинства клиентов хватит Gmail.
- **WhatsApp, Instagram и т.д.** — отдельная история, не в этом ТЗ.

## 5. Маршрутизация входящих сообщений

**Алгоритм при поступлении входящего:**

1. Найти контакт по `(channel_type, external_id)`. Если нет — создать.
2. Проверить активные проекты этого контакта (любого типа — лид или рабочий):
   - **Один активный проект** → создаём/находим тред в нём, кидаем сообщение туда.
   - **Несколько активных** → сообщение попадает в Inbox воркспейса со статусом «требует выбора проекта». Менеджер выбирает.
   - **Ни одного активного** → создаём **новый лид** (проект с шаблоном «Лид» по умолчанию для воркспейса), привязываем к контакту, кидаем сообщение в новый тред в нём.
3. **Кнопка «привязать тред к другому проекту»** доступна всегда, как страховка от ошибок маршрутизации.

**Открытый вопрос:** окончательный механизм при множественных активных проектах (отложили). Пока действует логика выше.

## 6. Конверсия «лид → рабочие проекты»

**Конверсия — это явная ручная операция** (кнопка в карточке лида), а не автоматический переход. Слово из CRM-словаря, означает «лид сконвертировался в клиента/сделку».

Когда лид выигран, менеджер нажимает «Создать рабочий проект из лида» **один или несколько раз** — по числу услуг, на которые согласился клиент.

Каждое нажатие:
- Открывает диалог выбора шаблона рабочего проекта.
- Создаёт новый проект с этим шаблоном, привязывает к тому же контакту.
- Опционально (галочка в диалоге): копировать/связать выбранные треды и файлы из лида.
- В лиде сохраняется ссылка «созданные рабочие проекты: …» для навигации.

**Сам лид при конверсии не удаляется и не превращается во что-то другое.** Он закрывается в финальном статусе `won` (см. п. 3.6 про `final_kind`) и живёт дальше как архивная запись:
- История продажной переписки, обещания, возражения — сохраняются.
- Аналитика воронки считается по выигранным лидам / общему числу.
- Из рабочего проекта можно вернуться к исходному лиду через `parent_lead_id` или общий контакт.
- При возврате клиента через год — вся продажная история под рукой.

Опциональное поле в `projects`: `parent_lead_id` (NULL для самих лидов и для проектов, не созданных из лида) — для аналитики и быстрой навигации. Не критично, но дешёво, лучше заложить сразу.

## 7. Распределение входящих между менеджерами

**Решение для MVP:** вручную старшим менеджером.

Новый лид при создании имеет `assignee_id = NULL` (или дефолтное правило воркспейса — если есть). Старший видит «нераспределённые лиды» в общем Inbox-е и назначает руками.

**Не делаем сейчас:** round-robin, дежурный, правила маршрутизации по источнику. Если потом потребуется — добавляем поверх.

## 8. UI

### 8.1. Карточка контакта

Новая страница / модалка. Показывает:
- Общую информацию (ФИО, заметки, теги).
- Каналы связи (список с возможностью добавить).
- Все проекты контакта (лид + рабочие проекты, активные и завершённые).
- Унифицированная история переписки по всем каналам и всем проектам (опционально, может быть отдельной вкладкой).

### 8.2. Воронка лидов

**Ложится на существующий механизм досок без архитектурных изменений.** Проверено: доски уже поддерживают `entity_type='project'` (есть `BoardProjectRow`, фильтры по статусам проекта, группировка, кастомные `visible_fields` и `card_layout`). Воронка = доска `entity_type=project` с фильтром `template_id = Лид`.

Возможные косметические доработки (не структурные):
- Drag-n-drop между статусными группами для проектов — если сейчас есть только для тасков, перенос на проекты делается переиспользованием существующего механизма.
- Канбан-режим отображения (горизонтальные колонки) — если сейчас только список-с-группами. UI-доработка.

Несколько воронок = несколько досок с разными фильтрами (по категории услуги, по источнику и т.д.). Закрепление в сайдбар уже работает через существующий механизм `pinned_boards`.

### 8.3. Inbox воркспейса

- Расширение существующего Inbox.
- Секция «Новые лиды без ответственного».
- Секция «Сообщения, требующие выбора проекта» (при множественных активных проектах контакта).

### 8.4. Раздел «Интеграции» в настройках

`/workspaces/[id]/settings/integrations` — единый раздел со списком подключений. Доступ только владельцу воркспейса. Каждая интеграция — карточка со статусом, конфигом, кнопкой «отключить».

## 9. Edge Functions

- `telegram-webhook` — расширить:
  - Идентификация воркспейса через `X-Telegram-Bot-Api-Secret-Token`.
  - Обработка апдейтов `business_connection`, `business_message`, `edited_business_message`, `deleted_business_messages`.
  - Поиск контакта по `(channel_type, external_id)`, маршрутизация в проект (см. п. 5).
- `telegram-send-message` — расширить под отправку с `business_connection_id`.
- `gmail-*` — миграция привязки ящика к воркспейсу.
- Новая `convert-lead-to-projects` (или серверный action в Next.js API) — операция конверсии.

## 10. Открытые вопросы (отложены)

- **Маршрутизация при нескольких активных проектах у контакта** (п. 5, шаг 2): окончательный UX.
- **Реактивация недавно закрытого лида** (если клиент вернулся через день): автоматом или руками. Пока — руками.
- **Метрики воронки и SLA** (время первого ответа, конверсия по этапам). Нужен отдельный модуль аналитики.
- **Уведомления** менеджерам (новый лид, новое сообщение в моём проекте, нарушение SLA). Нужен отдельный модуль.
- **Исходящий аутрич** (рассылки, холодные касания). Не входит в первую волну.
- **Мердж дублей контактов** — UI и логика. После MVP.
- **IMAP-почта**, **WhatsApp** и прочие каналы. После того, как закрепится база.

## 11. Порядок реализации

Жёсткие пререквизиты сверху, чем ниже — тем дальше по времени.

1. **Кастомные поля проекта** (полностью самостоятельная фича, нужна не только под CRM).
2. **Сущность «Контакт» и каналы** (`workspace_contacts`, `contact_channels`, `projects.contact_id`). Базовый UI карточки контакта.
3. **Шаблон «Лид»** (создание, статусы воронки, `final_kind`, дефолтные кастомные поля).
4. **Воронка лидов в Boards** (фильтр + канбан, переиспользуем существующее).
5. **Раздел «Интеграции»** (`workspace_integrations` + UI настроек).
6. **Per-workspace Telegram-бот** (рефакторинг `telegram-webhook` под `secret_token`-маршрутизацию). Поддержка групп и личных DM.
7. **Telegram Business** (новые апдейты в `telegram-webhook`, отправка с `business_connection_id`).
8. **Gmail per-workspace** (миграция модели `email_accounts`).
9. **Маршрутизация входящих в проект** (поиск контакта, выбор активного проекта, fallback на новый лид).
10. **Inbox: секции для лидов** (нераспределённые + требующие выбора проекта).
11. **Конверсия «лид → рабочие проекты»** (операция, диалог, `parent_lead_id`).
12. *(После MVP)* Мердж дублей контактов, метрики воронки, уведомления, IMAP, исходящий аутрич.

## 12. Модульность и риски ломки существующего

Стратегическая цель — реализовать всё как **отдельные модули**, чтобы в будущем продавать «модуль CRM» как отдельный платный пакет в подписке. Хорошая новость: 90% задуманного **аддитивно** и не трогает существующее.

### 12.1. Чисто аддитивные изменения (ничего не ломают)

Это новые таблицы, новые nullable-колонки, новые компоненты. Существующий функционал работает как раньше. Если выключить «модуль CRM» — всё это просто не показывается и не используется, остальной сервис продолжает жить:

- `workspace_contacts`, `contact_channels` — новые таблицы, изолированные.
- `projects.contact_id` — nullable-колонка, старые проекты не требуют значения.
- Шаблон «Лид» — это просто запись в существующем справочнике шаблонов.
- `final_kind` на статусах — новая nullable-колонка.
- `workspace_integrations` — новая таблица. Параллельно работает со старой моделью, пока всё не переедет.
- `parent_lead_id` — новая nullable-колонка.
- Воронка как доска — новая запись в `boards`, существующие доски не затрагиваются.
- Конверсия «лид → проекты» — новая кнопка/операция, старого кода не касается.
- Telegram Business — новая ветка обработки апдейтов в `telegram-webhook`. Старые апдейты от групп идут по существующему пути.
- Inbox-секции для лидов — расширение существующего Inbox через флаг.
- Раздел «Интеграции» в настройках — новая страница в settings.
- Карточка контакта — новая страница.
- Кастомные поля проекта — новая фича, не зависит от CRM.

**Риск ломки: нулевой.** Можно катить по очереди, каждый пункт включается/выключается отдельно.

### 12.2. Полу-инвазивные (требуют рефакторинга, но обратно совместимы)

Меняют существующий код, но без потери работоспособности. Существующие пользователи не замечают изменений:

- **Рефакторинг `telegram-webhook` под `secret_token`-маршрутизацию N ботов.** Текущий бот заводится в `workspace_integrations` как один из, продолжает работать, но через новый код. Откат через флаг возможен.
- **`linked_user_id` у контакта** — связь между контактами и `auth.users`. Аддитивно: контакты без ЛК работают как раньше.

**Риск ломки: низкий.** Нужно тестирование на проде после деплоя, но архитектурного слома нет.

### 12.3. Реально инвазивные (меняют модель)

Это уже не «добавил колонку», а переосмысление существующей модели. Требуют миграции данных и рефакторинга RLS:

- **Gmail per-workspace.** Сейчас `email_accounts.user_id` (ящик принадлежит юзеру). Нужно перевезти на `workspace_id` + добавить разрешения «кто из менеджеров может писать с этого ящика». Это переосмысление прав. Миграция данных существующих ящиков.

**Риск ломки: средний.** Нужно делать в отдельном релизе, с миграцией и обратно совместимым переходным периодом. Потенциально откатывается, но через бэкап.

### 12.4. Архитектура под продажу модулями

В системе уже есть механика `enabled_modules` (на уровне проекта/воркспейса — модули типа `digest`, `chats`, `documents` и т.д.). Новый функционал ложится в эту же модель:

- **Модуль `crm`** — флагует фичи: справочник контактов, шаблон «Лид», воронка как пресет доски, маршрутизация входящих в лиды, конверсия.
- **Модуль `integrations.telegram`** — per-workspace боты, групповые чаты с ботом.
- **Модуль `integrations.telegram_business`** — Telegram Business для личных аккаунтов.
- **Модуль `integrations.gmail`** — Gmail per-workspace.
- **Кастомные поля** — обычно базовая фича во всех тарифах, не отделять.

При выключенном модуле `crm` у воркспейса:
- Контактов нет в навигации.
- Шаблон «Лид» скрыт.
- Воронка-пресет не показывается.
- Маршрутизация входящих по контактам не работает (старая логика «один тред на чат»).
- Кнопка конверсии скрыта.

При этом **технически** контакты в БД могут существовать (если когда-то модуль был включён) — данные не теряются, просто UI и логика их не показывает. Это даёт чистый upgrade/downgrade без потерь.

### 12.5. Что критично заложить сейчас, чтобы потом не ломать

- **Все новые сущности привязывать к воркспейсу через `workspace_id`** — это позволяет включать/выключать модуль на уровне воркспейса.
- **Не вшивать «лид» в логику жёстко** — это просто шаблон проекта. Чтобы при выключенном `crm`-модуле проекты с этим шаблоном продолжали работать как обычные проекты (или скрывались — в зависимости от настройки).
- **Маршрутизация входящих** должна иметь fallback-режим: если модуль `crm` выключен, система работает по старой логике «один Telegram-чат = один тред», без поиска контактов.
- **`workspace_integrations`** изначально проектировать с прицелом на N типов — структура `type + config jsonb` это закладывает.

### 12.6. Резюме

Из 12 пунктов в порядке реализации (раздел 11):
- **9 пунктов — чисто аддитивные** (риск нулевой).
- **2 пункта полу-инвазивные** (telegram-webhook рефакторинг — низкий риск).
- **1 пункт реально инвазивный** (Gmail per-workspace — средний риск, нужна аккуратная миграция).

Архитектурно весь фрейм отделяется от ядра как **модуль `crm` + модули интеграций**. Ничто из этого не переплетается с базовыми сущностями (проекты, треды, права) так, чтобы при выключении модуля что-то сломалось. Можно продавать отдельно.

## 13. Что НЕ нужно делать

- Не создавать сущность `leads` — это `projects`.
- Не делать отдельный мессенджер для лидов — это `project_threads`.
- Не делать отдельный Inbox для лидов — расширяем существующий.
- Не делать pipeline-сущность — это шаблон проекта + статусы + Boards.
- Не использовать MTProto.
- Не привязывать Gmail-ящики к юзерам (после миграции п. 8).
