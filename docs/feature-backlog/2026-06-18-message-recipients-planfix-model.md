# План: адресаты сообщения (модель Planfix) — «доступ» vs «кому прилетит»

**Статус:** план, к реализации не приступали.
**Зона:** мессенджер (карантин — особенно стадия B, путь отправки).
**Дата:** 2026-06-18.

## 1. Идея

Сейчас «доступ к треду» и «кому прилетит уведомление/непрочитанное» — **одно и то
же**: непрочитанное считается всем, у кого есть доступ. Из-за этого: владелец/админ
видит красное на чужих командных чатах; отправитель не понимает, кто прочитает.

Модель Planfix разделяет два понятия:

| Понятие | Кто | Поведение |
|---|---|---|
| **Доступ** | широкий (роли/участие/assignee/member/view_all) | может открыть и прочитать |
| **Адресаты** | явный выбор над композером | получают уведомление + бейдж непрочитанного |

Над полем ввода — селектор с двумя группами:
- **Сотрудники** — галочка = внутреннее уведомление/непрочитанное. Без галочки: может зайти и прочитать, но не дёргается.
- **Контакты (клиент)** — галочка = **фактическая доставка во внешний канал** (TG/email). Без галочки = **внутренняя заметка** в клиентском треде (клиенту НЕ уходит).

Это разом решает: прозрачность (видно, кто адресат), отсутствие флуда, фантомное
непрочитанное у view_all, и управляемую границу «внутреннее / клиенту».

## 2. Модель данных

- **`message_recipients (message_id uuid, participant_id uuid)`** — кто адресован (сотрудники, получающие уведомление/непрочитанное). PK (message_id, participant_id). Индекс по participant_id.
- **`project_messages.deliver_external boolean DEFAULT true`** — уходит ли сообщение во внешний канал. Для внутренней заметки в клиентском треде = false. (Контакт-сторона селектора; у треда обычно один внешний канал, поэтому булева, а не per-contact.)

## 3. Непрочитанное = «адресовано мне» (главное изменение)

Сейчас `recompute_thread_unread_for` считает ВСЕ сообщения треда после `last_read_at`.
Станет — только адресованные мне:

```
unread = count(pm) WHERE pm.thread_id = T AND pm.sender <> me AND pm.created_at > last_read_at
  AND ( EXISTS(message_recipients mr WHERE mr.message_id=pm.id AND mr.participant_id=me)   -- новое: явный адресат
        OR NOT EXISTS(message_recipients mr WHERE mr.message_id=pm.id) )                    -- бэкафилл-фолбэк (см. §5)
```

**Фан-аут на запись становится дешевле и чище:** триггер инкрементит непрочитанное
только адресатам сообщения (а не всем с доступом). Это **в корне убирает** проблему
view_all-фантомов (владелец не адресат → не считается). Переиспользует материализацию,
которую уже построили (`thread_unread_state` + триггеры), — меняется только условие
подсчёта (join на `message_recipients`).

## 4. Дефолты и «отметить = вступить»

- **Что отмечено по умолчанию** при открытии композера:
  - Сотрудники: текущие участники треда (`project_thread_members` + assignees), НЕ пассивные view_all-наблюдатели.
  - Контакт: отмечен в клиентском треде (обычный ответ уходит клиенту); снимаешь — внутренняя заметка.
- **Отметить нового сотрудника = добавить в тред** (`project_thread_members`), чтобы он попал в будущие дефолты и увидел тред во «Входящих» (= join-on-write).
- **Написал в тред, где ты не участник → тебя добавляют участником** (тост «Вы добавлены · Отменить»). Решает исходный кейс владельца.

## 5. Миграция (без массового бэкафилла)

Старые сообщения не имеют строк в `message_recipients`. Фолбэк (см. формулу §3):
**сообщение без адресатов = адресовано всем текущим участникам/доступным** → старая
семантика сохраняется для истории. Новые сообщения используют явных адресатов.
Никакого тяжёлого бэкафилла истории не нужно.

## 6. Доставка во внешний канал (стадия B, карантин)

`deliver_external=false` → триггер маршрутизации (`dispatch_message_to_channels` /
`notify_telegram_on_new_message`) **не отправляет** во внешний канал (внутренняя
заметка). Нужен skip-кондишн в триггере + явный визуальный стиль «внутренняя заметка»
в баббле (чтобы не путать с доставленным). **Карантин** — трогает путь отправки,
смок-тест по всем каналам обязателен.

## 7. UI

- **Селектор адресатов над композером**: две группы (Сотрудники / Контакты), чекбоксы, преотмечены по дефолту (§4).
- **Внутренние заметки** (`deliver_external=false`) — отдельный стиль баббла (например, жёлтый фон «только команда»).
- (Опц.) В шапке треда — индикатор «кто видит» (доступ) через `inbox_accessible_participant_ids` — отдельно от «кто получит» (адресаты).
- Запоминать выбор адресатов в пределах сессии треда (не отмечать каждый раз).

## 8. Фазы

- **Фаза A — внутренние адресаты (низкий риск):** `message_recipients` + селектор «Сотрудники» + подсчёт непрочитанного по адресатам + фан-аут по адресатам + join-on-write + фолбэк-миграция. Переиспользует материализацию. **Уже решает: прозрачность, отсутствие флуда, view_all-фантомы.**
- **Фаза B — доставка контактам (карантин):** `deliver_external` + skip в маршрутизации + стиль «внутренняя заметка». Осторожно, смок по каналам.
- **Фаза C — полировка:** индикатор «кто видит» в шапке, запоминание дефолтов, мультиконтактные треды если понадобятся.

## 9. Риски и заметки

- **Карантин (Фаза B)** — путь отправки. Только со смок-тестом всех каналов.
- **Семантика непрочитанного меняется намеренно** — НЕ сверять с v2 (поведение улучшается осознанно). Сверять с НОВЫМ ожиданием на тест-кейсах.
- **Дефолты адресатов** — продуктовая тонкость, неудобный дефолт раздражает.
- **Перф:** `message_recipients` добавляет join в подсчёт непрочитанного — индекс по participant_id обязателен; материализованный `thread_unread_state` всё равно держит итог по индексу.
- **Связь с моделью доступа:** «адресат» ⊆ «имеет доступ». Нельзя адресовать тому, у кого нет доступа к треду (или адресация автоматически даёт доступ через membership — решить).
- **Реакции/события** — оставить как есть (адресность только у сообщений) или тоже привязать к адресатам — решить отдельно.

## 10. Рекомендация

Начать с **Фазы A** — низкий риск, переиспользует материализацию, и сразу даёт
главный продуктовый эффект (прозрачность + конец флуда + фикс view_all). Доставку
контактам (Фаза B, карантин) — отдельным заходом со смок-тестом.
