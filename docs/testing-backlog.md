### 🔴 Волна 1c — реалтайм-хуки (приоритет 1)

**Что**: `useProjectMessages` (226 строк), `useDocuments` (не разведано).

**Почему важно**: критичные хуки с подписками на Realtime, optimistic
updates, мутациями. Без тестов — любой регресс в инвалидации кэша или
сценариях оффлайн-онлайн ломает мессенджер незаметно.

**Сложность**: высокая. Нужно расширить мок `supabase.channel()` в
`src/test/setup.ts` — сейчас он минимальный и не поддерживает
проверку подписок. Это блокер: без расширения мока тесты не написать.

**План**:

1. Расширить мок `channel()` в setup.ts: `on()` должен записывать
   подписки в массив, `subscribe()` должен возвращать объект с
   методами, `removeChannel()` — снимать подписки.
2. Тесты `useProjectMessages`: загрузка истории, optimistic send,
   получение Realtime сообщения, обновление кэша React Query,
   обработка ошибок отправки, deleteMessage, editMessage.
3. Разведать `useDocuments` — только после `useProjectMessages`,
   потому что там может быть upload через `supabase.storage`,
   что потребует ещё одного расширения моков.

**Оценка**: 1-2 сессии. Очень полезно, но дорого.

### 🟢 Волна E2E — Playwright (приоритет отдельный)

**Что**: 5-10 ключевых сценариев "прокликивания":

1. Логин → редирект на dashboard
2. Создание воркспейса → проекта → треда
3. Отправка сообщения в тред
4. Создание задачи → переключение статуса → закрытие
5. Корзина: удалить → восстановить → удалить навсегда
6. Права доступа: чужой не видит чужое (заходим разными аккаунтами)
7. Загрузка документа → AI-проверка → редактирование
8. Добавление в knowledge base → поиск → удаление

**Почему**: юнит-тесты не ловят ошибки реальной интеграции
(сломанная вёрстка, неправильный роутинг, косяки авторизации).
E2E — единственный способ убедиться, что прод **реально работает
для пользователя**.

**Сложность**: высокая, отдельная категория работы.

**План**:

1. Поставить Playwright (`npm i -D @playwright/test`).
2. Создать `playwright.config.ts` и `e2e/` директорию.
3. Написать первый тест — логин (самый базовый).
4. Расширять по одному сценарию за коммит.
5. **Не запускать E2E в quality gate сразу** — сначала отдельным
   job, который не блокирует деплой. Потом, когда стабилизируются —
   перенести в blocking.
6. Интеграция с Playwright MCP (есть в окружении) для отладки.

**Оценка**: 3-5 сессий на минимальный набор. Это отдельный
большой проект.

## Принципы (не забывать)

Эти правила выработаны за прошлые волны. Не нарушать без сильной
причины.

1. **Не мокать всё подряд.** Мокать только границу с внешним миром
   (Supabase клиент, контексты). Внутренняя логика — настоящая.
   Иначе тестируется мок, а не код.

2. **Контрактные проверки важнее проверок результата.** Тест должен
   проверять, что метод вызвал `.eq('is_deleted', false)`, а не
   только что вернул правильный массив. Иначе можно случайно убрать
   фильтр, а тест с моком пройдёт.

3. **Тесты должны падать когда код ломается.** Плохой тест — тот,
   который всегда зелёный независимо от кода. Перед коммитом
   мысленно проверь: "если я сейчас сломаю экшен, упадёт ли тест?"

4. **`TaskPanel.tsx` не трогать.** Это незакоммиченное изменение из
   параллельной сессии, висит в working tree уже несколько коммитов.
   Не мой файл.

5. **Параллельные сессии могут менять `package.json`.** Перед коммитом
   проверять `git diff package.json` — там могут быть чужие изменения,
   которые не относятся к моей работе. Стейжить только своё.

6. **Стиль тест-описаний — на русском, простыми словами.** "должен
   вернуть пустой массив если нет участников", а не "should return
   empty array when no participants".

7. **Каждый коммит — одна логическая волна.** Не смешивать тесты
   из разных областей в одном коммите. Имя коммита: `Тесты: Волна N
   — <тема> (X файлов, +Y тестов)`.

8. **localStorage в jsdom этого проекта неполный.** Если тестируешь
   код, который использует localStorage напрямую — мокай через
   `vi.stubGlobal('localStorage', createLocalStorageMock())`.
   Шаблон есть в `src/store/sidePanelStore.test.ts`.
