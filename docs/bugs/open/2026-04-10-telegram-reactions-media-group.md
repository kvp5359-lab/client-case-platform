## Симптомы

Когда клиент ставит реакцию в Telegram на сообщение, для которого у нас в БД нет соответствующего `telegram_message_id`, реакция прилетает на сайт **как отдельное текстовое сообщение с эмодзи**, а не как реакция под сообщением.

Пример (скрин от 10.04.2026): в треде «ТЕСТ / Клиент» внизу ленты три «паразитных» сообщения — `❤`, `👍`, `🔥` — хотя на самом деле это реакции, поставленные в Telegram на сообщения с файлами.

На одиночные текстовые сообщения («привет», «аивипаи», «прарапртаит») реакции приходят **корректно** — как эмодзи под сообщением.

## Как воспроизвести

**Кейс 1 — media group:**
1. Отправить с сайта в Telegram сообщение с текстом + 2 и более файлами
2. В Telegram поставить реакцию **на второй** (или следующий) файл в группе, не на первый
3. На сайте появится новое сообщение с эмодзи в ленте чата вместо реакции

**Контроль (работает правильно):**
- Реакция на одиночное текстовое сообщение без файлов → появляется корректно
- Реакция на одиночный файл без группы → предположительно работает (не проверено железно)

## Расследование — 2026-04-10

### Факты

1. **`handleReaction`** в telegram-webhook (строки 793-899 в `supabase/functions/telegram-webhook/index.ts` оригинального проекта ClientCase) ищет сообщение по паре `(telegram_chat_id, telegram_message_id)`. Если не нашёл — срабатывает **fallback**, который создаёт новую запись в `project_messages` с `content = emoji`, `telegram_message_id = null`.

2. **`telegram-send-message`** при отправке media group сохраняет в `project_messages.telegram_message_id` **только `message_id` первого файла** из группы:
   ```ts
   // строки 439-443 в supabase/functions/telegram-send-message/index.ts
   const firstMsgId = Array.isArray(tgData.result) ? tgData.result[0]?.message_id : null;
   ...
   .update({ telegram_message_id: firstMsgId, ... })
   ```
   Остальные файлы группы в БД как отдельные записи не сохраняются.

3. **Fallback добавлен коммитом `4eef699` от 10 марта 2026** в проекте ClientCase: "feat: messenger improvements — file captions, large file warnings, **reaction fallback**, scroll fix". До этого коммита было `if (!msg) return;` — реакция на ненайденное сообщение просто игнорировалась.

4. **В БД на 10.04.2026 найдено 10+ записей** вида «реакция как сообщение» в разных чатах, начиная с 10 марта. Ранее таких не было. Это совпадает с датой коммита.

5. **Запрос для проверки** (работает на общей БД Supabase):
   ```sql
   SELECT created_at, content, telegram_chat_id, thread_id
   FROM project_messages
   WHERE source = 'telegram' AND telegram_message_id IS NULL
   ORDER BY created_at DESC;
   ```

### Гипотеза

Telegram Bot API при реакции в media group присылает `message_reaction` с `message_id` **конкретного файла**, на который нажал пользователь. Если пользователь нажал на второй/третий/... файл — в нашей БД такого `telegram_message_id` нет (мы сохраняем только первый), поиск `handleReaction` возвращает пусто, срабатывает fallback и создаёт паразитное сообщение.

Гипотеза объясняет все наблюдаемые факты, но **не доказана железно** — см. раздел «Что не проверено».

### Что НЕ проверено

- **Нет логов тел webhook-запросов.** Supabase Edge Function logs показывают только HTTP status code (200/500), без тел запросов. Невозможно увидеть, какой именно `message_id` Telegram шлёт в конкретном случае.
- **Реакция на одиночный файл (без группы) не проверена как контроль.** Если она сработает корректно — гипотеза подтвердится; если тоже упадёт в fallback — значит проблема шире (возможно, одиночные файлы тоже не сохраняют свой `telegram_message_id`).
- **Не проверено** поведение `telegram-send-message` при отправке только файлов **без текста** (attachments_only).
- **Сценарии, когда fallback-сообщения попадают не в тот тред**: в БД найдены случаи, когда `telegram_chat_id` одной Telegram-группы привязан к разным тредам в разное время (пересоздание привязки). Требует отдельного изучения.

## Варианты фикса

### A. Откатить fallback (быстро, частичная потеря данных)

Вернуть `if (!msg) return;` в `handleReaction`. Реакции, которые не нашли исходник, молча игнорируются.

- **Плюс:** чат перестаёт засоряться, возврат к поведению до 10 марта.
- **Минус:** реакции на 2-й+ файл в группе теряются полностью — они не будут видны ни как реакция, ни как сообщение.
- **Объём:** одна правка в telegram-webhook, 5 минут + деплой.

### B. Сохранять все `telegram_message_id` группы (правильно, долго)

1. Добавить колонку `telegram_message_ids int[]` в `project_messages` (или отдельную таблицу `project_message_telegram_ids`).
2. В `telegram-send-message` при media group сохранять **массив всех** `message_id`, которые Telegram вернул.
3. В `handleReaction` искать по `telegram_message_id = ANY(telegram_message_ids)`.
4. (Опционально) миграция для исторических сообщений — не делать, просто работать от новых.

- **Плюс:** реакции на любой файл группы найдут исходное сообщение.
- **Минус:** миграция БД + 2 edge-функции + возможно, типы на фронте.
- **Объём:** несколько часов работы.

### C. Hybrid — fallback пишет в `message_reactions`, а не в `project_messages`

Вместо создания нового `project_messages` — попытаться найти **ближайшее** сообщение треда по времени и записать реакцию в `message_reactions` с привязкой к нему.

- **Плюс:** чат не засоряется, реакции всё-таки видны как реакции.
- **Минус:** эвристика «ближайшее» может ошибаться (например, если между файлом и реакцией пришло чьё-то новое сообщение).
- **Объём:** умеренно, ~час работы.
