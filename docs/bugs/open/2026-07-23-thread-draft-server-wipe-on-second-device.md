# Открытие треда на втором устройстве стирает серверный черновик поля ввода

- **Severity:** high (потеря данных фичи кросс-девайс черновиков)
- **Область:** messenger, thread_input_drafts, useDraftMessage, MessageInput
- **Замечен:** 2026-07-23 (в живом эксперименте при замере производительности)

## Симптом

Пользователь набрал черновик на устройстве A (текст сохранён на сервер в `thread_input_drafts`,
в списке «Входящие» виден чип «Черновик: …»). Устройство B (свежая вкладка/другой компьютер,
БЕЗ локального localStorage-черновика) просто **открывает этот тред и переключается на другой** —
ничего не печатая. Результат: строка в `thread_input_drafts` **удаляется** (наблюдён
`DELETE /rest/v1/thread_input_drafts?thread_id=eq.<id>&user_id=eq.<me>` при переключении).
Чип «Черновик» пропадает; другие устройства черновик больше не увидят. На устройстве A текст
остаётся только в localStorage/редакторе — при следующей правке пересохранится, но если
устройство A недоступно, черновик потерян насовсем.

## Доказательство (2026-07-23, прод)

Замер производительности из отдельной вкладки Chrome того же пользователя: открытие
прочитанных тредов кликами (без набора текста) → в network-логе DELETE thread_input_drafts
на каждый открытый тред с серверным черновиком. После сессии `SELECT * FROM thread_input_drafts
WHERE user_id = '<owner>'` → **0 строк** (до сессии черновики были — чипы «Черновик: Ну значит»
и «Черновик: 📎Файл» видны на скриншоте пользователя). `thread_input_draft_files` тоже 0 строк —
вероятно, зеркальная логика `useMessageFiles` (mirror пустого локального набора) стёрла и файлы,
но этот путь отдельно не перехвачен.

## Гипотеза механизма (не подтверждена пошагово)

`useDraftMessage` (src/components/messenger/hooks/useDraftMessage.ts):
1. На маунте у устройства B localStorage пуст → `editor.clearContent()`; какой-то путь зовёт
   `saveDraft('', '')` (пустой контент) → remote-debounce 2с планирует `saveThreadDraft(..., '')`,
   а пустой контент в сервисе = DELETE строки.
2. Restore серверной версии (`getThreadDraft` → `setContent(remote.content)`) применяется, но
   tiptap `setContent`/`clearContent` по умолчанию НЕ эмитят update → `saveDraft` не
   перепланируется с непустым контентом → отложенный «пустой» save побеждает и удаляет сервер.
3. Race с 500мс localStorage-таймером усугубляет.

Требуется замер (логи в saveDraft/restore), НЕ гадать — см. глобальное правило отладки.

## Варианты фикса (обсудить)

- При restore/clearContent НЕ планировать remote-save вовсе (remote-save только на реальный
  пользовательский ввод — например, флаг «был user-originated update после маунта»).
- `saveThreadDraft('')` не должен удалять строку, если её `updated_at` новее момента маунта
  композера (optimistic concurrency: `DELETE ... WHERE updated_at <= <mount_ts>`).
- Отменять pending remote-таймер при применении серверного restore.
