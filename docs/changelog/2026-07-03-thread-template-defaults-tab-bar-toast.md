## 1. Вкладки правой боковой панели (`TaskPanelTabBar`)

- **Закреплённые вкладки всегда видимы, прокручиваются только незакреплённые.**
  Раньше все вкладки (закреплённые + разделитель + обычные) лежали в одном
  скролл-контейнере. Теперь ряд разбит: закреплённые — в блоке `shrink-0` (не
  уезжают, не скроллятся), разделитель, обычные — в `overflow-x-auto`. DnD не
  затронут (dnd-kit работает по списку `sortableItems`, а не по вложенности DOM).
- **Активная вкладка всегда на экране.** Поиск активной вкладки для автопрокрутки
  переведён со скролл-контейнера на весь ряд (`rowRef`): закреплённая активная и
  так видна, незакреплённая проматывается внутри своего контейнера.
- **Фикс `setState` во время рендера (`TaskPanelTabbedShell.togglePanel`).**
  Навигация роутера (`activateTab`/`clearUrlActive` → `router.replace`)
  вызывалась **внутри** функционального апдейтера `setHidden((h)=>…)`. Апдейтер
  React выполняет во время фазы рендера → «Cannot update a component (Router)
  while rendering». Побочки вынесены из апдейтера (читаем `hidden` из замыкания,
  `setHidden(next)` обычным значением, навигация — в обработчике).

## 2. Шаблоны тредов — дефолты и вид как в форме задачи

Редактор шаблона треда (`ThreadTemplateDialog`/`ThreadTemplateFields`)
приведён к виду формы создания задачи (`ChatSettingsDialog`), только в редакторе
шаблона (флаг `taskStyleThreadBlock`) — окно «Повторяющиеся» не тронуто.

- **Проект по умолчанию.** Новая колонка `thread_templates.default_project_id`.
  Если задан — тред из шаблона (из глобального меню «+/Новый», где нет контекста
  проекта) сразу создаётся в этом проекте. Внутри проекта контекст побеждает.
  Применение — в `QuickActionsProvider.runNewThread`
  (`action.targetProjectId ?? template.default_project_id`).
- **Описание по умолчанию.** Колонка `thread_templates.default_description` —
  подставляется в `project_threads.description` создаваемого треда (внутренняя
  заметка команды). Отдельно от `thread_templates.description` (та описывает сам
  шаблон, видна в списке/поиске; в редакторе переименована в «Описание шаблона»).
- **Единая рамка** «Название треда + Описание» (Gmail-стиль): строка
  `[статус-точка] [название] [иконка]` → разделитель → textarea описания.
  Переиспользованы `ChatSettingsStatusPopover`, `ChatSettingsIconColorPicker`.
- **Чипы под рамкой** `[Срок] [Статус] [Выбрать проект]` (`ChatSettingsProjectSelector`).
  - **Дедлайн** — чип в стиле «Срок» задачи, но внутри поле числа дней («дней
    после создания»); хранится как `deadline_days` (шаблон многоразовый — дата не
    устаревает). Календарь намеренно НЕ используется.
  - **Статус** — текстовый цветной чип (как «Новая задача» в задаче).
- **Унификация задача/чат/email:** статус и дедлайн теперь доступны во всех типах
  шаблона (сняты type-гейты в `ThreadTemplateFields` и `handleSave`). Автопереход
  проекта («При завершении…») остался task-only.
- **Блок первого сообщения / письма — на акцентном фоне** (`acc.bgSoft`), как в
  задаче. Email собран в единый блок «Письмо» (иконка конверта): «Кому»
  (`EmailRecipientInput` с префиксом `Кому:` внутри поля — новый проп `prefix`),
  «Тема:» (префикс внутри поля), «Текст письма» (белое поле).

## 3. Исполнитель из шаблона теперь реально применяется (fix)

- **Симптом:** в шаблоне выбран исполнитель (владелец «Я»), при создании задачи из
  шаблона исполнитель пустой.
- **Причины (две):** (1) `applyTemplate` отсеивал исполнителей, которых нет в
  участниках проекта (владелец не в `project_participants`); (2) гонка —
  применение шаблона срабатывает при открытии, до загрузки участников проекта →
  набор пуст → отсеивались все.
- **Фикс:** `applyTemplate` применяет **всех** исполнителей шаблона безусловно
  (сохранение берёт сырой набор `form.taskAssignees` → назначение проходит;
  назначение и так даёт доступ к задаче без доступа к проекту). Для отображения
  чипа попап исполнителей в форме создания использует **объединённый** список
  (участники проекта + сотрудники воркспейса, `assigneeParticipants`);
  workspace-участники грузятся в форме всегда. `effectiveParticipants` (упоминания/
  дефолты) не тронут.
- **Побочно (осознанно):** в задаче внутри проекта в исполнители можно назначить
  любого сотрудника воркспейса — единообразно с редактором шаблона и моделью
  доступа.

## 4. Тост нового сообщения — верхняя строка как во «Входящих»

`MessageToastContent`: верхняя строка теперь **Проект** *(Тред)* (проект жирным,
тред в скобках серым) — как заголовок в списке «Входящих». Для личного диалога
(без проекта) — имя треда. Имя отправителя — отдельной строкой ниже (не
дублируется, если совпадает с заголовком). Дальше — текст сообщений, как раньше.

## Грабли

- В редакторе шаблона `new Date()` в теле рендера ловит lint `react-hooks/purity`
  — расчёт даты держать в эффекте/обработчике (в итоге для чипа дедлайна дата не
  понадобилась — храним дни).
- Любой новый потребитель «моей личности»/исполнителей из шаблона должен
  переживать случай, когда участники ещё не загружены — не гейтить по ним.
- Селектор проекта/статуса/иконки в редакторе шаблона — те же leaf-компоненты, что
  в форме задачи; фон блоков — `acc.bgSoft` (в safelist уже есть).
