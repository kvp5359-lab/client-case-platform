# Подпись имени отправителя в Telegram-боте: фильтр по thread_id

**Дата:** 2026-04-29
**Тип:** fix
**Статус:** completed

---

## Контекст

При отправке сообщений из ЛК в Telegram-группу через бота имя отправителя добавляется первой строкой (`<b>Кирилл:</b>\n…`). Чтобы не дублировать имя в каждом сообщении подряд, есть правило: если предыдущее сообщение в этом же канале за последние 60 минут от того же сотрудника через web — имя скрыть.

Пользователь поймал баг: отправил два сообщения через бота, между ними клиент ответил из Telegram. Второе сообщение от сотрудника пришло **без подписи имени** — хотя по логике подпись должна была восстановиться (между ними было сообщение клиента).

## Причина

Запрос «последнее сообщение в этом же канале» в `telegram-send-message` фильтровал по `project_id` + `channel`, **но не по `thread_id`**. У клиентских тредов проекта общий `channel = "client"`, поэтому если тот же сотрудник за последние 60 минут писал в **соседний тред** того же проекта (Касание, Финмодель и т.п.), система брала это сообщение как «последнее», совпадал sender_name + source="web" → имя ошибочно скрывалось. Сообщение клиента в нужный тред в фильтр не попадало, потому что в соседнем треде было сообщение свежее.

## Решение

В `telegram-send-message`:

1. Подтянуть `thread_id` текущего исходящего сообщения по `body.message_id` из `project_messages`.
2. Если `thread_id` есть — фильтровать «последнее сообщение» по `thread_id` (точная корректная область).
3. Fallback на старую логику (фильтр по `channel`) только если у сообщения нет `thread_id` — для совместимости с возможными edge-кейсами.

## Файлы

- `supabase/functions/telegram-send-message/index.ts` — фильтр по thread_id + fallback

## Почему так

**Thread_id — корректная область «диалога».** Channel — это широкая категория (client/internal/etc). В одном проекте может быть пять клиентских тредов, и они независимы — рамка «не дублировать имя подряд» должна работать в пределах одного диалога, не пересекать треды.

**Fallback оставлен.** Сообщение без `thread_id` — теоретически возможно (старые записи, нестандартные пути insert'а). Чтобы не сломать им подпись, оставляем старое поведение как fallback.

**Деплой**: функция деплоится с `--no-verify-jwt` (вызывается из БД-триггера `notify_telegram_on_new_message` через `net.http_post`, без пользовательского JWT).
