# Имя файла в превью входящих для сообщений без текста

**Дата:** 2026-04-23
**Тип:** feat
**Статус:** completed

---

## Контекст

В списке «Входящие» (`InboxSidebar`) если последним сообщением в треде был только файл без подписи, превью показывало одиноко стоящую эмодзи-скрепку «📎» (Telegram-бот кладёт такой плейсхолдер в `content` для сообщений без текста). По имени файла непонятно, что вообще прислали, и приходилось открывать тред. Пользователь попросил показывать имя файла.

---

## 1. Миграция RPC `get_inbox_threads_v2`

Миграция [`20260423_inbox_attachment_preview.sql`](../../supabase/migrations/20260423_inbox_attachment_preview.sql). В функцию добавлены два выходных поля:

- `last_message_attachment_name text` — имя первого по `created_at` вложения у последнего сообщения треда.
- `last_message_attachment_count integer` — общее число вложений у этого сообщения.

Реализация — CTE `last_message_attachments` с двумя коррелированными подзапросами к `message_attachments`, оба опираются на существующий индекс `idx_message_attachments_message_id`. `last_messages` и раньше был `DISTINCT ON (thread_id)` — 1 строка на тред, поэтому подзапросы выполняются по числу аксессибельных тредов, а не по всем сообщениям.

**Производительность.** Замеры на продовой БД, воркспейс с 278 тредами:
- `EXPLAIN (ANALYZE, BUFFERS)` — 37 ms total execution.
- Добавленные подзапросы — ~556 индекс-lookup'ов по btree, ~1-2μs каждый, суммарно ~1 ms к запросу.
- Bottleneck остался прежним — сам function scan тредов со всеми агрегатами.

## 2. Типы и превью

- [`InboxThreadEntry`](../../src/services/api/inboxService.ts) — расширен двумя полями.
- [`InboxChatItem.tsx`](../../src/components/messenger/InboxChatItem.tsx) — логика определения что показывать в превью:
  - Если у сообщения есть текст И (нет вложений ИЛИ текст — не плейсхолдер) → показываем текст (прежнее поведение).
  - Если текста нет ИЛИ текст — короткий эмодзи-плейсхолдер от бота → показываем имя файла с иконкой `Paperclip`. При `count > 1` рядом индикатор «+N».

Определение плейсхолдера — helper `isAttachmentPlaceholderText`: считает таковым любую строку длиной ≤4 юникод-юнитов без букв/цифр (эмодзи/символы). Обычная подпись к файлу будет длиннее или содержать буквы.

**Почему не просто «текст пустой → показать имя файла».** Бот шлёт сообщения с `content = '📎'` (один эмодзи) для документов без подписи. Поле не пустое, просто бесполезное. Фильтровать на уровне бота — отдельная задача (менять серверную функцию `telegram-webhook`), на клиенте это решается одной проверкой — дешевле и не трогает приход данных.

## Результат

Превью «Входящих» теперь показывает `Maxim: 📎 Brief_Bogdanov.docx` вместо `Maxim: 📎` — видно имя файла сразу, без захода в тред. Для сообщений с реальной подписью к файлу поведение прежнее: показывается подпись.
