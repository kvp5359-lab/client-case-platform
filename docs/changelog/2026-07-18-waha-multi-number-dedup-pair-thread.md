# WAHA multi-number: дедуп по ядру + атрибуция по отправителю + общий тред пары коллег + группа с двумя нашими номерами

**Дата:** 2026-07-18
**Тип:** feat + fix (мессенджер WAHA, карантин) + БД (2 миграции)
**Статус:** edge задеплоены в прод (CLI), миграции через MCP; ⏳ живой смок за владельцем

---

Закрытие двух multi-number сценариев WhatsApp (WAHA). Диагностика — на реальных id
прода. Детальное расследование — [`messenger-ledger.md`](../../.claude/rules/messenger-ledger.md)
запись 2026-07-18 (5).

## Общий корень

Дедуп WhatsApp-сообщений был по **полному** `waha_message_id`, а две наши сессии
видят одно физическое сообщение с **разным** полным id (`true_<чат>_<ЯДРО>` vs
`false_<чат>_<ЯДРО>`). У групп полный id совпадал → работало; у 1:1 коллег и у наших
сообщений в общей группе — расходился → задваивалось + неверная атрибуция.

## Фаза 0 — фундамент (оба сценария)

- **Дедуп по ЯДРУ** (`project_messages.waha_msg_core` — id сообщения без префикса
  направления/чата) + backfill + partial-unique. Одно физическое сообщение = одна
  запись при любом числе наших сессий. Экстрактор `wahaMsgCore` в
  `_shared/whatsappThread.ts`. Обычные клиентские треды (одна сессия) не затронуты.
- **Атрибуция по ТЕЛЕФОНУ отправителя** (не по `fromMe` наблюдателя): наш
  подключённый номер → тот сотрудник (staff), иначе клиент. Обходит баг NOWEB #1350
  (`fromMe=false` у своих в группе). Гейт по `multiNumber` (>1 наших сессий).
- `source==='api'` (наши сервисные отправки) — skip на приёме; `waha-send/markSent`
  тоже пишет `waha_msg_core`.

## Фаза 1 — сценарий 2 (оба наших номера в клиентской группе)

Вытекает из Фазы 0: наши сообщения не двоятся (дедуп по ядру), коллега помечается
сотрудником (атрибуция), оба уже участники общего треда (`ensureThreadMember`).
Отдельного кода не потребовалось.

## Фаза 2 — сценарий 1 (1:1 коллег → один общий тред)

- Миграция: `project_threads.whatsapp_pair_key` (неупорядоченная пара телефонов) +
  partial-unique.
- `ensureWahaThread`: если собеседник по телефону — наш подключённый номер →
  `ensurePairThread` (общий тред по паре, оба в `project_thread_members`), а не
  per-owner тред.
- `waha-send`: для pair-треда сессия = автора, цель-чат = **второй номер пары**
  (двунаправленно). Триггер маршрутизирует как обычно (waha_session_id непуст),
  реальные сессия/чат waha-send пере-резолвит.

## Ограничение

**Существующие** per-owner треды коллег (созданные до фичи) не мигрируют
автоматически — `findThread` находит старый и переиспользует. Пара-объединение
работает для НОВЫХ переписок (старые тест-треды удалены для чистого теста).

## Проверки

- `deno check` чист (waha-webhook, waha-send); миграции в проде через MCP
  (backfill, 0 коллизий ядер). Фронт не менялся.

## ⏳ Живой смок (карантин, за владельцем)

- Сц.2: оба номера в клиентской группе → клиент = одна запись; я/коллега в группе =
  по одной, атрибуция верная, без дублей.
- Сц.1: свежая 1:1 переписка коллег → один общий тред, оба видят, авторы верные,
  ответы в обе стороны.
- Регресс: обычный клиент 1:1/группа + сервис-отправки без дублей.
