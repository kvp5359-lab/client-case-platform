# 2026-07-21 — Реакция не воскрешает прочитанный чат + реакция поднимает чат вверх в инбоксе

## Что было не так

- Приходит реакция в Telegram → чат непрочитан → жмёшь «Прочитано» → погас → через
  час-два-три тот же чат снова непрочитан **из-за той же реакции**.
- Плюс визуальное: чат с реакцией показывал **время реакции** (напр. 20:37), но
  стоял на позиции последнего **сообщения** (старее) — «время не совпадает с
  позицией».

## Корень (замер, не гадание)

- Снял живое тело `recompute_thread_unread_for`: реакция считается непрочитанной,
  если `mr.created_at > v_wm`, где `v_wm = GREATEST(last_read_at, max(created_at
  СВОИХ сообщений))`. «Прочитано» (`markAsRead`) двигает `last_read_at = now()` →
  сразу гаснет. Own-message watermark (фикс 2026-07-10) реакции НЕ покрывает,
  отдельного «реакцию видел» в схеме нет.
- **Реакции Telegram синкались через DELETE + INSERT** (`_shared/syncTelegramReactions.ts`),
  а `message_reactions.created_at = DEFAULT now()`. Telegram переприсылает одно и то
  же обновление реакции (изменение набора реакций на сообщении, повторная доставка и
  особенно **несколько ботов в группе — каждый бот получает копию события**). Каждый
  повтор пересоздавал строку с `created_at = now()` → время реакции прыгало вперёд ЗА
  момент «Прочитано» → `has_unread_reaction=true` → чат воскресал.
- Доказательство (прод): тред `adbb9b0d…` — 6 реакций на 6 РАЗНЫХ сообщений (от 12.07)
  имеют `created_at` в одном 2-секундном окне 13.07 10:01:56–58 → явная пакетная
  переустановка при ре-синке.
- Сортировка инбокса **нигде** не учитывала реакцию (везде `GREATEST(last_message_at,
  last_event_at)`), а превью/время строки берутся из реакции → отсюда рассинхрон
  «время vs позиция».

## Фикс 1 — идемпотентный синк реакций (корень)

- Обе функции (`syncTelegramReactions`, `syncTelegramReactionsAggregated`) переведены
  с «снести всё + вставить заново» на **идемпотентную сверку**: снимаем только
  исчезнувшие эмодзи, вставляем только новые, у оставшихся `created_at` сохраняется.
  Пустой `new_reaction` = снять все (поведение сохранено). Защита соседних баблов
  (`telegram_source_message_id`) и «снять реакцию» целы.
- Выбор поля времени проверен на данных прода: идемпотентная вставка надёжнее поля
  `date` от Telegram — у 100% реакций есть естественный ключ (анонимных 0; уникальные
  индексы `message_reactions_telegram_unique` / `..._participant_unique` уже есть), а
  у внутренних реакций из приложения telegram-даты нет вовсе.

## Фикс 2 — реакция поднимает чат вверх (сортировка инбокса)

- Добавил время реакции в сортировку в двух местах (`CREATE OR REPLACE`, сигнатуры те
  же → гранты целы):
  - `compute_thread_inbox_meta` — `+v_lr.created_at` в глобальный `sort_at` (вкладка
    «Все» / keyset `get_inbox_threads_page`, любая реакция);
  - `get_inbox_unread_threads_impl` — время реакции в ORDER BY с гейтом
    `has_unread_reaction` (уже прочитанная реакция чат наверху не держит).
- Разово пересчитал 108 уже отреагированных тредов (`compute_thread_inbox_meta`),
  чтобы встали по времени реакции сразу.
- Делал ПОСЛЕ фикса 1 — иначе перепрыгивающее `created_at` дёргало бы чаты вверх без
  причины. Вкладки «Нужно ответить / Ждём клиента» не трогал (там смысл «кто ответил
  последним», реакция ≠ ответ).

## Грабли

- Реакции Telegram синкать ТОЛЬКО идемпотентно (сохранять `created_at` у
  неизменившихся) — `DEFAULT now()` + DELETE+INSERT переписывает время и после
  «Прочитано» воскрешает чат. Мультибот = усилитель, не корень.
- Реакция входит в сортировку инбокса в ДВУХ местах: `sort_at` (глобально, любая
  реакция) и ORDER BY unread-вкладки (гейт `has_unread_reaction`, пер-юзер) — держать
  оба; `get_inbox_threads_page` берёт `sort_at` из меты, отдельно не сортирует.
- ⚠️ Дрейф манифеста: изменены `compute_thread_inbox_meta` и
  `get_inbox_unread_threads_impl` → хеши в `schema-manifest.json` разошлись; обновить
  `db-drift-check --update` при случае.
