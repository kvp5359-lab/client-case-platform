# Единый механизм «базовый шаблон + переопределения»: канал — такой же владелец, как шаблон проекта

**Дата:** 2026-07-15
**Тип:** refactor + feat + test (шаблоны тредов: БД, фронт, приём лид-ботов)
**Статус:** БД — в проде (12 миграций); фронт — push в main → CI/CD; edge `telegram-webhook-v2` — ждёт ручного редеплоя (CI его не катит)

---

Продолжение лид-ботов ([2026-07-13](./2026-07-13-telegram-lead-bots.md)). Вместо
отдельных полей «иконка/цвет/ответственные» у бота канал теперь указывает
**шаблон диалога** и может переопределить его поля — ровно тем же механизмом, что
шаблоны проекта. Решение владельца — «правильно с нуля», поэтапно.

Мотив унификации на дистанции: каналов будет несколько, и «своё поле у каждого»
не масштабируется, а «базовый шаблон + переопределения» уже работает у проектов.

Подробности расследования — в [`messenger-ledger.md`](../../.claude/rules/messenger-ledger.md)
(журнал, раздел «2026-07-13 (5)», подразделы про обобщение junction и ревью-раунды).

## Фаза 1: junction обобщена на владельца «проект ИЛИ канал»

`project_template_thread_templates` приведена к идеальной форме без окна поломки:

- суррогатный PK `id`; исполнители (`project_template_thread_assignees`) переехали
  на `binding_id` (PK `(binding_id, participant_id)`);
- владелец через **exclusive-arc**: `template_id` (nullable) XOR `integration_id`
  (nullable FK → `workspace_integrations`) + CHECK `num_nonnulls = 1`;
  partial-uniques `uq_pttt_project` / `uq_pttt_integration`;
- RLS обеих таблиц обобщены (ветка владельца проект-шаблон ИЛИ интеграция;
  запись — Владелец/Администратор, как в соседней политике `20260622`).

**Backward-compat:** триггер `ptta_fill_binding_id` заполняет `binding_id` из пары
(template_id, thread_template_id) → старый проектный фронт пишет исполнителей БЕЗ
правок. Миграция `20260713170000`.

> 🪤 Порядок в миграции: `DROP NOT NULL` нельзя, пока колонка в составном PK —
> сначала смена PK, потом nullable + CHECK.

## Фазы 2–4: одна функция применения на всех

- **RPC `resolve_thread_template_binding(binding_id)`** — владелец-агностичный
  folding «переопределение ?? база» (срок/доступ/сообщение/статус), иконка и цвет
  всегда из базового шаблона, исполнители по режиму. `resolve_thread_template_for_integration`
  — обёртка «привязка канала, иначе базовый шаблон» (её зовёт приём).
- **Приём** (`lead.ts`): `resolveLeadTemplate` → одна RPC; ручной override через
  `config.responsible_user_ids` для шаблонного пути убран.
- **Создание проектов** (`seedProjectContent`) переведено на ту же RPC — folding
  больше не дублируется на фронте.

**🔴 Сверка нашла реальный баг в моей же RPC:** `access_roles` брались
`COALESCE(b.access_roles, tt.access_roles)`, а фронт берёт роли переопределения
ТОЛЬКО при заданном `b.access_type` → при override access_type без ролей
подставлялись базовые роли вместо пустых. Исправлено (`20260715140000`); после
фикса сверка RPC против формулы фронта **на всех 38 привязках — 0 расхождений**.

## Шаблоны треда: унификация полей

- **Исполнители и срок — для всех типов треда** (задача/чат/email): гейты
  `isTask||isEmail` сняты в UI, форме и применении. Срок у чата раньше задавался,
  сохранялся и **молча отбрасывался**. В базе чат/email-шаблонов с
  исполнителями/сроком не было → созданные проекты не задеты.
- **«Создатель задачи»** — псевдо-исполнитель в списке (`CREATOR_ASSIGNEE_ID`,
  флаг `thread_templates.assign_to_creator`, миграция `20260715150000`). В таблицу
  исполнителей его не записать (FK на participants), поэтому форма раскладывает
  набор: сентинел → флаг, остальные → `assignee_ids`. Пикер выносит его в
  отдельную группу «Автоматически». **Лид-боты флаг игнорируют** — у входящего
  диалога создателя нет.

## 🔴 Конфликт двух полей об исполнителях (потеря данных, исправлено)

Введённый было режим «дополнить» (`assignees_mode='extend'`) породил **два пути
записи в одну строку**: блок бота писал `assignees_mode`, диалог шаблона —
`override_assignees` (+ DELETE записей), а чтение диалога режим `extend` не
видело. Итог воспроизводился в 3 клика: задал дополнительных исполнителей →
открыл «Настроить шаблон» → сохранил → **они молча исчезли**.

**Решение владельца:** режим «дополнить» убрать совсем — у канала такое же
переопределение, как у шаблона проекта («заменить целиком»), и правится оно
только в диалоге шаблона. Довод, который это подтвердил: «заменить на X, потом
дополнить Y» ≡ «заменить на X∪Y» — второе поле не добавляет выразительности.

- источник правды об исполнителях привязки — **`assignees_mode`**
  (`inherit`/`override`); `resolve_thread_template_binding` читает только его;
- **мост совместимости** (`20260715170000`): триггер `pttt_sync_assignees_mode`
  держит булев `override_assignees` = `(assignees_mode='override')` в обе стороны —
  прод-фронт, пишущий старое поле, работает до выката. Долг: дропнуть колонку и
  мост после деплоя;
- существующие `extend`-привязки **материализованы** (исполнители шаблона досыпаны
  в набор → режим `override`) → состав исполнителей у настроенных ботов не
  изменился (сверено на живой привязке). Миграция `20260715190000`.

## Найдено проверкой в браузере (пре-существующее)

- **Гонка инициализации формы:** диалог открывался ДО загрузки переопределений
  канала → форма инициализировалась базовым шаблоном (ключ монтирования —
  `template.id`, одинаковый у базового и переопределённого) и показывала «Из
  общего» вместо «Индивидуально». Страдали и срок, и доступ, и сообщение. Фикс:
  открывать только с готовыми данными.
- **«Создатель задачи» у канала** молча подмешивался к переопределённым
  исполнителям (в UI «3» вместо 2), хотя приём его игнорирует, а снять из
  канального диалога нельзя (флаг живёт в базовом шаблоне) → скрыт
  (`showCreatorAssignee={false}`).
- **Сирота-привязка:** «Настроить» создавала строку при ОТКРЫТИИ диалога → уход
  без «Сохранить» оставлял её при пустом `config.template_id`. Теперь кнопка
  доступна только для сохранённого шаблона, привязка создаётся при сохранении.
- **Конфиг лид-бота стирал поля другого пути:** выбор шаблона затирал
  `responsible_user_ids`/`owner_user_id`/`welcome_message` → снятие шаблона
  оставило бы бота без ответственных. Сборка конфига вынесена в чистую
  `buildLeadBotConfig` с правилом «пустое = не трогаем, а не очистить».
- **Владелец лид-диалога:** при шаблоне без ручного пула `owner_user_id` был null →
  добавлен фолбэк на аккаунт первого исполнителя шаблона.
