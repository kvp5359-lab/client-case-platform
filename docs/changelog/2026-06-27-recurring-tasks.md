# Повторяющиеся задачи

**Дата:** 2026-06-27
**Тип:** feat
**Статус:** completed (БД в проде через MCP, ждёт деплоя фронта)

Задача может автоматически пересоздаваться по расписанию (ежедневно / по дням
недели / по числу месяца), с отдельно настраиваемыми временем (или интервалом)
будущей задачи и опережением создания. Модель — отдельная сущность-правило
(как раздел «Recurring» в Планфиксе), но расписание считается в БД, а генерация
идёт через pg_cron. План и обоснование архитектуры —
[`docs/feature-backlog/2026-06-27-recurring-tasks.md`](../feature-backlog/2026-06-27-recurring-tasks.md).

## Модель и движок (БД)

- Таблица `recurring_task_rules` — правило повторения: снимок параметров задачи
  (название с подстановкой `{date}`/`{project_name}`, статус, иконка/цвет, доступ,
  исполнители, первое сообщение) + расписание (структурное подмножество RRULE:
  `freq`/`byweekday`/`bymonthday`/`fire_time`/`end_time`) + тайминги
  (`create_lead_minutes`) + пределы (`starts_on`/`until_date`) + состояние
  (`next_occurrence_at`/`occurrences_count`/`is_active`). RLS по образцу
  `item_lists` (общие/личные правила).
- Колонка `project_threads.recurring_rule_id` — пометка сгенерированной задачи +
  ссылка на правило.
- `recurring_next_occurrence(...)` — расчёт следующей даты в plpgsql (кламп
  «31-е → последний день», корректный переход зима/лето).
- Триггер авто-вычисления `next_occurrence_at` при создании/правке расписания.
- `generate_recurring_tasks()` (SECURITY DEFINER) — создаёт задачу по правилу,
  ставит дедлайн (или `start_at`/`end_at` для интервала → попадает в календарь),
  назначает исполнителей/доступ, продвигает дату; пропущенные прогоны
  фаст-форвардит (создаёт одну, прыгает в будущее).
- pg_cron `generate-recurring-tasks` — каждые 10 минут (паттерн как
  `dispatch_scheduled_messages` / `inbox-reconcile`, без edge-функции и секретов).
- Дата повтора = дедлайн задачи; время/диапазон выбирается явно; «срок через N
  дней» убран в пользу конкретного времени. Миграции:
  `20260627_recurring_tasks.sql`, `…_offsets_minutes.sql`, `…_due_time_range.sql`.

## Фронт

- Раздел **Настройки → Шаблоны → «Повторяющиеся»** (`TemplatesTab`, группа «Треды»,
  роут `settings/templates/recurring`) — таблица правил с расписанием, следующей
  датой, счётчиком, переключателем вкл/выкл, правкой и удалением
  (`RecurringTasksContent`).
- Кнопка **«Сделать повторяющейся»** в меню задачи (`TaskActionsMenu` +
  `TaskPanelTaskHeader`) — префилл параметров из задачи.
- Окно `RecurringRuleDialog` — две вкладки: «Параметры задачи» и «Настройка
  повторения» (проект, периодичность, дни/число, время + длительность, создавать
  заранее, пределы, превью ближайших дат).
- **Общий вынос полей `ThreadTemplateFields`** — единый блок параметров задачи,
  используется и редактором шаблона треда (`ThreadTemplateDialog`), и окном
  повторения. Добавил/переставил поле здесь — меняется в обоих местах. Поля,
  специфичные для шаблона (имя шаблона, дедлайн-N-дней, автопереход статуса, email,
  тип треда), гейтятся флагами.
- Пикер времени `TimeOfDayPopover` — тот же 15-минутный грид, что в сроке задачи.
- Пикер проекта — переиспользован `ChatSettingsProjectSelector` (поиск, иконки,
  «Без проекта», создание нового проекта из шаблона).
- Хук `useRecurringRules` (CRUD), ключи `recurringKeys`, типы `types/recurring.ts`,
  чистый модуль `lib/recurring/schedule.ts` (превью дат + описание, покрыт тестами),
  `types/database.ts` дополнен вручную (таблица + колонка).

## Заметки

- Повторяются только задачи (тип фиксирован). Чат — нишево, recurring-email
  требует авто-отправки (карантинный слой доставки) — отложено.
- Часовой пояс фиксирован `Europe/Madrid` (как у дневника). Полный RRULE + сложные
  интервалы + per-rule TZ — Фаза 2 (см. feature-backlog).
- Живой прогон генератора проверен на проде с очисткой тестовых данных
  (подстановка даты, дедлайн/интервал, исполнитель, привязка к правилу, сдвиг даты).

## Заодно (hover-prefetch, из прошлой сессии)

Прогрев по наведению на тред расширен: помимо сообщений греет `last_read_at`
(красный контур непрочитанного) и «мою личность» (`currentParticipant` → сторона и
цвет баблов) — без этого при первом открытии треда контур и раскраска «своих»
появлялись с заметной паузой. `usePrefetchThreadMessages` принимает `projectId`;
вызовы из `DraggableBoardTaskRow`, `InboxChatItem`, `ItemListsPage/ThreadRow`.
