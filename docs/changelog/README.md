# Changelog

История «что делали по дням». **Не источник правды по инфре** — для этого `.claude/rules/`. Сюда смотрят, когда нужен контекст: «когда мы это сделали и почему».

## Формат имени файла

```
YYYY-MM-DD-краткое-описание.md
```

Дата = дата работы. Один файл = один день или одна крупная фича. Если за день несколько разных блоков — можно объединить в один файл с разделами «1. ...», «2. ...».

## Шаблон содержимого

```markdown
# Короткое название

**Дата:** YYYY-MM-DD
**Тип:** UX | bugfix | refactor | feature | infra | docs (комбинации через +)
**Статус:** completed | in_progress | reverted

---

## Контекст

Что произошло, почему понадобилось сделать. 1-3 абзаца простым языком.

## Что сделано

Список изменений. По возможности — со ссылками на конкретные файлы:
- [`src/path/file.ts`](../../src/path/file.ts) — что изменилось.

## Миграции / Edge Functions

Если есть — указать, что задеплоено и куда.

## Известные ограничения

Если осталось что-то незакрытое.
```

## Правила

- Пишет AI или автор после завершения работы. Один день — один (обычно) файл.
- Не дублировать сюда контракты из `.claude/rules/` — только описание изменений и ссылки.
- Если фича стала постоянной частью инфры — добавить раздел в соответствующий `.claude/rules/*.md`, в changelog оставить ссылку.
- Закрытые баги — в `docs/bugs/resolved/`, не дублировать в changelog (но можно сослаться).
