# Единый модуль базы знаний: статьи + Q&A (права, деревья, поиск, история)

**Дата:** 2026-07-11
**Тип:** feat (БД + фронт + edge)
**Статус:** БД-миграции в проде (через MCP + файлы в репо); фронт — деплой push в main → CI/CD blue/green; edge бота (`knowledgeRag`) — **ручной деплой (ещё не выполнен)**

---

База знаний сведена в **единый модуль для статей и Q&A**: одинаковые права
доступа, одинаковые виды (Дерево/Таблица), общий движок дерева. Ключевой принцип —
**переиспользовать механизмы, а не дублировать**, при этом **данные групп у Q&A
свои** (отдельная иерархия). Плюс доработан поиск и добавлена история открытых
статей. Разрабатывалось маленькими шагами; здесь — сводка.

## 1. Унификация прав доступа (4 состояния)

Раньше «Везде» у статьи было семантически неверным: резолвер видимости не знал
такого состояния — статья без прямых привязок попадала в проект только через
группу, а «Везде» в UI фактически означало «наследовать от группы» либо «нигде».

Введена явная колонка `template_access_mode` (у статей, групп и Q&A) с 4 состояниями:

- `inherit` — статья наследует от групп; группа — от родителя (корневая без
  родителя = нигде);
- `everywhere` — во всех типах проектов;
- `selected` — только в выбранных, **замещая** наследование от групп;
- `nowhere` — нигде, даже если родительская группа привязана.

Иконка в дереве теперь **отражает состояние** (Globe / стрелка-наследование /
список+счётчик / EyeOff), а не одинаковый «шаблон». Каскад по **вложенным
группам** теперь работает (раньше привязка родительской группы не подтягивала
статьи из подгрупп).

Резолвер вынесен в единую БД-функцию `resolve_template_article_ids` (рекурсивный
обход дерева групп) — её зовут **и** вкладка БЗ в проекте, **и** бот/RAG, так что
логика гарантированно одинакова.

Q&A **переиспользуют тот же попап/кнопку доступа** (добавлен тип `qa`) —
через новую колонку `template_access_mode` на `knowledge_qa` и junction
`knowledge_qa_templates` (зеркало `knowledge_article_templates`, RLS 1:1).

## 2. Единое дерево статьи + Q&A (общий DnD-движок)

Извлечён обобщённый adapter-driven стек дерева — `src/components/knowledge/tree/`
(`GroupTreeBody`, `GroupTreeNode`, `useGroupTreeDnd`, `types.ts`). Дерево статей
**переведено** на него (рефактор живого дерева), а Q&A получили переключатель
**Дерево/Таблица** с полным DnD-паритетом (перенос между группами + порядок
внутри) через RPC `move_qa_to_group` (зеркало `move_article_to_group`).

Переиспользованы: движок DnD, узлы дерева, `EditGroupDialog` (отвязан от `page`),
CRUD групп (`useKnowledgeGroups`), кнопка доступа. Различие статьи/Q&A — только
реализация `TreeSource` (данные, мутации, рендер строки).

## 3. Раздельные группы Q&A (`knowledge_groups.kind`)

Группы у Q&A — **свои**, отдельные от статей. Одна таблица `knowledge_groups`, но
колонка `kind` (`'article' | 'qa'`) разводит иерархии: дерево статей грузит
`kind='article'` (существующие 14 групп), Q&A и его фильтры/редактор —
`kind='qa'` (свои группы, отдельный кэш-ключ). Механизмы (CRUD/RLS/попап) общие,
данные раздельны.

## 4. Поиск: сохранение значения + история

- **Не сбрасывается при открытии/закрытии статьи.** Раньше открытие уходило на
  другой роут, список размонтировался и `useState('')` терялся. Теперь значение
  хранится в `sessionStorage` (per workspace + область) и восстанавливается.
- **История поиска** — выпадающий список (localStorage, per workspace + область),
  показывается при фокусе/вводе, с навигацией стрелками, удалением пункта и
  «Очистить всё». Общий компонент `KnowledgeSearchInput` + хуки
  `usePersistentSearch`/`useSearchHistory` применены во всех трёх полях КБ (статьи
  Дерево/Таблица — общий стейт; Q&A — свой).

## 5. История открытых статей (per-user)

Кнопка-выпадашка в тулбаре статей: список открытых статей, **сгруппированный по
дате** (Сегодня/Вчера/дата) с **временем** открытия. Переиспользует серверную
`recently_viewed` (per-user, дедуп — последнее открытие каждой). Добавлен трек
открытий из бокового окна (раньше писались только открытия через страницу-редактор).

## 7. Прочее в этом пуше (батч ранее незапушенных коммитов)

Вместе с базой знаний пушится накопленный тех-долг из прошлых сессий (детали —
в `docs/audit/` и messenger-ledger):

- **channels:** visibility-backstop на `wazzup-send`/`telegram-mtproto-send`/
  `email-internal-send` — внутреннее сообщение с файлом не утечёт в канал (второй
  слой поверх фронт-гейта).
- **palette:** `ThreadAccentColor`/`MessengerAccent`/`ACCENT_BORDER` привязаны к
  `accentPalette.AccentSlug` — новый акцент палитры ломает сборку, а не молча
  падает в дефолт.
- **layers:** осиротевшие общие типы/константы вынесены из `components/` вниз.
- **ci/ops:** блокирующий drift-gate на PR, guard DB-инвариантов, хук имён
  миграций, ручной backend-deploy workflow + deploy-backlog.
- **db:** revoke `anon` на `record_telegram_bot_msg_id` (write-RPC мессенджера —
  service-role-only); паритет тел прод-функций repo↔prod.
- **style:** компактный редактор быстрых ответов.

## 8. Что осталось

- **Edge бота** (`knowledgeRag.ts` → `knowledge-search` / `chat-with-messages` /
  `knowledge-index`) — **ручной деплой** (CI edge не катит). До него бот
  использует старую резолюцию (union, без `nowhere`/замещения/каскада).
- **Браузер-проверка** дерева статей после рефактора (DnD между/внутри групп) +
  Q&A-дерева со своими группами.

Проверки на коммит: `tsc` + `eslint` + `vitest` (231 тест) зелёные.
