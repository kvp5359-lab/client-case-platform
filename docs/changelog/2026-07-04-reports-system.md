# 2026-07-04 — Система отчётов (MVP конструктора)

## Что сделано

Гибкий конструктор отчётов на уровне воркспейса: датасет → фильтр →
группировки → показатели. Первый закрытый кейс — «кто сколько оплатил /
кто сколько должен за период».

### БД (в проде через MCP + миграции в репо)

- `report_definitions` — сохранённые отчёты (общие/личные, зеркало модели
  `item_lists`), RLS 1:1 с item_lists. Миграция `20260704130000_report_definitions.sql`.
- RPC `run_report(workspace_id, config)` + хелперы `_report_filter_sql` /
  `_report_condition_sql`. SECURITY INVOKER (RLS применяется к вызывающему),
  сборка SQL строго из whitelist-реестра, значения через `quote_literal`,
  гранты только authenticated/service_role. Миграция
  `20260704130100_run_report_engine.sql`.
- Датасеты v1: платежи (`transactions`), услуги (`services`), баланс
  клиентов (`client_balance` — выставлено − оплачено = долг), проекты,
  треды. Протестировано на проде под ролью authenticated (summary с
  2-уровневой группировкой + totals, detail с фильтрами, фильтр по балансу).

### Фронт

- `src/types/reports.ts`, `src/lib/reports/registry.ts` (клиентское зеркало
  серверного реестра: лейблы, типы, форматы, дефолтные конфиги),
  `src/lib/reports/runtime.ts` (период → даты, секционирование с
  подытогами, CSV) + 17 тестов.
- `reportService.ts`, `useReports.ts`, `reportKeys`; загрузка вариантов
  фильтров по optionsKind (участники/проекты/статьи/статусы/шаблоны/услуги).
- Страницы `/workspaces/[id]/reports` (список: общие/мои, создание,
  удаление) и `/reports/[reportId]` (быстрый период, таблица с
  группировками/подытогами/итого, CSV, настройки: вид, группировки до 3
  уровней с гранулярностью дат, показатели, колонки, фильтры).
- Пункт сайдбара `nav:reports` («Отчёты», BarChart3).

## Решения

- **Whitelist-реестр вместо динамического SQL от клиента** — клиент шлёт
  только ключи; сервер резолвит их в SQL-снippets из константы в функции.
- **SECURITY INVOKER** — отчёты видят ровно то, что видит пользователь по
  RLS (финансы, треды, проекты).
- **Период не хранится** — пресет резолвится в даты на каждый запуск,
  «последние 30 дней» в сохранённом отчёте скользящие.
- **Два реестра** (сервер+клиент) — осознанная цена: SQL и TS не могут
  делить константу. При добавлении поля/датасета править оба.

## Ограничения v1 (осознанные)

- Валюта EUR (как во всём финансовом модуле).
- «Долг» — арифметика (сумма услуг − доходы), без привязки платежа к услуге.
- Кастомные поля проектов ещё не в датасетах (план: join
  `project_field_values` через реестр).
- Вывод — таблица + CSV; графики/расписание рассылки — следующие фазы.
