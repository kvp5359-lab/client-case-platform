**Дата:** 2026-07-17
**Тип:** fix + feat (фронт: layout, доски, мессенджер)
**Статус:** деплой (push в main → CI/CD blue/green)

---

Три независимые фронтовые задачи одной сессии. Диагностика — замерами (геометрия в
реальном Chrome через DevTools, фактические данные из прода по SQL), не по догадкам.

## 1. Белая полоса между контентом и боковой панелью на широких экранах (fix)

**Симптом.** На окне шире ~1800px (в Chrome; во встроенном браузере уже — не
видно) слева от правой панели зияла белая полоса ~35px.

**Причина (замер геометрии).** Всё приложение обёрнуто в `max-w-[1800px] mx-auto`
((app)/layout.tsx) — на широком окне оно центрируется, справа остаётся авто-поле
`(100vw − 1800)/2`. Боковая панель — `position: fixed; right: 0`, т.е. привязана к
краю ОКНА, а контент (`main`) — внутри обёртки 1800px. Разница = ширина этого поля
= белая полоса. Не про скроллбары и не про прод — чистая ширина окна.

**Решение.** CSS-переменная `--app-right-inset` (правое поле обёртки), к ней
привязаны правый край панели, resize-handle и тостов → они прижимаются к краю
ПРИЛОЖЕНИЯ, а не окна. На окне ≤1800px инсет = 0, поведение прежнее.

**Доработки по ревью:**
- `1800px` вынесен в единую переменную `--app-frame-max` — используется и в
  обёртке (`max-w-[var(--app-frame-max)]`), и в расчёте инсета (дубля литерала
  больше нет).
- Точное значение инсета выставляет `WorkspaceLayout` JS-замером реальной геометрии
  обёртки (`clientWidth − frame.right`) — так корректно учитывается ширина
  классического скроллбара; CSS-`max(...)` остаётся фолбэком до гидрации.
- `FloatingPanelButtons` проверена — она `absolute` внутри обёртки, не `fixed` к
  окну, инсет не нужен. Других fixed-к-окну элементов нет.

Проверено в реальном Chrome: панель прижата к краю обёртки, зазор `main`↔панель = 0.

## 2. Сортировка списка доски «по сроку» (fix)

**Симптом.** «Сортировать по сроку» / колоночный сорт «Дедлайн» ставил задачи без
срока НАВЕРХ, а также задачи «на весь день» (дата без времени) — выше задач с
конкретным временем того же дня.

**Причина (замер по данным прода).** Два дефекта:
1. Задачи без срока получали `Infinity`, и при `sort_dir='desc'` (это ДЕФОЛТ)
   уезжали наверх.
2. Дата без времени хранится как полночь UTC → сортировалась как 00:00, т.е. раньше
   «сегодня 10:30».

**Решение.** Новый общий `deadlineSortValue` (utils/deadlineUtils):
- нет срока → в конец ВСЕГДА, независимо от направления (как у проектов);
- «весь день» (полночь UTC) → ключ = конец дня (+почти сутки), поэтому задачи с
  конкретным временем того же дня идут выше, а «на весь день» — под ними;
  просроченные остаются наверху по своей дате.

Ключ только для сравнения — сам дедлайн не меняется. Подключён в оба пути: пункт
меню «Сортировать по сроку» и колоночный сорт. +8 тестов (`compareTasks` по сроку
и `deadlineSortValue`).

## 3. Номера и буллеты списков в цитате (feat)

**Симптом.** При «Цитировать» (меню сообщения и попап по выделению) нумерованный
список приходил без «1., 2., 3.», хотя клиент их видит.

**Причина.** Номера `<ol>` и буллеты `<ul>` в HTML — это CSS-маркеры, а не текст;
обычный strip их теряет.

**Решение.** `htmlToQuoteText` / `quoteTextFromRange` (utils/format/messengerHtml)
восстанавливают маркеры как текст: `<ol>` — номера с учётом `start`, каждый список
нумеруется независимо; `<ul>` — буллет. Для выделения номера считаются по ЖИВОМУ
DOM бабла (верны даже при частичном выделении). Копирование оставлено точным
текстом выделения. +6 тестов.
