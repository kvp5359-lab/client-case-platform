# Служебные события задачи считаются непрочитанными только исполнителю (+бейдж при назначении)

**Дата:** 2026-07-08
**Тип:** feat (БД + фронт)
**Статус:** БД в проде (через MCP + миграция в репо); фронт — деплой (push в main → CI/CD blue/green)

---

Служебные события задачи (создание, смена статуса, переименование, смена настроек
и т.п.) теперь попадают в **счётчик непрочитанного только исполнителю** задачи
(`task_assignees`), а не любому участнику или наблюдателю. Если у треда нет
исполнителей вовсе — событие считается всем участникам (как раньше). Плюс: при
назначении исполнителем задача помечается непрочитанной у нового исполнителя —
явный сигнал «тебя назначили».

Тостов (всплывающих уведомлений) на события задач **нет и не добавляли** — тосты
существуют только для новых сообщений и новых внешних диалогов. Правка касается
только счётчика непрочитанного (бейдж + красный контур события в ленте).

## 1. Что было не так

Событие треда считалось непрочитанным для **любого** подписанного участника. На
практике event-бейджи копились именно у не-исполнителей — участников, владельцев,
view_all-админов, которым чужая смена статуса/срока не адресована. Замер на проде
перед правкой: из 79 пар с текущим `unread_event_count>0` — **0 у исполнителей**,
66 у не-исполнителей, 13 на тредах без исполнителей. То есть весь event-шум шёл
не тем людям.

Отдельная находка: audit-событие при назначении исполнителя почти **не пишется**
(8 из 1669; `change_assignees` в проде = 0). Поэтому «бейдж при назначении» нельзя
построить на «непрочитанном событии назначения» — его в БД нет.

## 2. База (в проде)

Миграция [`20260708120000_thread_events_unread_only_for_assignees.sql`](../../supabase/migrations/20260708120000_thread_events_unread_only_for_assignees.sql).

- **`recompute_thread_unread_for`** — в блок подсчёта событий `v_events` добавлено
  условие: `EXISTS task_assignees[thread, me] OR NOT EXISTS task_assignees[thread]`
  (я исполнитель ИЛИ у треда нет исполнителей). Строки про `change_deadline`
  (не считается) и `silent_transition`-статусы (не считаются) сохранены.
  Сообщения (`v_unread`), реакции (`v_reactions`) и приоритетные упоминания/ответы
  (`v_priority`) — **не тронуты**. Тело снято с прода (drift repo↔prod).
- **`seed_read_status_on_assignee`** — при добавлении в исполнители теперь ставит
  `manually_unread=true` новому исполнителю, если назначил **не он сам**
  (`participants.user_id IS DISTINCT FROM auth.uid()`). Точку прочтения
  (`last_read_at`) НЕ сдвигаем → старая история не всплывает. Если строка уже
  существовала (человек читал тред раньше) — только `manually_unread := existing OR mark`,
  прочтение не перетираем. Это и есть «бейдж при назначении» — булева пометка
  непрочитанного, а не выдуманное событие. Работает на будущие назначения.

## 3. Фронт (ждёт деплоя)

Красный контур события прямо в ленте чата считается отдельной клиентской формулой
(в `MessageList`), которая раньше не знала про исполнительство → бейдж и лента
разошлись бы. Синхронизировано:

- `MessageList` получил проп `viewerGetsEvents` (по умолчанию `true` — где флаг не
  передан, поведение прежнее). Добавлен в **обе** формулы непрочитанности события
  (`isEventUnread` в useMemo группировки + `eventIsUnread` при рендере одиночного
  события) и в deps useMemo.
- `MessengerTabContent` считает
  `viewerGetsEvents = assigneeIds.length === 0 || assigneeIds.includes(myPid)`, где
  `assigneeIds` = `participant_id[]` из `useTaskAssigneeIds`, `myPid` =
  `state.currentParticipant?.participantId`.

## Грабли (на будущее)

- «Непрочитанность события» гейтится исполнительством в ДВУХ местах — БД
  (`recompute_thread_unread_for` → `v_events`) и клиент (`MessageList`, две формулы).
  Держать синхронно.
- Бейдж «тебя назначили» = `manually_unread` (булева пометка), НЕ событие. Работает
  на будущие назначения (триггер); старым назначениям задним числом не проставляется.
- `viewerGetsEvents` считать по **participant-id** зрителя против `assigneeIds`
  (тоже participant_id). Событие в audit сравнивается по `user_id`, но исполнительство —
  по `participant_id`; две системы id не смешивать.
- Чаты/письма без исполнителей: их события считаются всем участникам (правило «нет
  исполнителей → всем»).

## Backfill

Массовый пересчёт задним числом **не делали** (по решению владельца — правка «на
будущее»). 66 текущих пар с непрочитанными событиями у не-исполнителей обнулятся
сами при следующей активности треда (триггеры пересчёта). При необходимости стереть
сразу — точечный `refresh_thread_unread_pairs` по затронутым тредам.

## Проверки

- tsc 0, eslint 0 по `MessageList.tsx` и `MessengerTabContent.tsx`.
- БД применена в прод; замер до/после сходится (0 непрочитанных событий у
  не-исполнителей будет после пересчёта пары).
- Смок за пользователем: сменить статус/переименовать задачу от другого сотрудника →
  бейдж только у исполнителя, у участника-не-исполнителя нет; красный контур события
  в ленте совпадает с бейджем; назначить кого-то исполнителем → у него задача
  помечена непрочитанной (сам себя назначил → без пометки).
