# Склейка email-цепочек в своём Gmail (исходящие через employee_mailbox)

**Дата:** 2026-06-29
**Тип:** fix (edge) + docs
**Статус:** completed — edge задеплоен, ждёт смок-теста

---

## Симптом

Ответ из сервиса на входящее письмо у отправителя в Gmail ложился **отдельной
веткой**, не склеивался с входящим. У получателя и в самом сервисе всё ок.

## Корень

Почта в проекте ходит двумя транспортами (Gmail API / Resend). Боевая связка
`kvp5359@gmail.com`: входящие приходят **пересылкой Gmail→Resend**
(`source='email_internal'`), исходящие уходят через **подключённый Gmail API**
(`employee_mailbox`).

Склейка ответа в СВОЁМ Gmail (при отправке через Gmail API) требует Gmail-`threadId`
той копии письма, что лежит в ящике. При схеме «пересылка→Resend» родной
`gmail-webhook` входящее не видел → `gmail_thread_id` не сохранён (или указывал на
ветку нашего же прошлого ответа) → ответ уходил без `threadId` → отдельная ветка.
`In-Reply-To`/`References` для получателя при этом работают.

## Фикс

`email-internal-send` → `sendViaEmployeeMailbox`: перед отправкой резолвим
настоящий Gmail-тред письма, на которое отвечаем, по его Message-ID —
`findGmailThreadByMessageId(token, inReplyTo)` → `GET users/me/messages?q=rfc822msgid:<id>`
→ `threadId`. Используем его (фолбэк — сохранённый `gmail_thread_id`, затем без
`threadId`). Самолечится: после успешной отправки сохраняется верный
`gmail_thread_id`.

- Один дополнительный GET к Gmail на исходящее.
- Безопасный фолбэк: при любой ошибке/индекс-лаге возвращаем `null` → отправка как
  раньше.
- Ограничение: работает, только если Message-ID входящего реально есть в ящике
  отправителя (для пересылки — да). Чистый Resend на доменный адрес в Gmail не
  лежит — там склейки в Gmail и не должно быть.

> Радикально «микс транспортов» лечится только единым транспортом (Resend-only) —
> это отдельное продуктовое решение.
