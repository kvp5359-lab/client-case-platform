# Удалённый тред: подсказка при открытии + мгновенное удаление на всех вкладках инбокса

**Дата:** 2026-07-07
**Тип:** fix (фронт)
**Статус:** ждёт деплоя фронта

---

Разбор жалобы «не могу открыть тред» (email от GitHub Actions во «Входящих»,
perfTrace показал «застряло после open, 4002ms»). Тред оказался удалён пакетным
действием, а строка залипла в кэше вкладки. Три связанных фикса, все — чистый фронт.

## Диагностика

- Тред `6243c4d0…` — `is_deleted = true`. Inbox-RPC его не отдаёт → строка во
  «Входящих» = **застрявший кэш вкладки**, открытой с момента до удаления.
- Удалён пакетно: в 08:12:06 одним UPDATE снесло 12 email-уведомлений
  (`notifications@github.com` × 11 + GitGuardian) — у всех одинаковый `deleted_at`
  и `deleted_by = null`. Источник — `BulkActionsBar.archive()` («В корзину» в
  Списках), который не проставлял `deleted_by`.

## 1. Зависание при открытии удалённого/недоступного треда

**Симптом:** открыл ссылку/строку удалённого треда → панель вечно в спиннере
(perfTrace «застряло после open»).

**Корень:** `useProjectThreadById` фильтрует `is_deleted = false` (+ RLS) → для
удалённого возвращает `null`, но панель этого не обрабатывала и крутила загрузку.

**Фикс** ([`useTaskPanelInternal.ts`](../../src/components/tasks/useTaskPanelInternal.ts)):
когда запрос треда отработал (`isFetched && !isLoading`) и вернул `null` при
открытом треде — показываем тост «Тред удалён или недоступен», убираем застрявшую
строку из инбокс-кэшей и закрываем панель. Это же **чистит залипшую строку из
спящей вкладки по клику** (когда realtime-удаление до вкладки не долетело).

## 2. Пакетное удаление не писало `deleted_by`

**Фикс** ([`BulkActionsBar.tsx`](../../src/page-components/ItemListsPage/BulkActionsBar.tsx)):
`archive()` теперь пишет `deleted_by: user.id` (через `useAuth`) → в Корзине виден
автор удаления.

## 3. Задержка ~2с при удалении из инбокса (вкладки-сегменты)

**Симптом:** удаление треда из задача-колонки — мгновенно, из «Входящих» —
с задержкой в пару секунд.

**Корень:** оба удаления идут через один `useDeleteThread`. Его оптимистичный
съём строки (`removeThreadFromInboxCaches`) патчил только базовый кэш
`inboxKeys.threads` (вкладка «Все») + агрегаты. Сегментные вкладки
(«Непрочитанные / Нужно ответить / Ждём клиента / Заглушённые») читают **отдельные**
плоские кэши (`inboxKeys.unread` и др.), которые съём не трогал → строка держалась
до следующего рефетча (realtime-дебаунс ~1.5с + запрос). На вкладке «Все» удаление
было мгновенным, на «Непрочитанные» — с задержкой. Задача-колонка обновлялась
быстрым рефетчем и выглядела мгновенной.

**Фикс** ([`threadCacheSync.ts`](../../src/hooks/shared/threadCacheSync.ts)):
`removeThreadFromInboxCaches` теперь чистит и сегментные кэши (`unread`,
`needsReply`, `awaitingReply`, `muted`) с корректным снимком для отката при ошибке.
→ удаление мгновенно на любой вкладке инбокса.

## Грабли (на будущее)

- Любой новый inbox-кэш-подмножество (как сегментные вкладки) должен попадать не
  только в realtime-инвалидацию, но и в оптимистичный съём `removeThreadFromInboxCaches`
  — иначе строка залипает до рефетча.
- Открытие треда, чей ряд не найден (удалён/нет доступа), должно завершаться
  подсказкой + закрытием, а не вечным спиннером.
- Пакетные soft-delete (`is_deleted`) должны писать `deleted_by`, как одиночный
  `useDeleteThread`.

## Не сделано (отдельно)

Авто-исчезновение залипшей строки в **спящей вкладке без клика** — нужна сверка при
реконнекте realtime (пропущенные события не переигрываются). Отдельная тема.
