# Входящие: счётчики вкладок «Нужно ответить»/«Ждём клиента» из агрегатов + ленивая загрузка списков

**Дата:** 2026-06-15
**Тип:** perf
**Статус:** completed (БД в проде; фронт ждёт деплоя+смок)

Перф-оптимизация доски «Входящие». Две вкладки «Нужно ответить» и «Ждём клиента»
грузили тяжёлые списки **всегда** при открытии доски (даже когда не активны) и
рефетчились при каждом входящем сообщении. Теперь полные списки грузятся лениво —
только при открытой вкладке, а цифры в бейджах считаются из уже-грузящегося
лёгкого RPC агрегатов.

---

## Замеры (read-only прод, тёплый кэш, мин/сред/макс мс, 1097 тредов)

| RPC | мин | сред | макс | строк |
|-----|-----|------|------|-------|
| `get_inbox_threads_v2` (база) | 123 | 299 | 578 | 1097 |
| `get_inbox_threads_page` (вкладка «Все», 1 стр) | 143 | 152 | 160 | 50 |
| `get_inbox_unread_threads` (Непрочитанные) | 128 | 182 | 275 | 11 |
| **`get_inbox_needs_reply_threads` (Нужно ответить)** | 168 | 415 | 894 | 321 |
| **`get_inbox_awaiting_reply_threads` (Ждём клиента)** | 138 | 242 | 425 | 99 |
| `get_inbox_thread_aggregates` (бейджи сайдбара) | 49 | 82 | 140 | 1097 |

**Корень перегруза:** обе обёртки (а) грузились всегда (`enabled` без гейта по
активной вкладке); (б) каждая = повторный скан всего `get_inbox_threads_v2` (тяжёлый
JOIN-граф) + 2 коррелированных подзапроса на ряд (направление последнего сообщения
+ EXISTS внешнего канала); (в) сидели в `useWorkspaceMessagesRealtime.doInvalidate`
и `invalidateMessengerCaches` → рефетч при каждом входящем (throttle 400мс) даже на
неактивной вкладке.

**Эффект:** из дефолтного пути доски убраны 2 самых дорогих запроса
(~0.65 сек среднем). Открытие доски легче на ~55–60% по серверному времени
запросов (до ~70% в пик). Добавлено ~34 мс к одному уже-грузящемуся агрегату.

## БД

Миграция `20260615_inbox_aggregates_segment_fields.sql` (применена в прод через MCP):

- `get_inbox_thread_aggregates` расширена 2 полями:
  - **`last_from_staff`** — последнее НЕ-сервисное сообщение отправил сотрудник?
    (`is_staff_role(sender_role)`; `NULL` = собеседник). Зеркало предиката обёрток.
  - **`has_external`** — в треде есть сообщение из внешнего канала
    (TG / TG Business / MTProto / Wazzup / Email).
- 2 новых CTE (`last_msg_meta` DISTINCT ON последней роли, `external_threads`
  DISTINCT) — измеренная стоимость ~9 + ~20 мс; агрегат вырос с ~50 до ~84 мс.

**Грабли:**
- Смена `RETURNS TABLE` требует `DROP FUNCTION` + `CREATE` (`CREATE OR REPLACE`
  не умеет менять тип возврата). DROP сбрасывает ACL → восстановлены гранты.
- Supabase default privileges грантят новые public-функции на `anon`, и
  `REVOKE … FROM PUBLIC` это **не** снимает → отдельный `REVOKE … FROM anon`.
  Итог грантов: `authenticated`, `service_role` (как было до правки).
- Drift repo↔prod: тело = живое тело прода + 2 CTE + 2 колонки, 1:1 остальное.

**Проверка корректности (read-only прод):** счётчики из агрегатов точно совпали с
тяжёлыми обёртками — `needs=322`, `awaiting=97`, 0 расхождений множеств. Прежний
дрейф (97 vs 99) оказался разницей живых данных между запросами, не логики.

## Фронт

- **`src/utils/inboxUnread.ts`** — чистые предикаты-классификаторы:
  `isThreadFullyRead`, `isNeedsReply`, `isAwaitingReply`, `countInboxSegments`.
  Точное зеркало WHERE обёрток needs/awaiting (включая read-гейт). +тесты.
- **`useFilteredInbox.ts`** — у `useFilteredInboxNeedsReply` / `…AwaitingReply`
  добавлен параметр `enabled` (ленивая загрузка по активной вкладке). Новый хук
  `useInboxSegmentCounts` считает счётчики из `useFilteredInboxAggregates`
  (тот же RPC, TanStack дедуплицирует → без лишнего запроса).
- **`useInboxFilters.ts`** — `filter` стал управляемым родителем (controlled),
  чтобы знать активную вкладку до вызова обёрток; счётчики awaiting/needs из
  `.length` убраны (теперь из агрегатов). +обновлён тест.
- **`InboxPage/index.tsx`**, **`BoardInboxList.tsx`** — гейт обёрток по активной
  вкладке + счётчики бейджей из `useInboxSegmentCounts`.
- **`inboxService.ts`**, **`types/database.ts`** — +2 поля в тип агрегата.

## Грабли на будущее

- Любой потребитель «непрочитанного/сегмента по треду» читает **агрегаты**
  (`inboxKeys.aggregates`, полные), а не пагинированный `inboxKeys.threads`.
- `isNeedsReply` / `isAwaitingReply` — зеркало WHERE обёрток needs/awaiting и
  read-гейта. При изменении набора source/ролей в обёртках — синхронить
  `inboxUnread.ts` И поля агрегата `get_inbox_thread_aggregates`.
- Счётчик(агрегаты) vs длина списка(обёртка над v2) может отличаться на ±1–2
  из-за дрейфа живых данных между запросами (множества идентичны). Для бейджа ок.
- При `DROP+CREATE` любой public-функции — проверять `anon` в грантах.
