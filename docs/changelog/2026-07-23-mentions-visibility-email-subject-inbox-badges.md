## 1. Упомянули человека, которого нет в треде, — он не узнавал (fix)

Кейс владельца: Аня @упомянула его в чате проекта, куда он ещё не был добавлен.
Упоминания записались, а сигналов не было. Два независимых корня:

- **Тост.** Для неподписанного всплывашка показывается только по признаку
  «в этом сообщении упомянули меня», но упоминания пишутся отдельным запросом
  ПОСЛЕ сообщения — realtime-проверка всегда обгоняла вставку, тост молча
  терялся. Для не-участников гонка систематическая (у них нет пропуска по
  подписке). Теперь при пустом результате проверка повторяется через 1.2 и
  2.5 сек — тост приходит с секундной задержкой.
- **Выдача доступа «съедала» упоминания.** Сиды точки отсчёта
  (`seed_read_status_on_project_access` / `_on_thread_member`) ставили
  baseline = момент добавления → упоминания, сделанные ДО добавления, разом
  становились «прочитанными». Теперь оба сида отматывают `last_read_at` к
  самому раннему упоминанию участника в только что засеянных тредах (только
  свежие строки — реальные точки прочтения не трогаются; «Только я» — никогда,
  «Команде» — только сотрудникам). Проверено rollback-транзакцией на живом
  кейсе. Миграция `20260723160000_seed_read_status_rewind_to_mentions.sql`.

## 2. Бейджи и «Непрочитанные» молчали у владельца (fix)

Продолжение того же кейса: в БД `unread_count=2` (упоминание + ответ пробили
mute), сервер отдавал тред первым в списке — а на экране ни треда в
«Непрочитанных», ни бейджа проекта, ни бейджа вкладки; F5 не помогал.

Резал клиентский `useAccessFilter`: `canAccessThread` вызывался с
захардкоженным `hasViewAllProjects: false` (ещё с до-подписочных времён).
Владелец не записан участником проекта → не проходил ни одну клиентскую ветку
→ строка вырезалась из всех отфильтрованных списков. Теперь фильтр берёт
реальный `canViewAllProjects` из `useWorkspacePermissions`: для владельца он
становится no-op (сервер — источник правды по доступу), для остальных ролей
остаётся страховкой.

🪤 Урок: клиентские дубли серверной access-логики обязаны учитывать все ветки
серверной модели — «упрощённый» захардкоженный параметр = тихая потеря строк.

## 3. Email: ответ со сменой темы = новый тред + дубль без ложной тревоги (fix)

- Письмо «Certificados pendientes» упало внутрь треда «Plan de negocios…»:
  корреспондент нажал «Ответить» и сменил тему — заголовки ответа остались, и
  по стандарту письмо клеилось в старую цепочку, хотя Gmail рвёт беседу при
  смене темы. Теперь применяем правило Gmail: после матча по In-Reply-To /
  References сравниваются нормализованные темы (`normalizeEmailSubject`:
  цикл Re:/Fwd:/Re[2]:, схлопывание пробелов, регистр; +6 тестов) — тема
  непуста и отличается → письмо идёт по subject-матчу либо создаёт новый тред.
- Письмо, доставленное дважды (to: сотрудника + cc: владельца через
  пересылку), штатно упиралось в защиту от дублей, но recovery-ветка не
  отличала дубль от сбоя → писала в «Несопоставленные» и вешала тревожную
  плашку «не загрузилось в чат» под нормально доставленным письмом. Теперь
  дубль распознаётся (23505 по `idx_project_messages_email_message_id`) и
  тихо пропускается.
