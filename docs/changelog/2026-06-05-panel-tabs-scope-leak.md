# Боковая панель: вкладки чужой области (scope-leak) — системный фикс

**Дата:** 2026-06-05
**Тип:** fix
**Статус:** completed

Личные диалоги (контакт-область) показывались во вкладках чужого проекта и
наоборот: вкладки одной области панели протекали в запись другой и
самовоспроизводились. Завершение серии точечных заплаток единым системным
решением.

---

## Диагностика (измерено в рантайме)

Через временные логи переключения scope поймана точная механика на реальных
тредах (контакт «София» ↔ проект «Клиенты»):

1. `hydrate` контакт-области загружал из `task_panel_tabs` уже битый набор —
   с `tasks:<projectId>` и тредами чужого проекта.
2. `persist→flush` тут же записывал этот же набор обратно (`willWrite:true`).

Вывод: первичная утечка произошла в прошлом race при переключении областей, а
дальше мусор стал **бессмертным** — при каждом открытии диалога перечитывался и
записывался обратно. Контакт-область **нигде не валидировалась** (старый
cleanup-эффект работал только для project-области, а `sanitizeTabsForScope`
чинил лишь `tasks`-вкладки в project-области). Поэтому чистка БД без кода не
держалась.

## Решение

Единая валидация принадлежности вкладок области в cleanup-эффекте
[`TaskPanelTabbedShell.tsx`](../../src/components/tasks/TaskPanelTabbedShell.tsx)
— расширен с project-области на contact-область:

- **project-область**: thread-вкладка валидна только если тред в этом проекте;
  `tasks`-вкладка — только своего проекта (`refId === projectId`). (как было)
- **contact-область** (личный диалог): `tasks`-вкладок быть не может (нет
  проекта) — удаляются; thread-вкладка валидна только если тред принадлежит
  этому контакту (через `useContactThreads`).

Невалидные вкладки закрываются (`closeTab` → перезапись чистого набора в БД).
Это **самоисцеление**: битый набor очищается при первом же открытии области и
больше не записывается — лечит и причину проявления, и существующие записи, и
будущие утечки. Системные вкладки (history/documents/…) не трогаются — они не
несут чужих данных, их видимость отруливает `usePanelTabsVisibility`.

Проверено вживую: запись контакта «София» при открытии очистилась с
`[tasks:Клиенты, …, thread:Клиенты×2, thread:София]` до
`[documents, history, project_context, assistant, thread:София]`.

## Контекст (предыдущие заплатки этой области)

- `tasks`-вкладка чужого проекта в баре (project-scope cleanup).
- Дедуп вкладок по id (`dedupeTabsById` в `useTaskPanelTabs`).
- `sanitizeTabsForScope` + race-guard в `persist`.

Текущий фикс обобщает их инвариант «вкладка принадлежит своей области» на
контакт-область, закрывая класс проблемы целиком.
