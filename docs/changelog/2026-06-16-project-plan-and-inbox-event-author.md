## 1. «Сформировать план» — текстовый план выполнения проекта

У вкладки «Задачи» проекта появилось меню «⋮» (как у «Анкет») с пунктом
**«Сформировать план»**. Открывает модалку с текстовым планом.

**Состав плана** (одно текстовое поле, две секции):
- **ЗАДАЧИ** — задачи в порядке плана, разбитые разделителями-заголовками
  (`project_plan_blocks` типа `heading`/`text`). Нумерация перезапускается
  внутри каждой группы; выполненные задачи (финальный статус) — зачёркнуты.
- **ДОКУМЕНТЫ** — слоты документов, сгруппированные по **наборам документов**
  (без вложенных папок: слоты набора нумеруются подряд). Название набора —
  жирным. У заполненного слота через тире выводится статус загруженного
  документа (`1. Копия карты TIE — Принят`).

**Настройки + редактирование:**
- Тумблер **«Выводить исполнителей»** — при включении имена исполнителей задачи
  выводятся в скобках.
- План открывается в **общем Tiptap-редакторе** (`MinimalTiptapEditor`) — текст
  можно править руками; кнопка «Копировать» берёт актуальный текст из редактора.
- ⚠️ Переключение настроек/подгрузка данных **перегенерирует** план (ручные
  правки в этот момент сбрасываются — настройки влияют на генерацию).

**Архитектура:**
- Чистый билдер `src/lib/projectPlanText.ts` (`buildProjectPlanLines` +
  `planLinesToText`/`planLinesToHtml`) — без зависимостей от UI, покрыт тестами.
  `htmlToPlain` инжектится параметром (lib не тянет компонентный слой).
- Модалка `ProjectPage/components/GeneratePlanDialog.tsx` — грузит задачи,
  блоки плана, наборы, слоты, статусы; данные тянутся лениво (только при
  открытии).

**Файлы:** `ProjectPage.tsx`, `GeneratePlanDialog.tsx` (нов), `projectPlanText.ts`
(нов) + тест.

## 2. «Входящие»: автор события в превью + аватар по действию

**Было:** в строке инбокса событие показывалось как «Статус: Выполнена» —
**без автора**; аватар строки всегда брался от собеседника. В задаче внутри
проекта (многоучастниковый тред) это сбивало: «собеседник» — кривой сигнал.

**Решение:**
- **Текст события** теперь начинается с автора: `Анна · Статус: Выполнена`.
  Имя автора (`audit_logs.user_id → participants`) вшито в `last_event_text`
  внутри `get_inbox_threads_v2`. UI подсвечивает имя статуса по позиции
  подстроки «Статус: ».
- **Аватар строки = автор показанного действия** — для многоучастниковых тредов
  (`thread_type='task'` или TG-группа `channel_type='telegram'`): реакция →
  реагировавший, сообщение → последний писавший, событие → автор события.
  В диалогах 1:1 (Business/MTProto/Wazzup/Email/прямой) — без изменений
  (собеседник), иначе в исходящем висела бы своя же аватарка.
- Для аватара автора события добавлено поле `last_event_sender_avatar_url`
  (резолв имени+аватара одним `LATERAL`-джойном — доп. нагрузка ≈ноль, участника
  уже искали ради имени).

**БД (в проде через MCP):**
- `20260616_inbox_event_actor_name.sql` — автор в тексте события.
- `20260616_inbox_event_actor_avatar.sql` — поле `last_event_sender_avatar_url`
  + проброс через все 7 inbox-функций (`get_inbox_threads_v2`,
  `get_inbox_threads_page`, `get_inbox_thread_one`, `get_inbox_unread_threads`,
  `get_inbox_needs_reply_threads`, `get_inbox_awaiting_reply_threads`,
  `get_inbox_search_threads`).
- ⚠️ Добавление колонки = смена типа возврата → `DROP`+`CREATE` всех 7 функций
  (обёртки тянут `t.*` из v2). Гранты восстановлены 1:1 с исходными (anon
  только у `needs/awaiting`, как было); проверено.

**Файлы:** обе миграции (нов), `inboxService.ts` (+поле типа),
`InboxChatItem.tsx` (текст+аватар), `useFilteredInbox.test.ts` (+поле в фабрике).
