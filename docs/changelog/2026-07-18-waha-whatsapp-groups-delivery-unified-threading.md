# WAHA (WhatsApp self-hosted): группы, галочки, реакции, единый тред по телефону, приём сообщений с телефона

**Дата:** 2026-07-18
**Тип:** feat + fix (мессенджер, карантин: WAHA/Wazzup edge) + БД (2 миграции)
**Статус:** edge задеплоены в прод (CLI, `--no-verify-jwt`); миграции применены через MCP; ⏳ живой смок за владельцем (приём — ядро)

---

Серия доработок канала WhatsApp через self-hosted WAHA. Диагностика — замерами
(логи WAHA/Supabase, `execute_sql`, официальная дока WAHA), не по догадкам.
Детальные расследования, гипотезы и грабли — в
[`messenger-ledger.md`](../../.claude/rules/messenger-ledger.md) (записи 2026-07-18 (2)–(4)).

## 1. Группа = ОДИН общий тред на команду (вариант А, feat)

**Проблема.** `waha_message_id` уникален глобально → при двух сотрудниках в одной
группе WAHA присылает сообщение дважды (по сессии на каждого), второй insert
отбивается 23505, но пустой тред второй сессии всё равно создавался.

**Решение.** Тред группы ключуется по `(workspace_id, waha_chat_id, waha_group)`, не
по сессии → обе сессии резолвят один тред. Гонка создания закрыта partial-unique
`uq_project_threads_waha_group` (миграция `20260718140000`). Доступ: сотрудник, чей
телефон в группе, добавляется в `project_thread_members` (`ensureThreadMember`) →
видит общий тред. Исходящие уходят с сессии автора сообщения (его телефон в группе →
клиент видит, кто написал), фолбэк — сессия треда.

## 2. Галочки прочтения (delivery status, feat)

WAHA-событие `message.ack` (уровни -1 err/1 sent/2 delivered/3 read) →
`handleAck` пишет `waha_status`; при read (≥3) — `recipient_read_at`, по которому
фронт рисует синие галочки (общее поле, правки UI не нужны). `handleAck` монотонен
(`ACK_RANK`) — delivered после read не откатывает. Событие `message.ack` добавлено в
`waha-sessions` (новые сессии) + вручную в конфиг активной (WAHA `PUT`).

## 3. Реакции в группах (fix)

Входящая реакция доходила до вебхука, но реагирующий брался из `p.from` (jid
**группы**), а не участника → не находился. Фикс: в группе реагирующий = `p.participant`
(через `ensureWahaContact`), контакт треда не перезаписывается.

## 4. Ревью-раунд (рефактор по итогам самокритики)

Поиск нашей строки по WhatsApp-id сведён в единый `findMessageByWahaId`
(был в 3 расходящихся реализациях — reaction/ack/reply); стратегия покрывает оба
направления (короткий stored ↔ полный пришедший id и наоборот, через «ядро» + ilike).
Убран PII-лог реакции; выбор сессии автора детерминирован; `"WORKING"` → const.

## 5. Единый WhatsApp-тред по телефону — Wazzup + WAHA универсально (feat, вынос логики)

**Задача.** Независимо от WhatsApp-канала (Wazzup ИЛИ WAHA) клиент всегда попадает в
ОДИН тред. Каналы могут временно сосуществовать.

**Правило.** WhatsApp-тред = (владелец, телефон). Канонический ключ — новая колонка
`project_threads.whatsapp_phone` (миграция `20260718150000` + backfill). На каждый
входящий тред переключается на канал-доставщик (ставим его привязку, **снимаем
привязку другого** WhatsApp-канала) → всегда ровно одна активная привязка →
исходящие маршрутизируются существующим триггером БЕЗ изменений («последний канал
клиента = канал ответа»).

**Вынос логики** — общий модуль `_shared/whatsappThread.ts` (`normalizePhone`,
`findWhatsAppThreadByPhone`, `bindThreadToWaha`, `bindThreadToWazzup`), которым
пользуются оба вебхука. Скрытый `@lid` (номер спрятан) сводится к телефону через
WAHA `GET /api/{session}/lids/{lid}`. Заодно закрыт дедуп «один клиент как `@lid` и
как `@c.us`».

**🪤 Ограничение** (в комментарии кода): не держать один и тот же номер одновременно
на обоих каналах — входящее придёт дважды (id-пространства независимы) → задвоится.

## 6. Приём сообщений, написанных с телефона (message → message.any, feat)

**Проблема (подтверждено офиц. докой WAHA).** Событие `message` отдаёт только
входящие. Сообщения, написанные прямо в WhatsApp с телефона (`fromMe`), шли под
`message.any` → терялись. В Telegram (MTProto) такие ловятся, в WhatsApp — нет.

**Решение.** Подписка сессий `message` → `message.any` (все сообщения, поле `source`:
`app`=с телефона, `api`=через наш сервис). Роутер вебхука принимает оба события.
`handleMessage`: наши api-отправки (`source:"api"`) игнорируются (уже в БД — не
задваиваем), ловим `source:"app"` и входящие. Баг NOWEB #1350 (в группах `fromMe`
приходит `false` у своих) обойдён: «своё» доопределяется по номеру сессии
(`waha_sessions.phone`).

- `deno check` чист (waha-webhook, waha-send, waha-sessions); wazzup-webhook — 1
  пред-существующая ошибка (strict-null `projectId` в storeAttachment, есть в HEAD,
  бандлер терпит).
- Миграции применены в прод через MCP; коллизий `(owner, phone)` после backfill = 0.
- Фронт не менялся — пуш не требует изменений UI.

## ⏳ Живой смок (за владельцем, карантин — приём)

- Группа: сообщение в группе → один общий тред; ответ уходит с номера отправителя;
  реакция из WhatsApp → появляется.
- Галочки: клиент прочитал → синие.
- Единый тред: клиент, имеющий тред (напр. через WAHA), пишет в Wazzup → тот же тред.
- С телефона: пишешь клиенту из приложения WhatsApp (личка и группа) → появляется в
  ЛК как исходящее; из сервиса — без дублей.
