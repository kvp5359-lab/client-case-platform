**Дата:** 2026-07-06
**Тип:** fix (edge, в проде) + feat (фронт, ждёт деплоя) + docs
**Статус:** edge — в проде; фронт/миграция — ждёт выката

## 1. Реакции в TG-группе без бота-секретаря (коммит `7ae595a`)

**Симптом:** реакция из сервиса на сообщение в TG-группе не появлялась в Telegram.
Касалось всех групп без бота-секретаря (только личные боты сотрудников). В логах —
`telegram-set-reaction → 500`.

**Корень:** функция собирала кандидатов-ботов (личный бот реагирующего → бот оригинала
→ секретарь) ДО цикла отправки. Секретарь резолвится через `resolveBotToken`, который
**бросает** `ERR_NO_SECRETARY_IN_GROUP`, когда секретаря в группе нет. Вызов не был в
try/catch → исключение → 500 ещё до попытки личного бота (который в группе админ и
реакцию поставить мог).

**Фикс:** `resolveBotToken` обёрнут в try/catch — нет секретаря → просто пропускаем его
как кандидата, реакцию ставит личный бот реагирующего.

**Промежуточная ошибка + откат (важно):** сначала также расширил ретраябельные ошибки
на `message not found` (в multi-bot группе у каждого бота свой `message_id`). Это
заработало, но с побочкой: реакция ставилась **ботом коллеги** (fallthrough на бота,
у которого верный `message_id`) → в Telegram подписывалась «от коллеги». Откатил:
`isNotMemberError` снова ловит только «бот не в группе». Историческое поведение
восстановлено — «правильно или никак»: реакцию ставит только бот реагирующего; не смог
(сообщение первым увидел чужой бот) → в Telegram реакции нет (в сервисе остаётся), но
ложной подписи не будет.

**Задеплоено:** `telegram-set-reaction` (напрямую, уже в проде).

**План (C):** чтобы реакция **всегда** доходила И правильно подписывалась — хранить
`telegram_message_id` по каждому боту (jsonb-карта на сообщении). Расписано в
`docs/feature-backlog/2026-07-06-per-bot-telegram-message-ids.md`.

## 2. Прямые чаты в карточке контакта + orphan-треды исполнителям (коммит `df9d674`, параллельная сессия)

- **Карточка контакта** (`useContactCard`/`ContactCardDialog`): показ прямых личных
  чатов контакта (Telegram MTProto / Telegram Business). Связка идёт по **числовому**
  `telegram_user_id`. Сопутствующее — `InboxChatItem`, `MessageList`,
  `ParticipantAvatars`.
- **Миграция `get_workspace_threads`** (`20260706120000_workspace_threads_orphan_assignee_visibility.sql`):
  исполнитель (`task_assignees`) теперь видит **orphan-тред** (без проекта, `project_id
  IS NULL`) **любого типа**, а не только `task`. Раньше для orphan email/chat смотрелся
  лишь `owner_user_id`, и коллега, добавленная в исполнители email-треда без проекта, не
  видела его в списках/досках, хотя RLS доступ уже давал. Зеркалит RLS.
  - ⚠️ `get_workspace_threads` тянет `get_board_filtered_threads` (`b.*`) — правка
    в WHERE, набор колонок не менялся, контракт цел.
