# Закрытие доступа клиента к чужим доскам + полировка композера и редактора сайдбара

**Дата:** 2026-06-22
**Тип:** fix (безопасность) + UX
**Статус:** completed (RLS+RPC в проде; фронт ждёт деплоя CI)

---

Три независимых блока. Главный — закрытие RLS-дыры с досками (аудит прав
клиента). Остальные два — UX-полировка из параллельных сессий, вкоммичены
заодно.

## 1. Безопасность: клиент не читает структуру чужих досок 🔴

**Симптом (аудит):** клиент, зашедший по прямой ссылке на доску, видел список
всех досок воркспейса и структуру их колонок/списков. Замер под реальным
клиентом: видел 5 досок (`boards`), 19 списков (`board_lists`), 2 доски через
UI-путь (`get_workspace_boards`).

**Корень:** RLS-политики `boards_select` / `board_lists_select` пускали
**любого** участника воркспейса ко всем доскам. `access_type`
(workspace/private/custom) проверялся **только** в RPC `get_workspace_boards` —
прямой `.from('boards')` / `.from('board_lists')` (и `get_board_lists`, который
работает как INVOKER) этот фильтр обходили.

**Что важно — НЕ утечка:** содержимое досок защищено отдельно и работает
корректно. Задачи/треды фильтруются через `can_user_access_thread`, события
календаря — через `external_calendar_events_select_owner` (клиент видит 0
событий владельца). Утекала только **структура** досок.

**Фикс** (миграция `20260622_board_access_rls_tighten.sql`, применена в прод
через MCP):
- **`can_user_access_board()`** — единый предикат доступа (как
  `can_user_access_thread` для тредов), 2 перегрузки: row-overload (терпим к
  INSERT…RETURNING) + uuid (для `board_lists`, читает row по id без рекурсии).
- **Модель доступа** (решение владельца): `board_members` → видит всегда
  (явный шеринг, любой `access_type`, в т.ч. клиент); `private` → только
  создатель; **`workspace` → только STAFF-роли** (Владелец/Администратор/
  Сотрудник/Исполнитель — через `is_staff_role`, клиент/внешние исключены);
  `custom` → совпадение роли с `access_roles`.
- **RLS `boards_select` / `board_lists_select`** переведены на этот предикат.
- **RPC `get_workspace_boards`** — на тот же предикат (CREATE OR REPLACE,
  сигнатура не менялась → гранты целы, фронт не трогаем). UI-список досок
  теперь байт-в-байт равен тому, что реально читается по RLS — без
  «фантомных» досок, которые открывались бы пустыми.

**Грабли:** `get_board_lists` — INVOKER (не SECURITY DEFINER), поэтому RLS на
`board_lists` его прикрывает автоматически. `resolve_short_id` /
`get_short_id_by_uuid` (DEFINER, anon-резолверы middleware) отдают только uuid
по короткому id — не структуру, не трогали.

**Проверка на проде (read-only, под реальным клиентом/владельцем/сотрудником):**

| Кто | Досок было | Стало |
|-----|-----------|-------|
| Клиент (UI/RPC) | 2 | 0 |
| Клиент (прямой RLS: boards / board_lists / get_board_lists) | 5 / 19 / структура | 0 / 0 / 0 |
| Владелец (RPC) | 4 | 4 (без изменений) |
| Сотрудник (RPC) | 3 | 3 (workspace-доски целы) |

**Остаётся отдельно (по желанию):** запись `board_lists` (insert/update/delete)
RLS всё ещё пускает любого участника — read-утечка закрыта, для defense-in-depth
можно ужесточить и запись; серверный guard на роуте `/boards`/`/calendar`
(сейчас клиента пустит, но увидит пусто).

## 2. Мессенджер: полировка переключателя видимости композера

*(карантинная зона, фронт-only; смок за пользователем)*

- **Цвет под акцент треда:** режим «Всем» в `ComposerVisibilitySwitch`
  окрашивается под акцент чата (`CLIENT_ACCENT_PILL`); цвет кнопки отправки
  зависит от режима (`composerSendButtonClass`, проброс `sendButtonClassName`
  в `MessageInputToolbar`).
- **Tooltip получателей** в личных диалогах без команды показывает «только вы»
  вместо пустого «команда — »; «Уведомление» при отсутствии подписчиков —
  «никто».
- **Себя исключаем по `participant_id`** (а не только по `user_id`) — работает
  и в личных диалогах без проекта.
- **`allowClientMode`** определяется по `state.isEmailChat` (email-тред внешний
  по типу `email`, не только по наличию link).
- **Скрыто от клиента:** переключатель видимости и кнопка @-упоминания
  (`!isClientOnly`) — внутренний инструмент команды.

## 3. Настройки сайдбара: трёхколоночный редактор профилей

- **`SidebarSettingsTab`** переверстан в 3 колонки: профили + быстрые действия
  (1fr) · зоны (2fr) · палитра + инспектор (2fr). `ProfilesManagerBar` — в левой
  колонке (новый проп `vertical`, вертикальная раскладка), `QuickActionsEditor`
  — в правой (`SidebarEditorCanvas` получил проп `rightExtra`).
- Грид-колонки внутри `SidebarEditorCanvas` выровнены 1:1.
