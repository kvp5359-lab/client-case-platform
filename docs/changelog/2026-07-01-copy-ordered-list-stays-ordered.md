# Фикс: нумерованный список из бабла вставлялся маркированным

**Дата:** 2026-07-01
**Тип:** fix
**Статус:** completed (ждёт деплоя фронта)

---

## Проблема

Пользователь копировал из бабла нумерованный список (`<ol>`), а в поле ввода он
вставлялся маркированным (буллеты, `<ul>`).

## Корень (замерено в браузере на живой странице)

Когда выделяешь мышью **пункты** списка (а не сам блок `<ol>`), выделение
охватывает `<li>`, но не их родителя. `range.cloneContents()` возвращает
«сиротские» `<li>` БЕЗ обёртки:

```
<li><p>Барселона</p></li><li><p>Мадрид</p></li>…   ← нет <ol>
```

Копирование в ленте идёт через `MessageList.handleCopy` (он клонирует выделение,
чистит фон и перезаписывает `clipboardData`). В буфер попадали голые `<li>`.
ProseMirror (tiptap) при вставке сирот-`<li>` без указания типа списка
оборачивает их в список **по умолчанию — bulletList (`<ul>`)**. Тип
«нумерованный» терялся уже в буфере обмена, не при вставке.

## Что сделано

Утилита `wrapOrphanListItems(container, range)` — если все элементы верхнего
уровня клонированного фрагмента `<li>`, оборачивает их в правильный тег
`<ol>`/`<ul>` (по ближайшему списку-предку выделения) с сохранением `start`,
если выделение начинается не с первого пункта. Вызывается в
`MessageList.handleCopy` после `cloneContents`, до чтения `innerHTML`.

## Грабли

- Копированием в ленте управляет **`MessageList.handleCopy`** (на контейнере
  ленты), а не бабл — он всплывает последним и перезаписывает буфер. Любую
  правку копирования делать здесь, не на баббле (bubble-обработчик перебивается).
- Оборачиваем только когда весь фрагмент — «голые» `<li>` (выделение внутри
  одного списка). Смешанный контент (текст + список) не трогаем — там обёртка не
  терялась.

## Проверки

- tsc 0, lint 0, **753 теста**.
- Сквозной замер на живой странице: копирование списка → буфер `<ol><li>…` →
  вставка → нумерованный список сохранён.
