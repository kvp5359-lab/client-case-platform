Три независимых блока, вкоммичены вместе.

## A. Пер-проектные переопределения полей шаблона треда

**Проблема.** Один и тот же шаблон треда (напр. «Написание БП»), добавленный в
разные типы проектов, всегда давал ОДИНАКОВЫХ исполнителей: исполнители лежали в
`thread_template_assignees` с привязкой к самому шаблону, а не к связке с типом
проекта. Разделить их между типами было нельзя — приходилось плодить дубликаты
шаблона в общей библиотеке.

**Модель (решение).** Общий шаблон (`thread_templates`) = дефолт-«рыба». Связка
`project_template_thread_templates` (тип проекта ↔ шаблон треда) может
**переопределить** для конкретного типа проекта 4 поля: исполнителей, дедлайн,
стартовое сообщение, доступ. Не переопределял → наследуется из общего шаблона.
Редактирование задачи внутри модуля «Задачи» шаблона проекта правит **только этот
тип проекта**; имя/иконка/название/тип остаются общими (правятся глобально).

Механика зеркалит уже существовавшее переопределение статуса/автоперехода
(они и раньше жили в связке) — расширили тот же приём на 4 поля.

**База (миграция `20260708120000_thread_template_project_overrides.sql`,
применена в прод).**
- В `project_template_thread_templates` добавлены override-колонки:
  `deadline_days`, `initial_message_html`, `access_type`, `access_roles`
  (NULL скаляр = наследовать) + `override_assignees boolean` (false = наследовать
  общих исполнителей; true = использовать override-набор, в т.ч. пустой).
- Новая таблица `project_template_thread_assignees (template_id,
  thread_template_id, participant_id)` — переопределённые исполнители пер-(тип
  проекта × шаблон). Составной FK на связку с `ON DELETE CASCADE` (открепил этап
  → override-исполнители уходят сами). RLS зеркалит связку: SELECT — участники
  воркспейса, запись — Владелец/Администратор.

**Чтение.** `useThreadTemplates` (`mapJunctionRow`) кладёт в `ThreadTemplate`
поле `projectOverride`; база объекта остаётся общей «рыбой», эффективное значение
= override ?? база. Override-исполнителей тянем **отдельным запросом**
(`fetchOverrideAssigneeMap`), а не встраиванием PostgREST — составной FK через
embed ненадёжен.

**Редактор.** В `ThreadTemplateFields` у 4 блоков (в режиме шаблона проекта)
появился заголовок-переключатель «**Из общего · переопределить**» /
«**Индивидуально · сбросить**» + затемнение неактивного поля. Логика наследования
и сброса — в `useThreadTemplateForm` (эффективная инициализация + запоминание
общих значений для сброса). Дедлайн в режиме проекта показывается отдельным
блоком (вместо чипа), чтобы дать ему такой же заголовок.

**Сохранение** (`useProjectTemplateThreadListMutations`). Пер-проектная ветка:
общее тело (имя/иконка/название/тип) — через RPC `update_thread_template_with_-
assignees` (общих исполнителей передаём как есть, чтобы RPC их не стёр), а
дедлайн/сообщение/доступ/`override_assignees` — в связку, override-исполнители —
в новую таблицу. Общий шаблон 4 полями больше не перетирается.

**Создание проекта** (`createProjectFromTemplate`). Сеялка применяет эффективные
срок/доступ/исполнителей (override ?? общее) при развороте тредов.

**Грабли.**
- Override-исполнителей читать отдельным запросом по `template_id`, не через
  embed составного FK.
- Пер-проектное сохранение НЕ шлёт 4 override-поля в общий `thread_templates`;
  общих исполнителей передаём в RPC неизменными (RPC делает delete+reinsert).
- «Переопределить дедлайн на пусто» неотличимо от «наследовать» (NULL в колонке)
  — осознанное ограничение v1.
- Компонент `ThreadTemplateFields` общий с `RecurringRuleDialog`: вся override-UI
  за опциональным пропом `projectOverride`, там ничего не меняется.

**НЕ сделано (осознанно).** Материализация стартового сообщения при создании
проекта: override сообщения сохраняется, но само сообщение при сидинге НЕ
вставляется (и общий шаблон его сейчас тоже не создаёт). Это правка messenger-
конвейера (карантин) + продуктовый выбор «отправить / черновик» — отдельным
шагом со смок-тестом.

**Файлы.** Миграция выше; `types/threadTemplate.ts`, `types/database.ts`,

`project-template-editor/ProjectTemplateThreadList.tsx`,
`project-template-editor/useProjectTemplateThreadListMutations.ts`,
`services/projects/createProjectFromTemplate.ts`.

## B. Ряд вкладок модулей проекта — измеряемый overflow + кнопка «Все вкладки»

**Было.** Ряд вкладок модулей проекта скроллился по горизонтали при нехватке
ширины (предыдущая итерация), подписи капались `max-width`.

**Стало.** Новый компонент `ProjectModuleTabBar`: все вкладки равноправны (иконка
+ подпись целиком), что не влезло в ширину ряда — уходит в кнопку-«бутерброд»
справа (список ВСЕХ вкладок, без горизонтального скролла). Активная вкладка
всегда остаётся видимой; остаток места заполняет «частичная» вкладка с обрезкой.
Ширины меряются в скрытом ряду-клоне, `ResizeObserver` пересчитывает при смене
ширины (в т.ч. при открытии боковой панели).

Заодно порядок модулей в реестре: «Финансы» подняты (order 9 → 3.5), «Подбор ВНЖ»
3.5 → 4.5; у «История»/«Дневник»/«Финансы» снят `iconOnly` (теперь с подписями).

**Файлы.** `ProjectPage/components/ProjectModuleTabBar.tsx` (новый),
`page-components/ProjectPage.tsx`, `lib/projectModuleRegistry.ts`,
`app/globals.css` (убран container-query `.project-tabs-cq`).
