# Deploy Backlog — очередь на деплой

> **Что это.** Снимок незадеплоенного/непроверенного из [`.claude/rules/messenger-ledger.md`](../.claude/rules/messenger-ledger.md) на **2026-07-11**. Ledger — источник правды; этот файл нужно **периодически перевыводить** из него (записи копятся, старые перекрываются новыми). Предыдущая ревизия (2026-07-04) сверялась через `supabase functions list` vs `git log` — тот же метод.
>
> **Как читать.**
> - **Frontend** — руками ничего не нужно, всё уедет одним `git push` (CI `deploy.yml`, blue/green). Ниже — что именно уедет и что смокать после.
> - **Edge / mtproto** — ручные шаги + смок каналов (карантин: деплой только вместе с проверкой, не вслепую).
> - **DB миграции** — уже применены в прод через MCP; файлы в репо для воспроизводимости. Отдельного действия обычно НЕ требуют.
>
> **🔑 Ключевой факт про Edge.** Деплой edge-функции выкатывает **весь текущий исходник** функции. Поэтому поздний деплой функции ⇒ в прод уехали и все более ранние закоммиченные правки той же функции. Из-за этого большинство edge-правок из ledger **де-факто уже в проде** (функция передеплоена позже по другой задаче) и им нужен только **смок**, а не деплой. Перед выкатом канала сверить `supabase functions list` (дата деплоя) vs `git log` (дата правки).

## Runbook — безопасный порядок выката

Порядок минимизирует окно рассинхрона: **миграции → edge → mtproto → frontend push → смок**.

1. **Миграции.** Обычно уже в проде (MCP). На чистом окружении:
   `supabase db push --project-ref zjatohckcpiqmxkmfxbs`
   После правок функций БД — `node scripts/db-drift-check.mjs` (сверка с эталоном).

2. **Edge (раздел 2а).** По одной или пачкой:
   ```bash
   scripts/deploy-edge.sh wazzup-webhook wazzup-send
   # gmail — при необходимости (storage-adapter), карантин:
   scripts/deploy-edge.sh gmail-webhook gmail-send gmail-watch-refresh
   ```
   `--no-verify-jwt` для webhook/`*-send`/delete проставляется **автоматически** (`scripts/deploy-edge.sh --list-nojwt`). Забыл флаг вручную → 401 от шлюза Supabase.

3. **mtproto (раздел 3).** rsync + `docker compose build mtproto && up -d mtproto` на VPS.

4. **Frontend.** `git push` в `main` → CI `deploy.yml` соберёт Docker → GHCR → blue/green переключение (без 502).

5. **Смок каналов (обязательно после edge/mtproto — карантин).**
   ```bash
   # Полная матрица по всем каналам × операциям на тест-тредах (allowlist smoke_test_threads):
   SUPABASE_URL=… SUPABASE_SERVICE_ROLE_KEY=… SUPABASE_ANON_KEY=… SMOKE_BOT_PASSWORD=… \
     node scripts/smoke-matrix.mjs --confirm
   # Read-only здоровье каналов (без отправки):
   SUPABASE_URL=… SUPABASE_SERVICE_ROLE_KEY=… node scripts/channel-health.mjs
   ```
   Плюс ручные смоки из разделов 1 (frontend) и 2б (smoke-only), которые матрица не покрывает (visibility client/team/self, склейка Gmail, per-bot реакции). Отправку реальному клиенту НЕ гоняем — только тест-треды.

6. **F1-хвост.** После смока треда «Клиенты» на v2 — удалить v1: `supabase functions delete telegram-webhook` + дроп колонки `bot_version` (по отдельному подтверждению).

## Известный латентный баг (НЕ deploy-item, чинить только со смок-окном)

**Гонка глобального `_botToken` в `telegram-webhook-v2` (§G10).** `sendMessage/editMessage/answerCallback/реакции` в `tg-api.ts` берут токен из мутабельной глобали; при 2+ ботах в одной группе параллельный запрос может перетереть токен между `await`. **Скачивание вложений уже исправлено** (токен из `ctx.botToken`), отправка — нет. Фикс: протащить `ctx.botToken` в `tg-api.ts` и вызывающих (`sync.ts`/`callbacks.ts`/`commands.ts`), убрать `getBotToken()/setBotToken()`. Слепо не править — карантин, нужен живой смок на группе с двумя активными ботами.
